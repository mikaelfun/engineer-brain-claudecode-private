# Scoping



# Summary

Combined registration for Self-Service Password Reset (SSPR) and Microsoft Entra multifactor authentication (MFA) in Azure are features that allow users to register their security information once and use it for both MFA and SSPR. This streamlines the process, reducing confusion and simplifying the user experience.

Before combined registration was enforced in all tenants (August 15, 2020 for each new tenants; March 14, 2023 for older tenants), users had to register authentication methods for MFA and SSPR separately, which could be confusing since similar methods were used for both features. With combined registration, users can register once and get the benefits of both MFA and SSPR

## Public Documentation

[Administrator documentation](https://aka.ms/securityinfodocs) describes how the new combined registration experience works.

[End-User documentation](https://aka.ms/securityinfoguide) provides guidance on how users can register different authentication methods and [End-User Experience Blog Post](https://techcommunity.microsoft.com/t5/microsoft-entra-blog/combined-mfa-and-password-reset-registration-is-now-generally/ba-p/1257355=) describes the end user experience.





# Feature Workflow

Combined registration has two distinct approaches: 'Interrupt' and 'Manage'.
- **Interrupt mode** is a wizard-like experience, presented to users when they register or refresh their security info at sign-in.
- **Manage mode** is part of the user profile and allows users to manage their security info.

(For both modes, users who have previously registered a method that can be used for Microsoft Entra multifactor authentication need to perform multifactor authentication before they can access their security info).

To validate the methods available for combined registration we should reference [this section of the Admin documentation](https://learn.microsoft.com/en-gb/entra/identity/authentication/concept-registration-mfa-sspr-combined#methods-available-in-combined-registration).



## 'Interrupt Mode' additional details

On top of what is described in [our public doc](https://learn.microsoft.com/en-gb/entra/identity/authentication/concept-registration-mfa-sspr-combined#interrupt-mode), it's important to note that **if only SSPR is enabled and Per-user MFA is not enabled/enforced, only the SSPR interrupt is called.**

When policies for MFA and SSPR are both enabled, **the MFA interrupt always comes before the SSPR interrupt**. If users do encounter an SSPR interrupt, they will see the credentials they just registered, and they only have to click "Done".

# Known Issues and Limitations
## Issue 1: Auth App - GetActivationStatus gets stuck sometimes and keeps showing spinner button
A user successfully registers their Work account in the Microsoft Authenticator application, but the service gets stuck trying to validate the registration and only fails after the login session times out.

## Issue 2: Oops, it looks like your administrator hasn�t enabled this experience for you. Please choose "Continue" to view your account
Users navigating to https://myprofile.onmicrosoft.com  to manage their registered credentials, receive this error because they have blocked 3rd party cookies in their browser.
To allow the enhanced experience to work with blocking of 3rd party cookies make all three of these URLs need to be added to the same zone:
https://login.microsoftonline.com 
https://mysignins.microsoft.com 
https://account.activedirectory.windowsazure.com 

- **Edge**: There is no way to manage allowed sites with Edge. Adding the URLs to Internet Explorer may allow Edge to work when blocking 3rd party cookies.

- **Chrome**: Click the vertical ellipses icon in the top right, click Advanced at the bottom, click Content settings, click Cookies and add all three URLs (without https://) Under Clear on exit.

This bug is to be addressed by GA and affects all browsers.


## Issue 3: Users are prompted to register a Mobile Application, but this option is not a required setting for SSPR.
This is by design, if one or both Mobile app options are enabled in MFA policy, the user should see Mobile app as a method to register.

## Issue 4: SSPR is blocked if the Admin changes the SSPR policy from 'Mobile app code' to 'Mobile app notification'
Users that completed credential registration in the past using a 3rd party authenticator application found they were blocked from performing SSPR with a message stating "You cannot reset your password at this time because necessary information is missing from your account." Additional details on that page cited error code SSPR_0014. If the user happened to remember their password and signed into https://aka.ms/ssprsetup  to see what authentication method was missing, they found all methods needed for MFA and to be compliant with SSP Policy were already registered.
- The Audit log records a Success event and the Activity Status shows a Reason of "User's account has insufficient authentication methods defined. Add authentication info to resolve this"

**Cause**: In this scenario SSPR was failing because the user registered their Mobile app using Code only, which does not satisfy the SSPR policy requirement of Mobile app notification.

**Note**: This issue doesn't happen if the administrator changes the SSPR policy from Mobile app notification to Mobile app code. When users register Microsoft Authenticator with Notification or Notification + Code enabled configured in the SSPR policy, they are registered for both Notification and Code capabilities. When the user clicks the URL on the mobile app registration prompt, they are registered for Code only even if they register using the Microsoft Authenticator application.

**Solution**: The user must Edit the Mobile application setting from the https://aka.ms/ssprsetup  page by removing their account from the authenticator application on their mobile device, then they must re-register their account. Once this is done they should see the Notification for approval in the Microsoft Authenticator application and they will be allowed to perform SSPR.


## Issue 5: OOBE flows show the old MFA page
Users going through the standard out of the box OOBE flow on Windows 10 RS4 or earlier only see the old MFA page. As of August 20, 2019, the OOBE flows did not support the new combined registration flows. The one exception where this is supported is where an account is added through an app, such as Outlook. This flow supports combined registration. In addition, the new Conditional Access user action for "Register security information" does not support OOBE flows.


## Limitation 1
Internet Explorer (IE) is not supported. Customers attempting to register or manage security info through combined registration or My Sign-ins should do so on a modern browser, such as Microsoft Edge. (While not officially supported IE11 may work in limited cases).

Older Microsoft products which have not been updated and/or using the Azure AD Authentication Library (ADAL) that relies on legacy webviews that fallback to older versions of IE will also cause problems. In such scenarios, users may experience a blank page when directed to combined registration / My Sign-ins. Please advise the customer to open up aka.ms/securityinfo on a modern browser in order to continue registration.

## Service-Side Tracing

### ASC Kusto Web UX
Service-Side tracing for SSPR and MFA in ASC Kusto Web UX.

1.	From Azure Support Center -> Tenant Explorer , open Kusto Web UX for the tenant
2.	Add the Kusto Cluster idsharedscus.southcentralus
3.	Once cluster is added, expand the databases and select the IAMUXPROD database
4.	Copy and Paste a query such as the one below to find logs for combined registration portal events:

   ```json
   let ts = datetime(2024-01-13T23:24); //replace with timestamp in UTC of error
   let delta = 15m;
   let cid = "ec11a060-c5db-4b9f-b34d-143b68f46b39"; // correlation ID of error,
   IfxDiagnosticsEventPII
   | where env_time >= ts - delta and env_time <= ts + delta
   | where * contains cid
   | project-reorder env_time, Log_Level, Log_Message, Actor_ObjectId, Action_Scenario, Action_SubScenario, Action_InternalCorrelationId
   ```

**NOTE**: The correlationID can be obtained from a HAR\Fiddler trace, or you can search for ASC AAD Audit Log events with service = `Authentication Methods` or Activity = `User started security info registration` .  Use the Log_Level and Log_Message columns to find any error\warning reasons.

#### Interpret Results

1. Identify the number of gates required by doing **CTRL+F** in the browser and searching for the word, '**challenge**'. This will highlight frames in the **Msg** column that have the word challenge in them.

2. Locate **RetrievedPasswordResetPolicyFromAduxp** under the *TraceEnumeCode* column. This contains a section called **NumberOfGatesRequiredForPasswordReset** which indicates the number of methods the admin requires to perform password reset. It looks like this:

```
Retrieved SSPR Policy: SelfServicePasswordResetEnabled:All, NumberOfGatesRequiredForPasswordReset:2, AvailableContactMethods:AlternateEmail,MobileCall,MobileSMS,SecurityQuestions,MobileAppNotification, ScopedAccessEnabled:False, ScopedAccessGroupId:680860e0-####-####-####-############, CustomContactAdminLinkEnabled:False, CustomContactAdminLink:None, NotifyAdmins:True, NotifyUser:True, EnableEnforcedRegistrationOnSignin:False, EnforcedRegistrationEnablementForUsers:On
```

3. The **RetrievedPasswordResetPolicyFromAduxp**�setting also has a section called **AvailableContactMethods** which lists of all of the methods that the Admin enabled within the SSPR Policy. SSPR will always require the most secure of these methods for users to perform password reset. Identify the most secure from the list by referencing the **Authentication Method Risk Levels** section above. If the number of gates is 2, then the two most secure need must be registered for the user to proceed with password reset, or they will blocked.

4. Next, in the browser, do **CTRL+F** and search for the word '**notification**' to highlight entries in the **Msg** column containing this word. If Mobile app and phone are selected in the SSPR policy you might see entries like this.

| TraceEnumeCode | Msg |
|---------|---------|
| PhoneOtpChallengeValidPhoneNumberCount | Found 1 valid phone number(s) for MobilePhoneCall challenge |
| MobileAppNotificationRegistration | User Registered� Status for MobileAppNotification challenge is�: False |

5. **Conclusion**: The number of "Gates" required for SSPR is 2 in policy. The Admin enabled these methods for SSPR AvailableContactMethods:AlternateEmail,MobileCall,MobileSMS,SecurityQuestions,MobileAppNotification, with MobileAppNotification and MobileCall being the most secure of these. MobileCall�is seen as registered, but MobileAppNotification is not. The user is not able to perform SSPR because they need to register the mobile app.
