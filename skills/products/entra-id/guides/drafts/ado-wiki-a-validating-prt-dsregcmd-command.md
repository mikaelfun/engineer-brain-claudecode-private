**Summary:**
This article explains the Primary Refresh Token (PRT) used in Azure Active Directory (Azure AD) authentication on various devices. It covers the content, issuance, and protection of PRTs, particularly on Windows 10 and newer devices.

## What does the PRT contain?

A PRT contains claims generally contained in any Azure AD refresh token plus device-specific claims:
- **Device ID**: determines the device the PRT was issued to the user on; used for Conditional Access based on device state or compliance.
- **Session key**: encrypted symmetric key generated by Azure AD authentication service; acts as proof of possession when PRT is used to obtain tokens.

## How is a PRT issued?

Device registration is a prerequisite. During registration, dsreg component generates two key pairs:
- Device key (dkpub/dkpriv)
- Transport key (tkpub/tkpriv)

Private keys are bound to the device TPM if available; public keys are sent to Azure AD.

### PRT issuance scenarios:

1. **Azure AD joined or Hybrid Azure AD joined**: PRT issued during Windows logon when user signs in with org credentials. Supports password and Windows Hello for Business. Azure AD CloudAP plugin is primary authority.

2. **Azure AD registered device**: PRT issued when user adds a secondary work account via:
   - "Allow my organization to manage my device" prompt after signing in to an app
   - Settings > Accounts > Access Work or School > Connect

## Interpreting dsregcmd /status

Run in a non-elevated command prompt:
```cmd
dsregcmd /status
```

Key fields to check:
- Device state (AzureAdJoined, DomainJoined, etc.)
- SSO state (AzureAdPrt, AzureAdPrtUpdateTime, etc.)
- Diagnostic data section
