## **Overview**

The Agents on the on-premise machines that Microsoft Entra Connect Health monitors periodically uploads data to Microsoft Entra Connect Health Service (henceforth called the Health Service). If the Health Service does not receive data from an agent, the information presented in portal will be stale. To highlight the issue, the Health Service will raise a **"Health service data is not up to date"** alert. This is generated when the Health Service has not received data in the last two hours.

- The **Warning** status alert fires if the Health Service has received only partial data elements (Data Types) sent from the server in the last two hours. Warning status alert does not trigger email notifications to the email notification recipients.

- The **Error** status alert fires if the Health Service has not received any data elements from the server in the last two hours. Error status alert triggers email notifications to the email notification recipients.

The Service gets the data from Agents that are running on the on-premise machines. Depending on the Service Type, the following tables lists the agents that run on the machine, what they do as well as the Data Type (or data element) that is generated by the service. In some cases, there are two services involved in the process, so either of them could be the culprit.

**Note:** As of 04/01/2024, the DataFreshness Alert or "Health Service data is not up to date" alert for PerfCounters was disabled from all service types other than a specific ADFS server scenario. The reasoning for that is, more often than not the issue was related to the customer machines not generating the data types. So, most CRIs would involve troubleshooting something that is not related to the service. We monitor other data types to make sure data is flowing from customer machines to our service.

### **Understanding the Alert**

The Alert Details blade has one section which indicates the unique Alert Id as well as the time when it was raised, and the time when it was last detected. The Alert is generated/re-evaluated by a background process which runs every two hours. The alert also has a "pre-warning" state, so it will become an actual alert only if it was evaluated twice and it failed both times.

The blade also details the time when a particular Data Type was last received by the Health Service.

## **Troubleshooting steps**

The steps required to diagnose the issue is given below. The first is a set of basic checks that are common to all Service Types.

### **Common Diagnostic steps**

1. Ensure that the latest versions of the agents are installed.
   - Version history is available here: https://docs.microsoft.com/en-us/azure/active-directory/hybrid/reference-connect-health-version-history
2. The required Microsoft Entra Connect Sync Agents/services are running on the machine.
3. Make sure requirements are met.
4. If you have HTTP Proxy, please follow configuration steps here.
5. You can also use the Test-Connectivity PowerShell cmdlet to discover connectivity issues.
   If Test-Connectivity runs successfully as a user, but if all Data Types are missing, it is possible that the proxy setting is correct for the user, but not for 'Local System' which is the context the service is running under. For this reason, please use `Test-MicrosoftEntraConnectHealthConnectivity`

   To check if Proxy settings are correct for 'Local System', download psexec (https://docs.microsoft.com/en-us/sysinternals/downloads/psexec).
   - Run IE as system: `psexec -s -i "c:\Program Files\Internet Explorer\iexplore.exe"`
   - Check the Proxy Settings
   - Update the proxy setting to match the current setup, and restart the services.

6. Check for connectivity issues during upload on customers environment.
   We can check for connectivity issues in the directory: `C:\ProgramData\Microsoft\AadConnectHealth\BootstrapAddOn\<machine-id>`
   Specifically, request.json/response.json files. Those two files should give us an idea of why upload isn't happening.

   The agent might be able to start so it passes startup validation once but something changes later. We can retry start up validation:
   - i. Retry startup validation by clearing the `startupstatus_<version>.json` files under the `C:\ProgramData\Microsoft\AadConnectHealth\BootstrapAddOn\<machine-id>` directory and restarting agent, which would generate a fresh startup trace file under `C:\ProgramData\Microsoft\AadConnectHealth\Trace\`
   - ii. OR Enabling Agent event log channel: Run Windows Event Viewer -> Application and Services Logs -> Microsoft -> AzureADConnectHealth -> MonitorDataManagement -> Operational. Restart the agent, and look for any errors that show up.

   If the agent has not connected for more than 30 days they would need to reregister it.

## **Next steps**

### **Kusto Web UX**
Refer to the kusto queries from the Connect Health Kusto Web UX wiki page to analyze the issue.

### **Collecting agent logs**

If the dashboard is not helping, the next step would be to collect diagnostics logs.

Start command prompt using: `PsExec.exe -i -s cmd`

PsExec can be downloaded here: https://docs.microsoft.com/en-us/sysinternals/downloads/psexec

#### **Collecting Insights Agent logs**

Start a command prompt in admin mode and change directory to:

`C:\Program Files\Microsoft Azure AD Connect Health Agent`

Run the executable with /console argument:

`Microsoft.Identity.Health.AgentV15.Service /console`

Search for "Error" in the logs and see if it indicates a problem (connectivity/proxy configuration, for example).

If that does not work, please file a CRI.

## Run Test-Connectivity as System specifying the agent role

Run PowerShell as Admin and execute:

```powershell
Test-MicrosoftEntraConnectHealthConnectivity -Role adds
Test-MicrosoftEntraConnectHealthConnectivity -Role adfs
Test-MicrosoftEntraConnectHealthConnectivity -Role sync
```
