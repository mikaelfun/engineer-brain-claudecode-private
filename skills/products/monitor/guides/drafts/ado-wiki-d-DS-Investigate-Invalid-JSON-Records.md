# Scenario
---
This troubleshooting guide applies to a scenario where customer sees a JSON parsing error in their Log analytics workspace.


# Sample error message
---
Data of type AzureDiagnostics is being dropped due to incorrect format at lineOffset: 88. Exception message: Invalid property identifier character: \. Path 'records[88].properties', line 1, position 53590

# Outcome of this TSG
---

*   This will assist in verifying the error reported by the customer.
*   It will aid in identifying specific records or groups of records responsible for the error.

# Information needed from customer.  
---

* Workspace Id
* Sample error message
* Repro in last 7 days

# General understanding
---
* These records are generated by the resource provider. So, this issue should be fixed by them
* We are aiding the resource provider in pin-pointing the record which is causing this error for quicker resolution.
* Diagnostic setting NOR LA ingestion team cannot fixed this issue. Issue should be deflected to RP after following below steps

# Taken from previous ICMs - SQL Resource
---

Upon investigation from the SQL RP on this issue, they have provided a root cause. please see below.

Reference case : 2502240040014372

**Issue**: Cx is getting operation alert on invalid json in portal. But they are able query every thing properly. So, nothing blocked them during this time.
**RCA**: Statement field in audit logs has 4000 character limit. Few of the cx queries are crossing the limit, and making the query overflowing the statement field and making a wrong json.
**Mitigation**: From cx side, they have to reduce the query size to mitigate it quickly. Or they have to ignore these alerts.
**Work Item**: I'll create a work item to handle scenario where the char limit is crossing 4k character.�
Cx agreed to close the case.

**Note: This RCA is for reference purposes in understanding - what may have been causing this issue.
You are still required to get the root of the issue by following the below steps and collaborating with  resource provider.**

2. If the query results in records, then continue with steps below. If not, then we need a new repro or need more help post a message here [AzMon POD Swarming | Autoscale, Activity Logs and Resource Logs | Microsoft Teams](https://teams.microsoft.com/l/channel/19%3Ae8340fd5f1784ae186e8873be02b9053%40thread.tacv2/Autoscale%2C%20Activity%20Logs%20and%20Resource%20Logs?groupId=2fb9049b-bc9c-4cca-a900-84f22c86116c&tenantId=72f988bf-86f1-41af-91ab-2d7cd011db47)

3. Take the following columns values from above result set.
* OBO_WorkSpaceId -> LA workspace Id
* OBO_OdsDatatypeId -> this indicates the table name which is seeing JSON parsing errors.
* FirstPartyBlobPath-> This indicates the blob path where RP wrote this record.
* Codes -> **OperationMsg:JSON Format,Filtered:ParseFailure,Filtered:NoOutputRecords**
* ResourceId -> this indicates resource for which this record was emitted and the presence of diagnostic setting. 

4. Run the below query
```
cluster('azureinsights.kusto.windows.net').database("Insights").InputBlobFirstTagMetadata
| where PreciseTimeStamp between (todatetime("03/13/2026 00:00") .. todatetime("03/14/2026 00:00")) //adjust the time frame
| where blobPath =~"<>"// take the value from FirstPartyBlobPath
| where firstTagValue contains "" //take the value from resource id column. if its Microsoft.web and application insights resource. please query "registrationtelemetry" table and extract the value from "firsttagvalue" column
| distinct eventNamespace, eventTableName, eventEnvironment
```
5. In the above result set, you should see two columns which is "eventNamespace", "eventTableName" and "eventEnvironment".
6. Next, use these three columns to build a DGREP to actually see the invalid record emitted by the RP which is a solid proof to the RP to fix this issue.
