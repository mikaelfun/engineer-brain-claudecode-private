# Azure Local Disconnected Operations (ALDO / Project Winfield) — Support Boundaries

> Generated by AI from meeting notes. Verify accuracy before use.

## Project Winfield (Arc Autonomous) Overview

Project Winfield is deployed on top of Azure Stack HCI and used to manage additional HCI clusters and workloads, operating **fully disconnected from Azure**. It uses a modified HCI OS that allows for disconnected deployments.

**Deployment sequence:** Customer deploys Azure Stack HCI first → then deploys Winfield → subsequent clusters managed through the local Winfield portal (not Azure portal).

## Support Boundary Agreement

| Scenario | Owner Team |
|---|---|
| Initial Winfield deployment issues | **Winfield team** (fronts triage) |
| Subsequent HCI cluster deployments | **HCI team** (engaged by Winfield team as needed) |
| HCI OS / HCI stack issues | **HCI team** |
| Winfield portal, local control plane | **Winfield team** |

### Support Model
- **Winfield team fronts initial triage** for all Winfield-related issues
- **HCI team engaged as needed** (white-glove support during beta)
- Goal: minimize customer inconvenience by avoiding ping-pong between teams
- Winfield acts as **workload layer** on top of HCI; HCI is the infrastructure

## Key Architecture Facts
- Winfield is deployed as a VM appliance (seed node) on the first HCI node
- Uses upstream Azure services baked into the appliance for cluster deployment
- Two NICs (Management & Ingress) on Appliance VM, connected via HCI network intents
- Once Winfield appliance is up → local control plane is active; customers use **Winfield portal** instead of Azure portal
- All extensions and bits for disconnected deployment are hosted locally by Winfield

## Escalation
- If initial triage indicates HCI layer issue → Winfield team engages HCI team
- For support boundary ambiguity → WinCloud PCY team coordinates
