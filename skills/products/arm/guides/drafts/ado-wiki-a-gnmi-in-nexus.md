# gNMI in Azure Operator Nexus

**Created by: Carlos Natera** - _Last reviewed: 08/01/26_

## What is gNMI?

**gNMI (gRPC Network Management Interface)** is an open protocol from the OpenConfig working group that uses **gRPC** and **protobuf** to _get_, _set_, and _subscribe_ to structured network configuration and telemetry data. In Nexus, gNMI is the standard interface paired with **OpenConfig models** to configure devices and stream telemetry efficiently and securely (TLS).

Within Azure Operator Nexus Network Fabric (NNF), gNMI is preferred over older mechanisms (e.g., NETCONF/SNMP) for efficiency, streaming telemetry, and consistency across vendors via OpenConfig.

## What does gNMI do in Azure Operator Nexus?

Nexus uses gNMI for two major workloads:

1. **Configuration & operational actions (on demand)**
   - The **NNF Operator** (a K8s custom controller running in the NFC Infrastructure AKS cluster) issues on-demand gNMI requests to apply device configurations and verify reachability
   - Can fall back to vendor-specific models or CLI-over-gNMI where OpenConfig objects do not exist
   - TLS certificates for device auth are generated by the RP services, propagated via K8s Bridge, persisted as K8s secrets

2. **Telemetry (continuous streaming)**
   - The **GnmiGw (Monitoring Extension)** establishes secure gNMI subscriptions to devices (e.g., Arista service port 6030)
   - Translates updates into metrics/logs forwarded to **Geneva/Azure Monitor**
   - GnmiGw runs as a K8s Deployment; Geneva agents run as DaemonSets

> Vendor posture: Arista has the most mature OpenConfig/gNMI adoption that Nexus relies on today; extending to other vendors requires investment to close model gaps.

## Why is gNMI important for Nexus?

- **Single, standardized control/telemetry plane**: Using OpenConfig over gNMI gives Nexus a vendor-neutral, model-driven way to configure devices and stream state
- **Real-time observability**: Streaming subscriptions eliminate polling delays; telemetry lands in Geneva/Azure Monitor
- **Security & compliance**: Mutual TLS between clients (Operator/GnmiGw) and devices; binaries are Microsoft-signed
- **Operational breadth**: About **60%** of config uses standard OpenConfig objects today; the rest uses vendor extensions or CLI-over-gNMI until models mature

## Key Components

| Component | Type | Role |
|-----------|------|------|
| NNF Operator | K8s controller | On-demand gNMI config requests |
| GnmiGw | K8s Deployment | Streaming telemetry via gNMI subscriptions |
| Geneva Agents | DaemonSets | Forward telemetry to Azure Monitor |
| Arista devices | Network switches | gNMI servers (port 6030) |
