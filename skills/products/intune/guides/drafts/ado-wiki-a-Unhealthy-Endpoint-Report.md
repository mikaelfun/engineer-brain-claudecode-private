# About Unhealthy Endpoints report

Unhealthy Endpoints in Endpoint Security refer to Intune-enrolled Windows clients that have any of the statuses below reported by the Defender Antivirus agent running on the machine.

**To locate Unhealthy Endpoints:**

•	MEM Admin Center -> Endpoint Security -> Antivirus

**Summary Tab:-**

This is the default dashboard for Unhealthy Endpoints detected by Intune. The admin will land on this section as soon as they navigate to the Antivirus blade. This section provides a high-level device count of the Unhealthy Endpoints and their respective categories of alerts/issues.

**Unhealthy Endpoints Tab:-**

This tab lists the devices detected as unhealthy by Intune and provides more detailed information about the devices with a greater number of columns. It can be referred to for more information on the unhealthy endpoint, such as Device Name, Malware Protection Status, Managed by, etc.

# How Unhealthy Endpoints Work

Unhealthy Endpoints reporting uses [Defender CSP](https://learn.microsoft.com/en-us/windows/client-management/mdm/defender-csp) to query the protection status of the Microsoft Defender Antivirus agent during the device check-in. It is then processed by the reporting service to display the devices matching the criteria present in the Summary Dashboard.

# How to Configure Unhealthy Endpoints

There is no explicit configuration required for this report. The reporting works if the device is managed by Intune.

# Support Boundaries

The scope of Intune here is to collect the device protection status generated by the Microsoft Defender Antivirus agent running on the device and reflect the same in MEM Admin Center. Hence, if there is a mismatch between the data collected and shown in the UI, it can be escalated to Intune PG after validation.

If there is a discrepancy in the protection status generated by Microsoft Defender, the MDE team should be engaged via collaboration, or the ticket can be transferred to them for investigation.

# Troubleshooting Unhealthy Endpoints

## 1. How does the Dashboard status map to nodes in Defender CSP?

| Status               | Description                                           | Defender CSP Node                          |
|----------------------|------------------------------------------------------|--------------------------------------------|
| Pending Update      | Security intelligence version is out of date         | Health/SignatureOutOfDate = True/False    |
| Pending Full Scan   | Pending full scan due to threat action               | Health/ComputerState = 1                  |
| Pending Restart     | Reboot required due to threat action                 | Health/ComputerState = 2                  |
| Pending Manual Steps| Manual intervention required due to threat action    | Health/ComputerState = 4                  |
| Pending Offline Scan| Offline scan required (applies to Windows 10 & 11)   | Health/ComputerState = 8                  |
| Critical Failures   | Threat remediation failed critically                  | Health/ComputerState = 16                 |
| Inactive Agent     | Defender service is disabled                          | Health/DefenderEnabled = True/False       |
| Unknown Status     | Defender service provided a null value for agent status | Health/DefenderEnabled = Null          |

## 2. How to validate the AV Protection status reason on the client machine?

```powershell
Get-MpComputerStatus | fl computerstate, AMrunningmode, DefenderSignaturesOutOfDate
```

- **ComputerState** - Confirms Pending Full Scan, Pending Restart, Pending Manual Steps, Pending Offline Scan, and Critical Failures.
- **AMRunningMode** - Confirms if Defender Service is Disabled. Any value other than 'Not Running' = service enabled. If null → device shown under 'Unknown Status'.
- **DefenderSignaturesOutOfDate** - Confirms if Security Intelligence version is outdated → 'Pending Update' status.

**Note:** Intune doesn't control any of the values returned by Get-MpComputerStatus — it only reads the values generated by Microsoft Defender Antivirus Agent.

## 3. How to validate the data collected by Intune?

Use the Kusto query below to fetch data collected by Intune (refer to Col1, latest record). Compare against device PowerShell output.

- If values match across device, Kusto, and UI → no issue with Intune functionality.
- If mismatch between Kusto Output and UI → escalate to PG.
- If PowerShell output doesn't reflect actual device state → engage/transfer to MDE.

# Kusto Query

```kusto
IntuneEvent 
| where env_time >= ago(30d)
| where env_cloud_name == "ASU"
| where ApplicationName == "IntuneDefenderServices"
| where ServiceName == "DefenderBTService"
| where ComponentName contains "ProcessDefenderReportBTProvider"
| where Col1 contains "IntuneDeviceID"
| project env_time,ComponentName,ServiceName,ApplicationName,EventUniqueName,Col1,AccountId
```
