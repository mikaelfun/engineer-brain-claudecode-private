## Troubleshooting Flow

### Phase 1: Playbook
> Sources: ado-wiki, mslearn

**1. Block AAD User incident playbook fails with InvalidAuthenticationToken error code - Access token is empty when trying to disable user in Azure AD**

- **Root Cause**: The playbooks managed identity has not been granted the required Microsoft Graph API permissions (User.Read.All, User.ReadWrite.All, Directory.Read.All, Directory.ReadWrite.All)
- **Solution**: Grant the required permissions to the managed identity using PowerShell: Connect-AzureAD, then use New-AzureAdServiceAppRoleAssignment to assign Directory.ReadWrite.All and User.ReadWrite.All permissions from the Microsoft Graph service principal (AppId: 00000003-0000-0000-c000-000000000000) to the playbooks managed identity. Also assign Microsoft Sentinel Responder role to the managed identity
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**2. Block AAD User incident playbook fails with HTTP - get user manager error: Not Found when trying to notify the users manager**

- **Root Cause**: The target user account does not have a manager configured in Azure AD. The playbooks Get user manager step requires the user to have a manager assigned
- **Solution**: Inform customer that the target user account needs a manager assigned in Azure AD for the notification step to work. Customer should ensure user accounts have managers set in Azure AD (Users > Profile > Manager). The playbook will still disable the user but the manager notification will fail
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**3. Block AAD User incident playbook fails with Update user - disable user: Forbidden when trying to disable a user account via automation**

- **Root Cause**: The target user account is an admin/privileged account. The Block AAD User playbook is designed to not disable admin users as a safety measure - this is by-design behavior
- **Solution**: Inform customer that the Block AAD User playbook intentionally does not disable admin accounts. If the customer needs to disable an admin account, they must do it manually through Azure AD portal or use a custom playbook with elevated permissions and explicit admin-disable logic
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**4. Logic App returns HTTP 301 error when calling MCAS (Microsoft Cloud App Security) REST API using the URL provided by MCAS portal that includes datacenter designation (e.g. https://<domainname>.eu2.por**

- **Root Cause**: MCAS API URLs with datacenter designation (e.g. 'eu2') issue HTTP 301 redirects. While REST API tools handle this redirect seamlessly, Azure Logic Apps do not automatically follow HTTP 301 redirects, causing the call to fail.
- **Solution**: Remove the datacenter designation from the MCAS URL. Change https://<domainname>.eu2.portal.cloudappsecurity.com to https://<domainname>.portal.cloudappsecurity.com. Additionally, for resolve/dismiss alert API endpoints, append a trailing '/' to the URL (e.g. /api/v1/alerts/resolve/ and /api/v1/alerts/<alertid>/dismiss/).
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**5. Get-MDEInvestigationPackage playbook generates a download URI for the investigation package, but customer receives an expired token error when trying to download it.**

- **Root Cause**: The SAS URI generated by MDE for downloading the investigation package has a very short expiration window (approximately 1 minute). If the customer does not use the URI immediately, it expires.
- **Solution**: Inform the customer that the download URI must be used immediately (within 1 minute of generation). Reference: https://learn.microsoft.com/en-us/microsoft-365/security/defender-endpoint/get-package-sas-uri?view=o365-worldwide#response
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**6. Get-MDEInvestigationPackage playbook fails with error: 'The application does not have any of the required application permissions (Machine.Read.All, Machine.ReadWrite.All) to access the resource'.**

- **Root Cause**: The managed identity used by the Logic App does not have the required Microsoft Defender for Endpoint API permissions (Machine.Read.All, Machine.ReadWrite.All) granted.
- **Solution**: Grant Machine.Read.All and Machine.ReadWrite.All permissions to the managed identity using PowerShell in Cloud Shell: Get-AzureADServicePrincipal for the MDE app (fc780465-2017-40d4-a0c5-307022471b92), then New-AzureAdServiceAppRoleAssignment for each required permission.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**7. Reset-AADUserPassword playbook (Logic App) in Microsoft Sentinel fails to execute or cannot reset user password after deployment from GitHub template**

- **Root Cause**: Managed identity is missing required permissions: Password Administrator (or Global Administrator) Azure AD role, User.ReadWrite.All Microsoft Graph API permission, and/or Sentinel Contributor + Logic App Contributor roles at resource group level
- **Solution**: Grant all required permissions to the managed identity: (1) Assign Password Administrator directory role via Add-AzureADDirectoryRoleMember, (2) Grant User.ReadWrite.All Graph permission, (3) Add Microsoft Sentinel Responder role via New-AzRoleAssignment at resource group scope, (4) Add Sentinel Contributor + Logic App Contributor roles to the managed identity. Use the PowerShell script from the playbook GitHub page. After deployment, attach the playbook to an automation rule.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**8. Customer wants to schedule a Microsoft Sentinel KQL query to run periodically (e.g., weekly report via email) but does not know how to configure scheduled playbooks**

- **Root Cause**: Customer is unaware that the Azure Monitor connector (monitoring connector) in Logic Apps can be used with a Recurrence trigger to schedule and deliver query results via email
- **Solution**: Use Logic Apps with the Azure Monitor connector: (1) Add a Recurrence trigger set to desired frequency (e.g., Weekly), (2) Set the time zone on the recurrence connector, (3) Use Run query and list results or Run query and visualize results action, (4) For visualization, results can be embedded directly in email body. Note: custom KQL query creation is the customers responsibility.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — ADO Wiki]`

**9. Sentinel automation rule fails to trigger playbook: The playbook was disabled or The subscription is disabled and marked as read-only**

- **Root Cause**: The Logic App playbook is in Disabled state, or the Azure subscription hosting the playbook is disabled/read-only.
- **Solution**: Enable the playbook from Sentinel Automation > Active Playbooks tab or Logic Apps resource page. If subscription is disabled, re-enable it in Azure portal.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**10. Sentinel automation rule fails to trigger playbook: Microsoft Sentinel is missing permissions to run it or Playbook was not migrated to new permissions model**

- **Root Cause**: Sentinel lacks the required RBAC permissions (Microsoft Sentinel Automation Contributor) on the playbook resource group, or the playbook was not migrated to the new permissions model.
- **Solution**: Grant Microsoft Sentinel permissions to run the playbook (assign Sentinel Automation Contributor role on the playbook RG). For legacy playbooks, re-grant permissions and re-save the automation rule.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**11. Sentinel playbook trigger fails: Access control configuration restricts Microsoft Sentinel or Access was forbidden due to managed identity misconfiguration or Logic Apps network restriction**

- **Root Cause**: Logic App has IP-based access control configured that blocks Sentinel service, or managed identity is not properly configured with required permissions, or network restrictions prevent triggering.
- **Solution**: Remove IP access restriction in Logic Apps settings to unblock Sentinel, or ensure managed identity has correct RBAC assignments. Check Logic App networking settings for private endpoint / VNet restrictions.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**12. Sentinel playbook trigger fails: Too many requests, exceeding workflow throttling limits or exceeding ARM throttling limits**

- **Root Cause**: The number of waiting workflow runs exceeded the Logic App maximum allowed limit, or ARM API throttling limits are being hit.
- **Solution**: Increase maximumWaitingRuns in Logic App trigger concurrency configuration. Review ARM throttling limits documentation and reduce concurrent invocations.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**13. Sentinel playbook trigger fails: The subscription or resource group was locked**

- **Root Cause**: An Azure resource lock (ReadOnly or CanNotDelete) is applied on the subscription or resource group containing the playbook, preventing Sentinel from triggering it.
- **Solution**: Remove the resource lock on the subscription or resource group to allow Sentinel to trigger playbooks. Use Azure portal > Resource Group > Settings > Locks to manage locks.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**14. Sentinel playbook trigger fails: Invalid credentials in connection, Invalid template definition, or Playbook ARM ID is not valid**

- **Root Cause**: API connection credentials used by the playbook are expired or invalid, the playbook workflow definition has errors, or the playbook ARM resource ID is malformed.
- **Solution**: For invalid credentials: check and refresh API connections in Azure portal > API connections service. For invalid template: open Logic App designer, fix errors and save. For invalid ARM ID: verify the playbook resource exists and rule references correct ID.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**15. Sentinel playbook trigger fails: Contains an unsupported trigger type**

- **Root Cause**: The playbook Logic App does not start with a supported Microsoft Sentinel trigger (Incident, Alert, or Entity trigger). It may use a different Logic App trigger type.
- **Solution**: Edit the playbook in Logic App designer and ensure it starts with the correct trigger: Microsoft Sentinel Incident, Microsoft Sentinel Alert, or Microsoft Sentinel Entity trigger.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**16. Sentinel entity trigger playbook fails to run to completion when triggered outside an incident context (e.g., during threat hunting)**

- **Root Cause**: The playbook uses the Incident ARM ID dynamic field which is null when the playbook is triggered from entity context without an associated incident, causing downstream actions to fail.
- **Solution**: Add a Condition action in Logic App before any action that references Incident ARM ID: check if Incident ARM ID is not equal to null. Put incident-dependent actions in the True branch and alternative actions in the False branch.
- ✅ **21V/Mooncake**: Applicable

**17. Sentinel playbook grayed out in automation rule dropdown and cannot be selected**

- **Root Cause**: Microsoft Sentinel service account lacks Automation Contributor role on the playbook resource group; or playbook uses wrong trigger type (must match automation rule trigger)
- **Solution**: Select Manage playbook permissions in automation rule; grant permissions on resource groups containing playbooks; user needs Owner permissions on resource group and Logic App Contributor role to run playbooks
- ✅ **21V/Mooncake**: Applicable

**18. Sentinel playbook on incident shows Cannot access data error in Defender portal**

- **Root Cause**: Incident not yet synchronized from Microsoft Defender XDR to Microsoft Sentinel; synchronization can take up to 5 minutes
- **Solution**: Wait for incident synchronization to complete; refresh the incident page after a few minutes; playbook will then run successfully

**19. MSSP multi-tenant playbook execution fails - Sentinel cannot run playbook in customer tenant from service provider automation rule**

- **Root Cause**: Microsoft Sentinel service account needs permissions in both customer and service provider tenants; Azure Security Insights app Object ID must be added to Azure Lighthouse delegation with Automation Contributor role
- **Solution**: In customer tenant: configure playbook permissions via Settings > Playbook permissions; in service provider tenant: add Azure Security Insights app to Lighthouse delegation with role f4c81013-99ee-4d62-a7ee-b3f1f648599a (Automation Contributor)
- ⚠️ **21V/Mooncake**: Not applicable

### Phase 2: Automation Rule
> Sources: ado-wiki, mslearn

**1. Sentinel Automation rule Logic App receives different/additional entities when triggered by incident update compared to incident creation. Entities appear in the update trigger output that were not in**

- **Root Cause**: Incident creation trigger uses entities from the original alerts associated with the incident, while the incident update trigger retrieves entities from GraphStore (GS). GraphStore performs entity resolution and enrichment, which can surface additional entities not present in the original alerts.
- **Solution**: This is expected behavior by design. The incident creation trigger pulls entities from original alerts, while the update trigger pulls from GraphStore which enriches entities. If consistent entities are needed, use only the incident creation trigger, or add entity filtering logic within the Logic App to handle the difference.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

**2. Sentinel automation rule import fails: rule is disabled with Unknown rule condition or Unknown custom details key, or deployment fails with Automation rules failed to deploy**

- **Root Cause**: The imported automation rule references an analytics rule or custom details key that does not exist in the target workspace, or the referenced playbook was moved/deleted/inaccessible, or the rule is past its expiration date.
- **Solution**: Export the referenced analytics rule from the original workspace and import to target. Ensure playbooks exist and target workspace has access to playbook resource group. For expired rules, update expirationTimeUtc in JSON to a future date before importing.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 7.5/10 — MS Learn]`

**3. Sentinel automation rule stops working after onboarding workspace to Defender portal - uses SecurityIncident Description field as condition**

- **Root Cause**: After onboarding to Defender portal, SecurityIncident table no longer includes Description field; automation rules using Description as condition will not match
- **Solution**: Update automation rule conditions to remove Description field dependency; use Analytics rule name condition or tags for specificity instead of incident titles or descriptions
- ⚠️ **21V/Mooncake**: Not applicable
- `[Score: 🔵 6.0/10 — MS Learn]`

**4. Sentinel automation rules using incident titles as conditions fail after Defender portal onboarding because incident names changed**

- **Root Cause**: Defender portal correlation engine renames existing incidents when applying alert correlation; incident titles no longer match original analytics rule names
- **Solution**: Avoid using incident titles as condition criteria in automation rules; use Analytics rule name condition instead; use tags for additional specificity
- ⚠️ **21V/Mooncake**: Not applicable
- `[Score: 🔵 6.0/10 — MS Learn]`

**5. Sentinel automation rule playbook trigger delayed up to 10 minutes in Defender portal**

- **Root Cause**: In Defender portal, incident creation flows through XDR first then forwards to Sentinel; alert-to-incident-to-automation chain introduces up to 10 minute latency
- **Solution**: Expected behavior in Defender portal; for time-sensitive automations consider keeping Azure portal workflow; automation rules with alert triggers only act on Sentinel alerts in Defender portal
- ⚠️ **21V/Mooncake**: Not applicable
- `[Score: 🔵 6.0/10 — MS Learn]`

### Phase 3: Automation
> Sources: ado-wiki

**1. Create playbook or edit playbook buttons are greyed out in Microsoft Sentinel USX portal, user cannot create or edit automation playbooks**

- **Root Cause**: User lacks Sentinel Contributor role (Azure) or Detection Tuning (URBAC portal), another VS Code editor tab/window is open, or tenant has no Security Copilot workspace with capacity in US/EU (or CrossRegion toggle is off)
- **Solution**: 1. Verify user has Sentinel Contributor role or Detection Tuning permission. 2. Close any other browser tabs/windows with the VSCode editor open. 3. Ensure tenant has Security Copilot workspace with capacity in US/EU region, or enable the CrossRegion toggle.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 5.5/10 — ADO Wiki]`

**2. Cline AI agent in Playbook Generator returns HTTP 429: Too Many Requests error when sending messages**

- **Root Cause**: Security Copilot workspace capacity is not located in US/EU region, or the Allow Overage option is not enabled
- **Solution**: Ensure the Security Copilot workspace capacity is located in US or EU region and verify that Allow Overage is checked in the workspace configuration.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 5.5/10 — ADO Wiki]`

**3. Playbook execution shows as Failed in Activities tab with message playbook is disabled or API call failures with HTTP error codes**

- **Root Cause**: Playbook was not enabled after creation/editing (playbooks must be explicitly enabled in the Playbooks configuration page), or integration credentials/API call structure is incorrect
- **Solution**: 1. If failure shows playbook not enabled, go to Playbooks page and enable the playbook. 2. For other failures, review the API calls and HTTP return codes in the Run Playbook activity side panel, then edit the playbook and consult Cline on the error. Check integration credentials and API call structure.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 5.5/10 — ADO Wiki]`

**4. Graph integration added with proper permissions in Playbook Generator, but playbook execution fails with 4xx HTTP errors during runtime**

- **Root Cause**: The app registration used for the Graph integration is newly created and its permissions have not yet propagated (AAD permission propagation delay)
- **Solution**: Wait for the newly created app permissions to take effect (propagation delay). Retry the playbook execution after a few minutes. Ensure the Graph integration has at least Read Alerts permissions.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 5.5/10 — ADO Wiki]`

**5. Automation rules fail to update MTP incidents in USX - MTP API throttling error: Reached Microsoft 365 Defender throttling limit - Maximum allowed 150 per 00:01:00**

- **Root Cause**: Each tenant has a limit of 150 API updates to MTP per minute across all workspaces. UI actions, automation rules, and API calls all count toward this limit. High-volume automation can exceed the quota.
- **Solution**: 1. Use Kusto query on securityinsights cluster (SecurityInsightsProd) to verify throttling: filter Operations by workspaceId and automationRuleName, look for TooManyRequests in resultDescription. 2. Reduce automation rule frequency or consolidate updates. 3. Stagger automation rule executions to stay under 150/min limit.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🔵 5.5/10 — ADO Wiki]`

### Phase 4: Workflow Automation
> Sources: ado-wiki

**1. Workflow Automation rule in Microsoft Defender for Cloud does not trigger or automation not working as expected**

- **Root Cause**: Multiple possible causes: (1) Rule set configured incorrectly (e.g., only medium severity selected instead of all severities); (2) Event Hub or Logic App URI/Key changed but not updated in automation config (invalid key); (3) User lost permissions on subscription or target resource; (4) KQL query issue - data sent but error occurred on processing side
- **Solution**: 1) Verify rule set configuration matches expected trigger conditions. 2) Check URI and Key in automation configuration are current - update if changed. 3) Verify user permissions (Reader on subscription, Security Administrator or Owner on resource group, write on target resource). 4) Check KQL invocation history with Kusto Data Explorer using appropriate time range to identify invocation errors.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

### Phase 5: Mdc Alerts
> Sources: ado-wiki

**1. Customer needs to automatically dismiss specific MDC security alerts based on conditions (e.g., command line content matching)**

- **Solution**: Create Logic App: 1) MDC Alert trigger 2) Initialize variable for target entity 3) Condition to match value 4) POST to REST API dismiss endpoint with dynamic System Alert Id 5) Enable managed identity with Contributor/Security Admin role 6) Add to MDC workflow automation. Use REST API: Alerts - UpdateSubscriptionLevelStateToDismiss (2022-01-01). Get location first via Get-AzSecurityLocation.
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`

### Phase 6: Automation Rules
> Sources: ado-wiki

**1. Sentinel Automation rule entity condition not matching expected incidents despite correct entity values appearing in Log Analytics (e.g. command line with hidden extra whitespace)**

- **Root Cause**: Log Analytics normalizes/cleans entity text for display (e.g. collapses double spaces to single space), but the raw entity data contains hidden characters or extra whitespace. The automation rule condition matches against the raw data, causing mismatches
- **Solution**: Create a plain LogicApp Playbook with only the Microsoft Sentinel Incident trigger (no additional blocks). Create a temporary Automation Rule pointing to this playbook with non-entity conditions. When triggered, check the Logic App Runs History > Show Raw Outputs to inspect raw JSON entity data for hidden characters. Fix the automation rule condition based on actual raw data, then delete the temporary playbook and rule
- ✅ **21V/Mooncake**: Applicable
- `[Score: 🟢 8.5/10 — ADO Wiki]`
