### 1. Vulnerability scan detects legacy cipher suite (3DES EDE CBC / TLS_RSA_WITH_3DES_EDE_CBC_SHA) on sto

**分数**: 🟢 9 | **来源**: [MCVKB] | **置信度**: high
**21V 适用**: ✅ 是

**根因**: Azure Storage enforces TLS version at HTTP application layer, not TCP/TLS network layer. Port scanning tools (SSL Labs, etc.) probe TCP layer and see legacy cipher suites because Azure Storage is a multi-tenant service maintaining backward compatibility at the network layer.

**方案**: Expected behavior - safe to ignore scan result. With TLS 1.2 enforced, any request using insecure TLS version gets HTTP 4xx error. Doc: https://docs.microsoft.com/azure/storage/common/transport-layer-security-configure-minimum-version. Advise customer: (1) TLS 1.2 enforcement protects data in transit, (2) scan result reflects network-layer capability not actual policy, (3) Azure Policy + Azure Monitor can enforce/monitor TLS across accounts.

**标签**: TLS, vulnerability-scan, cipher-suite, 3DES, private-endpoint, security, false-positive

### 2. Cannot identify client IP for storage audit: client behind private link shows IPv6 address in Jarvis

**分数**: 🟢 9 | **来源**: [MCVKB] | **置信度**: high
**21V 适用**: ✅ 是

**根因**: Private Link or Service Endpoint translates client IP to IPv6. Mapping back requires cross-referencing NRP data.

**方案**: Step 1: Get client IPv6 from Jarvis. Step 2: ASC Storage Account Tools convert IPv6 to IPv4. Step 3: Query NRP Geneva 293FC026 to find NIC. Step 4: ASC find VM by NIC.

**标签**: IPv6, IPv4, private-link, audit, client-IP, ASC, NRP

### 3. When attempting to add a Cloud Witness to a Windows Server 2016 Failover Cluster with TLS 1.2 enable

**分数**: 🔵 7.5 | **来源**: [KB] | **置信度**: medium
**21V 适用**: ✅ 是

**根因**: In several scenarios, SSL 3.0, TLS 1.0, or TLS 1.1 are being used in Azure Storage Account or SQL Server. SSL 3.0 is no longer supported on Azure Storage Account; a TCP connection reset is observed if configured this way. TLS 1.0 and TLS 1.1 are also considered weak protocols.

**方案**: Enable TLS 1.2: Open an elevated PowerShell session. Run the following command to enable TLS 1.2: [Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12 Configure Cloud Witness: Obtain the access key from your Azure Storage Account. Replace <accesskey> in the command below with the access key you obtained. Run the following command to configure the Cloud Witness: Set-ClusterQuorum -CloudWitness -AccountName "Name" -AccessKey "<Accesskey of Azure Storage Account>" Ensure Strong Protocol Configuration: It is recommended to use strong and secure protocol configurations between source and destination endpoints to ensure successful TLS negotiation. Note: The enablement of TLS 1.2 via PowerShell is only valid for that specific PowerShell session. If you need to reconfigure the Cloud Witness for any reason, you will need to repeat the above commands.

**标签**: Content Idea Request, contentidea-kb

### 4. Imagine a scenario where an administrator operates a legacy website that requires an ActiveX control

**分数**: 🔵 7.5 | **来源**: [KB] | **置信度**: medium
**21V 适用**: ✅ 是

**根因**: The issue arises because Windows 8.1, Windows Server 2012 R2, Windows 10, Windows Server 2016, and later versions of Windows natively support TLS 1.2 for client-server communications over WinHTTP. ActiveX Installer only initiates the TLS handshake with TLS 1.2, but the legacy non-windows application server only supports TLS 1.0 and TLS 1.1, causing the handshake failure with error SEC_E_ALGORITHM_MISMATCH.

**方案**: To enforce TLS 1.0, TLS 1.1, and TLS 1.2, verify the value of the DefaultSecureProtocols registry setting. For example: HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Internet Settings\WinHttp\
DefaultSecureProtocols = (DWORD): 0xAA0

HKEY_LOCAL_MACHINE\SOFTWARE\Wow6432Node\Microsoft\Windows\CurrentVersion\Internet Settings\WinHttp\
DefaultSecureProtocols = (DWORD): 0xAA0
Note:&nbsp;TLS 1.0 and 1.1 are weak protocols, please consider recoding the application which supports higher TLS protocols.

**标签**: unknown, contentidea-kb
