# Specification: Patrol new-case mode 应优先走 IR-first 路径

**Issue:** ISS-247

## Summary

Patrol new-case mode 在检测到 Case 已通过 phone call 完成 IR 但无工程师邮件时，没有走 IR-first 路径，而是直接 troubleshoot。应优先生成 IR 邮件草稿。

## Problem Description

### 当前行为（错误）

1. Case 通过 phone call 完成了 Initial Response（D365 记录了 phone call activity）
2. 工程师没有发过任何邮件（emails.md 中无 outbound 邮件）
3. Assess LLM 将状态判定为 `pending-engineer`
4. **但没有同时输出 `email-drafter(initial-response)`**，只输出了 `troubleshooter`
5. 结果走了标准路径（rule #2: 已有 IR → troubleshooter only），跳过了 IR 邮件

### 根因分析

Assess 决策规则第 5 条：
> 新 case（`new` 状态，无工程师邮件）+ 需初始排查 → troubleshooter + email-drafter(initial-response)

此规则的隐含前提是"new 状态"，但 **phone call IR 的 case 可能已不是 `new` 状态**（已有 phone call activity → assess 可能判定为 `pending-engineer`）。同时，规则中 "无工程师邮件" 的检查没有与 "phone call IR" 场景显式对齐：
- Phone call IR ≠ 工程师已发邮件
- Phone call IR 的 case 仍然需要一封正式的工程师邮件作为 Initial Response

### 期望行为

1. Assess 检测到：Case 有 phone call IR 记录 **但** 工程师没有发过邮件 → 应走 IR-first
2. Actions 应同时输出 `troubleshooter` + `email-drafter(initial-response)`
3. Act 路由：先生成 IR 邮件草稿 → 再 troubleshoot → reassess → phase2 email
4. 工程师先发 IR 邮件，后续根据 troubleshooter 的 request-info 回复具体信息

### 期望流程

```
phone call IR + 无工程师邮件
  → assess: actualStatus=pending-engineer, actions=[troubleshooter, email-drafter(initial-response)]
  → read-plan.sh: IR_FIRST=1
  → act: IR-first 路径 → email(IR) → ts → reassess → email(phase2)
```
