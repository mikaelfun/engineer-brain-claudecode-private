# Specification: Phase Retrospective — test-loop 阶段回顾机制

**Track ID:** phase-retro_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft

## Summary

在 test-loop 每个 phase 完成后新增 Phase Retrospective 步骤，让 agent 自主回顾执行结果，识别框架逻辑 bug，并升级到 fixQueue 由 FIX 阶段 opus agent 修复。将自愈架构从"规则驱动"升级为"推理驱动"。

## Context

当前自愈架构有三个检测机制，全部是规则驱动：
- `pattern-detector.sh` — 只看 fixQueue 中的失败测试，正则匹配已知签名
- `auto-heal` — 只处理 5 种预定义异常（data_integrity, stale_progress, orphan_lock, executor_crash, unknown）
- `learnings` — 记录经验但不触发修复

这导致框架逻辑 bug 无法被捕获。实际案例：issue-scanner.sh 的 `[^a-z0-9 ]` 正则过滤中文字符导致 GENERATE 阶段 100% 假阳性，系统已检测到但无升级路径到 fixQueue。

## User Story

As a test-loop framework, I want to reflect on my own execution results after each phase so that I can identify logic bugs in my own code and self-fix them through the existing FIX pipeline.

## Acceptance Criteria

- [ ] AC1: test-loop SKILL.md 新增 Step 2.2 Phase Retrospective，位于 Step 2 和 Step 2.1 之间
- [ ] AC2: Retrospective 通过 agent 推理（非脚本规则）识别异常模式，输出结构化判断
- [ ] AC3: 发现逻辑 bug 时自动执行三步：learnings-writer + fixQueue framework item(priority=1, retroContext) + self-heal-recorder
- [ ] AC4: 每轮每阶段最多创建 1 个 retro fix item（防刷屏）
- [ ] AC5: Retrospective 本身不修改代码，只创建 fix item，实际修复由 FIX 阶段完成
- [ ] AC6: retroContext 包含 phase/round/anomaly/rootCause/targetFile/targetLine 字段，为 FIX 阶段提供精准上下文
- [ ] AC7: FIX 阶段能识别 retroContext 并利用其中的定位信息修复框架代码

## Dependencies

- 依赖现有 test-loop SKILL.md 的 Step 2/2.1 结构
- 依赖现有 FIX 阶段的 framework fix 路径（已支持 category=framework）
- 依赖 learnings-writer.sh、self-heal-recorder.sh、state-writer.sh

## Out of Scope

- 不修改 auto-heal 的安全边界（supervisor 仍不直接改 executors）
- 不修改 pattern-detector.sh 逻辑（保留现有规则检测作为补充）
- 不实际修复 issue-scanner.sh 的中文 bug（由 test-loop 自愈验证）
- 不新增 bash 脚本（纯 SKILL.md prompt 层面的增强）
