# Specification: Fix Cancel Track Spinner Persists

**Track ID:** cancel-spinner-fix_20260322
**Type:** Bug
**Created:** 2026-03-22
**Status:** Draft
**Source:** ISS-061

## Summary

After cancelling track creation on the Issues page, the TrackProgressPanel's "Creating Track..." spinner persists instead of immediately disappearing.

## Context

The Issues page uses a multi-source boolean gate `showProgress` to control TrackProgressPanel visibility:
```
showProgress = isTracking || hasTrackMessages || isOptimisticTracking
```

The cancel handler at line 1177 calls:
```typescript
cancelTrack.mutate(issue.id)  // POST to backend → resets issue.status to 'pending'
clearTrackMessages(issue.id)  // Zustand store cleanup
```

## Problem Description

**Root cause:** `isTracking` is derived from `issue.status === 'tracking'` which reads from React Query cache. After cancel, the mutation invalidates `['issues']` query, but the refetch is async — `isTracking` remains `true` during the refetch window, keeping `showProgress = true`.

**Secondary issue:** The recovery effect (ISS-055, lines 681-687) uses `useActiveTrackSessions()` which queries backend unified sessions. After cancel, if the backend session status hasn't cleared from the query cache yet, `hasBackendSession` remains true and re-arms `isOptimisticTracking`, causing a flicker-back.

**Steps to reproduce:**
1. On Issues page, click "New Track" for any issue
2. Wait for "Creating Track..." spinner to appear
3. Click "Cancel" button
4. Observe: spinner continues indefinitely

**Expected:** Spinner disappears immediately after Cancel click.

## Acceptance Criteria

- [ ] Cancel click immediately hides TrackProgressPanel (no spinner persistence)
- [ ] Recovery effect (ISS-055) does not re-arm tracking after cancel
- [ ] Existing track creation flow unaffected (create → complete still works)

## Dependencies

None — modifies only frontend state management.

## Out of Scope

- Backend cancel route changes (already works correctly)
- SSE cancel sentinel handling (already works correctly via `__cancelled__`)

## Technical Notes

Fix approach: Add optimistic status update in `useCancelTrack` mutation's `onMutate` callback to immediately set issue.status to 'pending' in the query cache, making `isTracking` false instantly. Also invalidate `['unified-sessions']` query to prevent recovery effect re-arming.
