# Specification: Fix Issues Page Grouping with Pagination

**Track ID:** issue-page-group_20260321
**Type:** Bug
**Created:** 2026-03-21
**Status:** Draft
**Source:** ISS-026

## Summary

Issues 页面第二页的 issues 没有被正确分类到 pending/tracked/done 三个分组。原因是分页在服务端执行（只返回当前页 15 条），分组在客户端执行（只能对当前页的 issues 分组），导致第二页丢失正确的分组视图。

## Problem Description

### Steps to Reproduce
1. 打开 Dashboard Issues 页面
2. 确保 issues 总数 > 15（当前 pageSize）
3. 点击"下一页"到第二页
4. 观察分组情况

### Expected Behavior
所有 issues 按 pending/tracked/done 正确分组，分页应在分组之后执行（即先分组，每组内的 issues 列表再分页，或整体先分组再整体分页）。

### Actual Behavior
第二页只展示服务端返回的 15 条 issues 的分组结果，导致分组不完整、不正确。

### Root Cause
- 后端 `GET /api/issues` 接收 `page` + `pageSize` 参数，先 filter 再 slice 返回一页数据
- 前端收到一页数据后，再按 status 分成 pending/tracked/done 三组
- 分页先于分组 → 第二页的分组只是该页 15 条的局部分组

## Acceptance Criteria

- [ ] Issues 页面所有 issues 正确按 pending/tracked/done 分组显示
- [ ] 分组在分页之前执行（即分组看到的是全量数据，不受分页影响）
- [ ] 当 issues 数量多时，仍有分页机制避免页面过长（客户端分页）
- [ ] 搜索、status filter、type filter 仍然正常工作
- [ ] 页面加载性能不受明显影响（issues 通常 < 100 个）

## Dependencies

- 后端路由: `dashboard/src/routes/issues.ts`
- 前端页面: `dashboard/web/src/pages/Issues.tsx`
- 前端 hook: `dashboard/web/src/api/hooks.ts`
- 数据服务: `dashboard/src/services/issue-reader.ts`

## Out of Scope

- 虚拟滚动或无限加载
- 每个分组独立分页（过度设计）
- Issues 数据库迁移

## Technical Notes

修复方案：**全量获取 + 客户端分页**
1. 后端：新增不分页的 API（或去掉分页参数时返回全量）
2. 前端：一次获取全量 issues，先按 status 分组，再在整体列表上做客户端分页
3. 由于 issues 数量通常 < 100，全量获取无性能问题
