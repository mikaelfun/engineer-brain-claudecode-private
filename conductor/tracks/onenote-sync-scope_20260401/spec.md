# Specification: OneNote Sync 范围记忆 + Content Hash 增量校验

**Track ID:** onenote-sync-scope_20260401
**Type:** Bug
**Created:** 2026-04-01
**Status:** Draft

## Summary

onenote-export sync 模式缺少导出范围记忆，导致部分导出的 notebook 被全量拉入不需要的 section。同时 OneNote COM hierarchy 的 lastModifiedTime 在两次调用间不稳定（时间戳倒退），导致增量检测大量误判。

## Problem Description

**问题 1：导出范围记忆缺失**
- sync 模式对每个 notebook 只传 `-NotebookName`（无 filter）
- 之前用 `-SectionPath` 部分导出的 notebook，sync 时被全量拉入不需要的 section
- 例：只导出了 `MCVKB/Identity`，sync 后 General、Storage 等全部 section 也被导出

**问题 2：timestamp 不稳定导致假增量**
- OneNote COM hierarchy 的 `lastModifiedTime` 在两次调用间不一致（时间戳倒退）
- 实测 324/407 pages 被重导出，但 git diff 显示仅 frontmatter `modified:` 行变化，内容没变
- 增量检测依赖 timestamp 完全相等，任何抖动都导致全量重导

## Acceptance Criteria

- [ ] 每次导出（含 link/section/full）后，在 notebook 输出目录写入 `.export-scope.json` 记录导出范围
- [ ] sync 模式读取 `.export-scope.json`，按记录的 sections 传 `-SectionPath` 给脚本
- [ ] 无 `.export-scope.json` 的 notebook，sync 时提示用户确认范围（全量 or 选择 sections）
- [ ] timestamp 不同时，脚本计算本地文件 content hash 与新导出内容的 hash 比较
- [ ] hash 相同 → 仅更新 frontmatter `modified:` 和 `content_hash:`，不重写正文（快速路径）
- [ ] hash 不同 → 真正重导出（正常路径）
- [ ] frontmatter 增加 `content_hash:` 字段，存储正文部分的 hash

## Dependencies

- `.claude/skills/onenote-export/scripts/export-onenote.ps1` — PowerShell 导出脚本
- `.claude/skills/onenote-export/SKILL.md` — Sync 模式流程定义

## Out of Scope

- OneNote COM API 本身的 timestamp 不稳定根因修复
- RAG sync 逻辑改动（hash 相同时 RAG 不需要重 ingest）
- 全新 notebook 的自动发现（仍需手动首次导出）

## Technical Notes

- `.export-scope.json` 存放在每个 notebook 输出目录（如 `OneNote Export/MCVKB/.export-scope.json`）
- content hash 使用 MD5 或 SHA-256，取 frontmatter 之后的正文部分计算
- PowerShell 中可用 `[System.Security.Cryptography.MD5]::Create()` 计算 hash
- hash 比较需要在 `GetPageContent()` 之后、写文件之前进行（已经付出了 API 调用成本）
- 对于 timestamp 相同的页面（快速路径），不调用 `GetPageContent()`，直接 SKIP（保持现有行为）
