# Specification: fix-analyzer.sh 增加根因追溯，减少误判 env issue

**Track ID:** fix-depth_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft
**Issue:** ISS-135

## Summary

当前 fix-analyzer.sh 将 timeout/auth_issue/service_down 直接标记为 IS_ENV_ISSUE=true 并 skip。实际这些可能是代码配置问题（如 MCP 未传递）。需要增加 Root Cause Trace 步骤：从 test result 提取 API endpoint → 找到 handler → 检查 session spawn 配置 → 如果是配置缺失则降级为 config_missing(code_bug)。

## Context

fix-analyzer.sh 是 FIX 阶段的分析器，负责将测试失败分类为不同类型。当前将 timeout、auth_issue、service_down 三种 failure 直接归类为"环境问题"并 skip，不生成修复建议。但实际上很多 timeout/auth_issue 是因为 MCP server 未正确配置（如 ISS-134 的 mcpServers 缺失），属于代码/配置问题而非临时环境问题。这导致真正的 bug 被 skip 而无人修复。

## User Story

As a test-loop operator, I want fix-analyzer to trace root cause of timeout/auth failures before marking them as env issues, so that configuration bugs are correctly identified and fixed.

## Acceptance Criteria

- [ ] AC1: timeout/auth_issue/service_down 不再直接标记为 env issue
- [ ] AC2: Root Cause Trace 检查 API handler 和 session spawn 配置
- [ ] AC3: 配置缺失问题正确归类为 config_missing (code_bug)
- [ ] AC4: 真正的临时环境问题仍标记为 env issue
- [ ] AC5: analysis.md 输出包含 trace 路径

## Dependencies

- 依赖 fix-analyzer.sh 现有分类逻辑
- 依赖 Dashboard 源码结构（API handler → session spawn 配置链）
- 依赖 case-session-manager.ts 的配置结构

## Out of Scope

- 不修改 FIX 阶段其他 executor（fix-generator.sh, fix-applier.sh）
- 不修改 TEST 阶段的测试执行逻辑
- 不自动修复发现的配置问题（仅正确分类）

## Technical Notes

- 修改 `tests/executors/fix-analyzer.sh`
- 新增 failure type: `config_missing`（归类为 code_bug，不是 env_issue）
- Root Cause Trace 路径：test result → API endpoint → route handler file → session spawn config → MCP config
- trace 结果写入 analysis.md 的 "Root Cause Trace" section
