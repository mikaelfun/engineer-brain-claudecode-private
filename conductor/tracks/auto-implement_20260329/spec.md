# Specification: env.yaml 增加 autoImplementTracked 开关

**Track ID:** auto-implement_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft
**Issue:** ISS-137

## Summary

test-loop SCAN 发现 tracked（有 trackId+plan）但未 implemented 的 issue 时，默认跳过。增加 env.yaml 开关 `autoImplementTracked: false`，开启后 test-loop 自动 spawn `conductor:implement` 实现 tracked issue。默认关闭（保守策略）。

## Context

当 issue 通过 `/conductor:new-track` 创建了 track 后，状态变为 tracked。但实际实现需要手动执行 `/conductor:implement`。在 test-loop 自动化场景下，如果有多个 tracked issue，每个都需要人工触发实现，效率低。增加开关允许 test-loop 自动实现 tracked issue，实现后下轮 SCAN 自动生成验证测试，形成完整闭环。

## User Story

As a test-loop operator, I want an option to auto-implement tracked issues so that the test loop can achieve full automation from issue → implement → test → verify.

## Acceptance Criteria

- [ ] AC1: env.yaml 包含 autoImplementTracked 配置项（默认 false）
- [ ] AC2: autoImplementTracked=false 时跳过 tracked issue（记录 note directive）
- [ ] AC3: autoImplementTracked=true 时自动 spawn conductor:implement
- [ ] AC4: 实现完成后 issue 状态正确更新
- [ ] AC5: SKILL.md 文档清晰描述此功能

## Dependencies

- 依赖 env.yaml 配置加载机制
- 依赖 `/conductor:implement` skill
- 依赖 issue-scanner.sh（ISS-133）发现 tracked issue
- 依赖 issue JSON 的 status/trackId 字段

## Out of Scope

- 不修改 conductor:implement 本身的逻辑
- 不自动创建 track（只自动实现已有 track 的 issue）
- 不处理 pending issue（无 trackId 的不触发）

## Technical Notes

- env.yaml 新增字段：`autoImplementTracked: false`
- 触发条件：issue status=tracked + trackId 存在 + track 有 plan.md
- 实现方式：SCAN 阶段（或 issue-scanner.sh 后的处理逻辑）检测到 tracked issue → spawn conductor:implement {trackId}
- 实现后 issue status 变为 implemented，下轮 SCAN 时 issue-scanner.sh 会提取其 AC 生成测试
