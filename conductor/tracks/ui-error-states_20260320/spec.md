# Specification: UI Error States Fix

**Track ID:** ui-error-states_20260320
**Type:** Bug
**Created:** 2026-03-20
**Status:** Draft

## Summary

Fix 2 minor UI issues found during comprehensive Playwright testing: (1) CaseDetail page shows infinite "Loading case..." spinner for non-existent case IDs instead of an error state, (2) Agent Monitor page triggers 404 console errors on `/api/agents/patrol-state` when no patrol has ever been run.

## Context

These bugs were discovered during the post-implementation UI testing of the `dashboard-ux-fix_20260320` track. Both are edge-case error handling gaps that don't affect normal usage but degrade the user experience in specific scenarios.

## Problem Description

### Bug 1: Non-existent Case — Infinite Loading Spinner

**Steps to reproduce:**
1. Navigate to `/case/NONEXISTENT-99999` (any invalid case number)
2. Page shows "Loading case..." spinner indefinitely

**Expected:** Show an error state (e.g., "Case not found") with a back button
**Actual:** Infinite spinner — the API returns `{"error":"Invalid case number"}` with status 400, but the frontend `useCase` hook doesn't surface this error to the UI

### Bug 2: Patrol State 404 on Fresh Install

**Steps to reproduce:**
1. Start the dashboard on a fresh workspace (no patrol ever run)
2. Open browser DevTools → Console
3. Observe: `404 /api/agents/patrol-state` error on every Dashboard page load

**Expected:** API returns `{}` or `{"state": null}` with status 200
**Actual:** API returns 404 because `casehealth-state.json` file doesn't exist yet

## Acceptance Criteria

- [ ] Navigating to `/case/<invalid-id>` shows an error state ("Case not found") instead of infinite loading

- [ ] `/api/agents/patrol-state` returns 200 with empty state when no patrol state file exists
- [ ] No 404 console errors on fresh install Dashboard load

None — both fixes are isolated to specific error paths.
