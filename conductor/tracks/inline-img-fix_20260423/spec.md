## Summary

CaseDetail 页面中 Email Tab 和 Teams Tab 渲染 markdown 时，inline 图片（内嵌截图、日志截图等）无法正常显示。后端路由和前端渲染逻辑代码均已存在，但实际效果仍有问题，需要诊断定位具体断点并修复。

## Context

Dashboard CaseDetail 页面是 case 数据的主要展示界面。Email 和 Teams 是工程师排查 case 时最重要的信息源，其中 inline 图片（客户截图、错误截图、日志截图等）包含大量诊断信息。

现有代码状态：
- **后端**：`GET /api/cases/:id/images/:filename` (email) 和 `GET /api/cases/:id/teams/assets/:filename` (teams) 路由已存在
- **前端 Email**：`renderBody()` 有 `images/xxx` → `/api/cases/:id/images/xxx` 路径重写 + AuthImage 渲染
- **前端 Teams**：`rewriteImagePaths()` 有 `./assets/xxx` → `/api/cases/:id/teams/assets/xxx` 路径重写，MarkdownContent 中 AuthImage 处理 `/api/` URL

关联 Issue：
- ISS-188：Email 内联图片不显示 — 缺少 images/ 路由（pending，但路由已被添加）
- ISS-226：Teams chat 图片本地化（tracked，track teams-image-local_20260417 已完成）

## Problem Description

用户在 WebUI CaseDetail 页面看到 email 和 teams 中的 inline 图片未正常显示。具体原因不确定，需要排查以下可能断点：

### Email 图片
1. Email markdown 中图片标记格式是否与 `renderBody()` 的正则 `/(!\[.*?\]\(images\/[^)]+\))/` 匹配
2. `cleanBody()` 是否在图片标记到达 `renderBody` 之前将其剥离
3. 图片文件是否实际存在于 `{caseDir}/images/` 目录
4. AuthImage 组件的 JWT auth header 是否正确传递
5. 后端返回的 Content-Type 和文件读取是否正确

### Teams 图片
1. Teams markdown 中图片引用格式是否与 `rewriteImagePaths()` 的正则匹配（目前只匹配 `./assets/xxx`）
2. 图片文件是否实际存在于 `{caseDir}/teams/assets/` 目录
3. data-refresh 阶段是否成功下载了 hostedContents 图片
4. MarkdownContent 中的 AuthImage 路径处理是否正确

## Acceptance Criteria
