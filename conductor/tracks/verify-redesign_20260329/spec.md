# Specification: Conductor Verify 重新设计 — 推理驱动 + Recipe 自我演进

**Track ID:** verify-redesign_20260329
**Type:** Feature (Enhancement)
**Created:** 2026-03-29
**Status:** Draft

## Summary

重新设计 `conductor:verify`，从规则驱动的固定两步（npm test + Playwright UI）改为**推理驱动 + Recipe 自我演进**架构。LLM 读 spec 推理每条验收标准的验证方案 → 查找/复用 verification-recipes → 执行 → 反思并提取新 recipe。npm test 保留作为 regression guard。

## Context

当前 verify 流程（`conductor/workflow.md` Step 2）的三个核心问题：
1. **CLI/脚本类 track 无法验证** — 只有 npm test + Playwright UI 两种固定路径，dashboard-renderer.sh 等脚本无法覆盖
2. **npm test 只证明没回归，不证明新功能 work** — 单元测试 mock 了外部依赖，不能替代端到端验证
3. **验证经验不沉淀** — 每次 UI 验证都重新探索 Edge/JWT/port 等坑，没有结构化复用

项目已有三层经验沉淀（learnings.yaml / ERRORS.md / playbooks/guides/），但缺少**验证策略**的结构化沉淀。

## User Story

As an Azure Support Engineer using Conductor, I want verify to automatically reason about how to test each acceptance criterion and reuse proven recipes, so that verification covers all track types (UI/CLI/API/file) and gets smarter over time without manual recipe writing.

## Acceptance Criteria

- [ ] AC1: verify 读取 spec.md 提取验收标准，对每条推理验证类型（cli/api/ui/file/manual）
- [ ] AC2: `playbooks/guides/verification-recipes/` 目录结构建立，含 `_index.md` + 4 个初始 recipe（web-ui-playwright / cli-script-output / api-endpoint / file-content-check）
- [ ] AC3: verify 执行时先查 `_index.md` 匹配 recipe，有匹配则按 recipe 步骤执行
- [ ] AC4: 无匹配 recipe 时，LLM 从零生成验证脚本并执行
- [ ] AC5: 执行后反思提取 — 满足触发条件（重试>=2次 / 遇坑 / 脚本>10行 / recipe 过时）时自动创建或更新 recipe
- [ ] AC6: npm test 作为 Step 1 regression guard 保留
- [ ] AC7: 最终汇总报告 per-criterion PASS/FAIL with evidence，结果写入 metadata.json
- [ ] AC8: `conductor/workflow.md` 的 Step 2 更新为新流程

## Dependencies

- 现有 `conductor/workflow.md` 验证流程
- `tests/learnings.yaml` 和 `.learnings/ERRORS.md`（现有经验源，用于初始 recipe 编写）

## Out of Scope

- test-loop 集成变更（保持现有 test-loop 架构不动）
- WebUI verify 按钮行为变更（仅改 CLI 流程）
- 修改 npm test 本身的配置

## Technical Notes

- Recipe 文件放在 `playbooks/guides/verification-recipes/`（与现有 guides 同级）
- verify 新流程是 5 步：regression guard → 推理方案 → 执行 → 汇总 → 反思提取
- 反思提取有明确触发条件，不是每次都提取（顺利 1 次通过不提取）
- `_index.md` 作为 recipe 匹配入口，LLM 先读此文件判断
