# Specification: Graph API Token 过期后无法自动刷新

**Track ID:** graph-token-refresh_20260422
**Type:** Bug
**Created:** 2026-04-22
**Status:** Draft
**Issue:** ISS-238

## Summary

Token Daemon 的 Graph API token stale 检测存在 bug，导致过期 token 被反复写入 cache，形成 "expired → refreshed → expired" 死循环。手动点击 Dashboard Refresh 按钮也无法修复，因为它依赖相同的 daemon 刷新逻辑。

## Problem Description

### 现象
- Dashboard Token 状态卡片中 Graph token 持续显示 "Expired"
- 其他 token（Teams/ICM/DTM/D365）正常
- 点击 Refresh 按钮后 Graph token 仍然无法恢复
- daemon log 显示每分钟循环：`[graph] Token expired, refreshing...` → `[graph] Token refreshed (len=3718)`

### Steps to Reproduce
1. 等待 Graph API token 自然过期（~55 分钟 TTL）
2. 观察 Dashboard Token 状态卡片
3. Graph token 显示 Expired，且不会自动恢复
4. 点击 Refresh 按钮无效

### Root Cause Analysis

`token-daemon.js` 60s 轮询中的 stale 检测逻辑（第 795-801 行）存在两个 bug：

**Bug 1: expiresOn 空字符串绕过检测**
```javascript
// extractFromLocalStorage 返回 { secret, expiresOn: val.expiresOn || '' }
// 当 MSAL entry 没有 expiresOn 字段时，返回空字符串 ''（falsy）
if (tokenData.expiresOn) {  // ← '' 是 falsy，跳过
  isStale = ...;
}
```

**Bug 2: fallback 检查错误的属性名**
```javascript
else if (tokenData.token) {  // ← localStorage 用的是 .secret，不是 .token
  const jwtExp = getJwtExp(tokenData.token);  // 永远不会执行
}
```

**结果：** `isStale` 始终为 `false` → 过期 token 直接写入 cache → 下一轮 `isCacheValid` 再次判定 expired → 死循环。

### 更深层问题
即使 stale 检测触发了 `page.reload()`，OWA 的 MSAL 可能不会在 localStorage 中立即生成新 token。需要更可靠的刷新机制。

## Acceptance Criteria

- [ ] Token stale 检测覆盖所有 tokenSource 类型（localStorage 和 request-intercept）
- [ ] 当 localStorage 中的 token 过期时，daemon 能通过 page reload + MSAL 触发获取新 token
- [ ] 如果 reload 后 MSAL 仍未刷新 token，有 fallback 机制（如清除 localStorage MSAL 缓存强制重新认证）
- [ ] 添加 max-retry 机制，避免无限 stale → reload → extract → stale 死循环
- [ ] 手动点击 Dashboard Refresh 按钮能有效触发 Graph token 刷新
