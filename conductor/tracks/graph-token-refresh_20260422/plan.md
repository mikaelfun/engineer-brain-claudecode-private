# Implementation Plan: Graph API Token 过期后无法自动刷新

## Overview

修复 token-daemon.js 中 localStorage token 的 stale 检测逻辑，使其覆盖 `expiresOn` 缺失/空字符串 + `secret` vs `token` 属性名不匹配两个 bug。增加 MSAL 缓存清除 fallback 和 max-retry 防护。

## Phase 1: 修复 Stale 检测逻辑

修复 token-daemon.js 60s 轮询中的 stale 检测，确保 localStorage tokens（Graph/Teams）的过期检测覆盖所有场景。

### Tasks

- [x] Task 1.1: 修复 stale 检测 — 增加 JWT exp fallback for `tokenData.secret`
  - 在 `tokenData.expiresOn` 检查之后，增加 `else if (tokenData.secret)` 分支
  - 对 `tokenData.secret` 调用 `getJwtExp()` 解码 JWT exp claim
  - 确保 request-intercept tokens (`tokenData.token`) 和 localStorage tokens (`tokenData.secret`) 都能通过 JWT exp 检测 stale
- [x] Task 1.2: 增强 `extractFromLocalStorage` — 同时查找最新的未过期 token
  - 当 localStorage 中存在多个匹配 token 时，优先返回 expiresOn 最大（最新）的那个
  - 如果所有匹配 token 都已过期，返回 expiresOn 最大的（让 stale 逻辑处理 reload）

### Verification

- [ ] 验证修改后，当 localStorage 中只有过期 token 时，stale 检测能正确触发

## Phase 2: 增强 Stale Token 恢复机制

当 page.reload() 后 MSAL 仍未在 localStorage 中生成新 token 时，增加 fallback 机制。

### Tasks

- [x] Task 2.1: stale reload 后二次检测 + MSAL 缓存清除 fallback
  - 在现有 reload → re-extract 失败后，增加：清除 localStorage 中匹配的 MSAL AccessToken 条目 → reload → 等待 MSAL silent renew → 第三次 extract
  - 这会强制 MSAL acquireTokenSilent 用 refresh_token 获取新 access_token
- [x] Task 2.2: 增加 stale retry 计数器 + max-retry 防护
  - 为每个 token 维护 `staleRetryCount`
  - 连续 3 次 stale retry 失败后，降低检查频率（每 5 分钟才重试一次而非每分钟）
  - daemon log 中记录 retry 计数，方便诊断

### Verification

- [ ] 验证在极端情况下（MSAL refresh_token 也过期时），daemon 不会无限重试
