# Specification: 新增 issue-scanner.sh 扫描 issues/ 目录生成测试

**Track ID:** issue-scanner_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft
**Issue:** ISS-133

## Summary

spec-scanner.sh 只扫 conductor tracks，不扫 issues/ 目录。需要新增 issue-scanner.sh 作为 SCAN 第7步，按 issue 状态过滤：tracked/implemented（有 trackId+spec）→ 提取 AC 作为 gap；done → 检查回归覆盖；pending → SKIP。

## Context

当前 SCAN 阶段通过 spec-scanner.sh 扫描 conductor tracks 的 spec.md 提取 AC 作为测试 gap。但很多需求以 issue 形式存在于 `issues/` 目录，这些 issue 关联了 conductor track 后有完整的 spec，却不会被 SCAN 发现。导致 tracked/implemented 的 issue 对应功能没有自动生成测试覆盖。

## User Story

As a test-loop operator, I want SCAN to also scan issues/ directory so that tracked/implemented issues get automatic test coverage based on their linked spec's ACs.

## Acceptance Criteria

- [ ] AC1: issue-scanner.sh 扫描 tracked/implemented issue 并提取 AC gap
- [ ] AC2: pending issue 被跳过（无 trackId 或 trackId 对应的 spec 不存在）
- [ ] AC3: done issue 检查回归测试覆盖
- [ ] AC4: SCAN 阶段调用 issue-scanner.sh 并将 gap 追加到 state.json
- [ ] AC5: 输出格式与 spec-scanner.sh 一致: ISSUE_GAP|{issueId}|{criterion}|{category}

## Dependencies

- 依赖 `issues/*.json` 文件格式（id, status, trackId 字段）
- 依赖 conductor tracks 的 `spec.md` AC 格式
- 依赖 `tests/registry/*.json` 用于比对已有覆盖
- 依赖 state-writer.sh 原子写入

## Out of Scope

- 不修改 spec-scanner.sh 的逻辑
- 不修改 issue JSON 的格式
- 不处理没有 trackId 的 issue（pending 状态直接 SKIP）
