# Specification: TestLab Activity Stream UX Improvement

**Track ID:** activity-stream-ux_20260405
**Type:** Feature
**Created:** 2026-04-05
**Status:** Draft
**Issue:** ISS-214

## Summary

Improve TestLab Activity Stream by replacing raw SSE event names with human-readable progress summaries, and add a Clear button to allow users to reset the activity list on demand.

## Context

The TestLab page includes an ActivityStream component that displays real-time SSE events from the test-loop pipeline. Currently it shows raw event types like `state-updated`, `pipeline-updated` with minimal context (queue counts like `T:5 F:2 V:1 R:0`), making it difficult for operators to quickly understand test progress. Additionally, there is no way to clear accumulated entries, so old noise persists throughout the session.

## User Story

As a test operator, I want the Activity Stream to show meaningful progress summaries and allow clearing old entries, so that I can quickly understand test status without noise.

## Acceptance Criteria

- [ ] Activity Stream messages display human-readable progress summaries (e.g., "Round 3 SCAN completed, found 2 new issues") instead of raw event names
- [ ] A Clear button exists that one-click clears all currently displayed activity entries
- [ ] After clearing, newly produced events still display normally
- [ ] Timestamps and phase badges are preserved; only the main content text is improved for readability

## Dependencies

- Existing ActivityStream component in `dashboard/web/src/pages/TestLab.tsx` (lines 1046-1228)
- SSE events from `dashboard/src/watcher/sse-manager.ts` and `dashboard/src/watcher/file-watcher.ts`
- Event types: test-state-updated, test-pipeline-updated, test-discoveries-updated, test-result-updated, test-evolution-updated, runner-status-changed

## Out of Scope

- No changes to backend SSE event structure or file watching mechanism
- No persistent storage of activity history
- No changes to other TestLab components (PhasePipeline, RunnerControl, etc.)

## Technical Notes

- The transformation from raw events to readable summaries should happen on the frontend (event-to-message mapping function)
- Clear button resets the local `useState<ActivityEvent[]>()` array to empty
- Consider adding a `formatActivityMessage(event: ActivityEvent): string` utility that maps each event type + data payload to a human-readable sentence
- The ring buffer on the backend (1000 events) and `GET /api/tests/recent-events` remain unchanged
