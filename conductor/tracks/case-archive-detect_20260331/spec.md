# Specification: Patrol/Casework Case Archive & Transfer Detection

**Track ID:** case-archive-detect_20260331
**Type:** Feature
**Created:** 2026-03-31
**Status:** Draft
**Issue:** ISS-099

## Summary

patrol 和 casework 应自动检测本地 `cases/active/` 中的 case 是否已在 D365 中归档（Resolved/Cancelled）或转移（不再分配给当前工程师），并将对应 case 目录移入 `archived/` 或 `transfer/` 文件夹。

## Context

当前 patrol 通过 `list-active-cases.ps1` 获取 D365 active case 列表，但不会与本地 `cases/active/` 目录交叉对比。已归档或转移的 case 仍残留在 `active/` 目录，占用巡检资源，且可能产生误导性操作。

已有基础设施：
- `list-active-cases.ps1` — 查询 `statecode eq 0` 的 active case（OData API）
- `cases/archived/` — 已存在的归档目录（手动移入）
- D365 Entity Model — `statecode=1` (Resolved) / `statecode=2` (Cancelled)

## User Story

As an Azure Support Engineer, I want patrol and casework to automatically detect archived/transferred cases so that my active case workspace stays clean and I don't waste time on cases I no longer own.

## Acceptance Criteria

- [ ] AC1: 新建 `list-closed-cases.ps1` 脚本，查询 D365 `statecode=1` (Resolved) 或 `statecode=2` (Cancelled) 且原分配给当前工程师的 case
- [ ] AC2: patrol 步骤 2 获取 active case 列表后，对比本地 `cases/active/` 目录，识别不在 active 列表中的 case
- [ ] AC3: 对不在 active 列表的 case，查询 closed list 确认状态：Resolved/Cancelled → `archived/`；找不到（不在 active 也不在 closed）→ `transfer/`
- [ ] AC4: 辅助证据链：检查 case 最新工程师邮件是否为 closure 邮件，作为关闭状态的辅证（D365 状态为主）
- [ ] AC5: 自动将 case 目录从 `active/` 移入 `archived/` 或 `transfer/`
- [ ] AC6: casework 单 case 处理时，在 data-refresh 后检测当前 case 是否已不在 active list，如已归档/转移则提前终止并归档
- [ ] AC7: 移动前记录日志（case number、原因、目标目录），移动后不再对该 case 执行后续 casework 步骤

## Dependencies

- `list-active-cases.ps1` — 已存在，查询 active cases
- `_init.ps1` — D365 API 初始化（Invoke-D365Api 函数）
- D365 OData API — 需确认 closed case 查询能力（`statecode ne 0` + `assignedto` 过滤）
- `emails.md` — 邮件历史文件，用于 closure 邮件辅证

## Out of Scope

- Dashboard UI 展示 archived/transferred cases（后续 track）
- 自动恢复误归档的 case（手动移回 active/）
- 归档时清理 casehealth-state.json 中的旧条目

## Technical Notes

- D365 OData 查询 closed cases：使用与 `list-active-cases.ps1` 相同的 FetchXML 模式，将 `statecode eq 0` 改为 `statecode ne 0`
- 文件移动使用 `mv`（Bash）或 `Move-Item`（PowerShell），需确保目标目录存在
- `transfer/` 目录目前不存在，需新建
- 检测逻辑应在 patrol 的步骤 2-3 之间插入（获取 active list 后、开始 casework 前）
- casework 检测应在 data-refresh 完成后、compliance-check 之前
