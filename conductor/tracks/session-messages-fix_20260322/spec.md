# Specification: Fix Agent Monitor Session Messages Not Displaying

**Track ID:** session-messages-fix_20260322
**Type:** Bug
**Created:** 2026-03-22
**Status:** Draft

## Summary

Expanding a case session row in AgentMonitor shows no messages. Root cause is threefold: (1) `formatMessageForSSE` fails to extract content from structured SDK messages (content is an array), (2) persisted message types use raw SDK names instead of frontend-expected names, (3) chat route doesn't persist messages and always broadcasts as `case-session-thinking`.

## Problem Description

When a user expands a case session in the AgentMonitor page, the detail panel shows "Waiting for messages..." or empty message bubbles because:

1. **Empty content in persistence**: `formatMessageForSSE()` does `message.content || message.text || message.result || ''` — but SDK `assistant` messages have `content` as an array of content blocks (e.g., `[{type: 'text', text: '...'}]`). Since an array is truthy, it passes through, but then `typeof formatted.content === 'string'` is false → persisted as `''`.

2. **Wrong type mapping in persistence**: The persistence call stores `formatted.messageType` which is the raw SDK type (`assistant`, `tool_use`, `tool_result`) instead of the frontend-expected types (`thinking`, `tool-call`, `tool-result`). The SSE event names ARE correctly mapped by `getSSEEventType()`, but the persisted type field is not.

3. **Chat route doesn't persist**: The `POST /case/:id/chat` route broadcasts SSE events but never calls `appendSessionMessage()`, so messages from chat interactions are lost on page refresh.

4. **Chat route ignores event type**: The chat route always broadcasts as `case-session-thinking` instead of using `getSSEEventType(message)` to determine the correct SSE event type.
