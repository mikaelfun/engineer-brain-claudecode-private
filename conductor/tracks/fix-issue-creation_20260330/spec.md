# Specification: Test-loop FIX 阶段发现产品 Bug 自动创建 Issue

**Track ID:** fix-issue-creation_20260330
**Type:** Feature
**Created:** 2026-03-30
**Status:** Draft

## Summary

在 test-loop FIX 阶段完成修复后，由 LLM 推理判断发现的是产品缺陷还是测试基础设施问题。产品缺陷自动创建 Issue 到 issues/ 目录，实现 test-loop → Issue Tracker 的反向链路。

## Context

当前 test-loop 发现 bug 后只记录到 discoveries.json 和 fixes/ 目录（44 条记录），不回写 issues/。链路是单向的（Issues → Tests），修完的产品 bug 不可追溯。需要推理驱动（非 grep/keyword 规则）的分类判断，符合 EngineerBrain 推理驱动架构优化方向。

## User Story

As a support engineer, I want test-loop to automatically create trackable issues when it discovers real product bugs, so that product-level defects are visible in the Issue Tracker and don't get lost after being fixed.

## Acceptance Criteria

- [ ] FIX 阶段每次修复完成后，LLM 基于修复内容、修改文件路径、错误性质推理分类：infrastructure vs product bug
- [ ] 分类为 product bug 时，自动创建 Issue JSON 到 issues/ 目录（格式与 /issue 创建的一致）
- [ ] 创建的 Issue 包含来源标记字段：discoveredBy, round, testId, fixRef
- [ ] Issue 的 type=bug, priority 由 LLM 推理判断（P0/P1/P2）
- [ ] discoveries.json 中对应条目新增 issueId 字段，建立双向关联
- [ ] 不重复创建：同一 testId 已关联 Issue 的不再创建
