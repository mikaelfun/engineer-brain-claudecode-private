# Specification: Teams Search Graph API Parallel Fetch

**Track ID:** teams-search-graph_20260422
**Type:** Feature
**Created:** 2026-04-22
**Status:** Draft
**Issue:** ISS-236

## Summary

将 teams-search-inline.sh 的消息拉取从 Agency ListChatMessages 切换为 Graph API，与 Agency Search 并行执行，减少串行等待时间。Agency 仅保留搜索职能，Graph API 成为消息拉取的主路径。

## Context

当前 teams-search-inline.sh 的流程是串行的：
1. 启动 Agency proxy（~5s）
2. Agency Search 搜索 case number（~3s）
3. Agency ListChatMessages 并行拉取消息（~7s，受限 max_workers=3）
4. 总耗时 ~15s（含 proxy 启动）

经 2026-04-22 验证：
- Graph API `GET /me/chats/{chatId}/messages` ✅ 可用（OWA token，`Chat.Read` scope）
- Graph API Search ❌ 不可用（缺 `ChannelMessage.Read.All`）
- 大多数场景搜索到的 chatId 已在本地缓存，新增 chatId 概率低

## User Story

As a support engineer running data-refresh, I want Teams message fetch to run faster so that patrol completes more quickly across multiple cases.

## Acceptance Criteria

- [ ] 缓存 chatId 的消息通过 Graph API 拉取，与 Agency Search 并行启动
- [ ] Agency Search 发现的新 chatId 通过 Graph API 补拉
- [ ] Graph API 失败时自动回退 Agency ListChatMessages
- [ ] _mcp-raw.json 输出格式与现有格式兼容（build-input-from-raw.py 无需改动或仅微调）
- [ ] 端到端耗时在有缓存 chatId 时 ≤ 现有耗时

## Dependencies

- Token Daemon 管理的 OWA Graph token（`$TEMP/graph-api-token.json`）
- 现有 `_mcp-raw.json` → `_relevance.json` 缓存链
- Agency `SearchTeamMessagesQueryParameters`（搜索不可替代）

## Out of Scope

- 替换 Agency Search（Graph Search API 不可用，已验证）
- 修改 build-input-from-raw.py 的核心逻辑（仅适配 Graph API 响应格式差异）
- Token Daemon registry 变更（graph token 已有）

## Technical Notes

- Graph API 响应格式：`{ "value": [ { "id", "from", "body", "createdDateTime", ... } ] }`
- Agency MCP 响应格式：`{ "result": { "content": [{ "text": JSON-string }] } }` 内嵌 `{ "messages": [...] }`
- 需要一个统一的消息格式转换层，或在 _mcp-raw.json 写入时标准化
- Graph API 限流：30 rps/app/tenant，3 workers 并行安全
- Agency proxy 启动仍需要保留（搜索依赖），但可延迟到 Graph fetch 开始后再启动
