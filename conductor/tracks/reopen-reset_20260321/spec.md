# Specification: Reopen Issue 应清空旧 Track 信息回到 Pending 状态

**Track ID:** reopen-reset_20260321
**Type:** Bug
**Created:** 2026-03-21
**Status:** Draft

## Summary

Reopen 一个 done/implemented issue 时，应该清空关联的 track 信息（trackId、plan、spec 等），让 issue 回到 `pending` 状态，使其可以重新 Create Track 生成全新的评估和计划。当前实现仅清除 verification 而保留 trackId，导致 reopen 后 issue 状态变为 `implemented` 而非 `pending`，无法重新走 track 创建流程。

## Context

Issue 的生命周期：`pending → tracking → tracked → in-progress → implemented → done`。当用户发现修复不完善需要 reopen 时，意味着之前的 track（spec + plan）可能不再适用，需要重新评估问题并生成新的修复计划。当前 reopen 只清除 verification 保留旧 track，导致用户只能在旧计划上继续 implement，无法重新审视问题。

## Problem Description

**现象：** Reopen 一个 done/implemented issue 后：
- 如果有 trackId，status 变为 `implemented`（从 track metadata 派生），而非 `pending`
- UI 显示 "Mark Done" / "Verify" / "Reopen" 按钮，没有 "Create Track" 按钮
- 用户无法为这个 issue 创建新的 track 来重新评估

**根因：** 后端 `/reopen` 路由对有 trackId 的 issue 只清除 track 的 verification，不清除 trackId 本身。而 `deriveIssueStatus()` 从 track metadata 的 `status: "complete"` 派生出 `implemented` 状态。

**期望行为：** Reopen 应该：
1. 清除 issue JSON 中的 `trackId`
2. 将 issue status 设为 `pending`
3. 旧 track 目录保留（不删除，作为历史参考），但 tracks.md 中标记为归档
4. issue 回到可以 "Create Track" 的状态

## Acceptance Criteria

- [ ] Reopen done/implemented issue 后，issue status 变为 `pending`
- [ ] Reopen 后 issue JSON 中 `trackId` 被清除
- [ ] Reopen 后 UI 显示 "Create Track" 按钮（而非 Implement/Mark Done）
- [ ] 旧 track 的 metadata.json 中标记 `status: "archived"` 或类似状态
- [ ] tracks.md 中旧 track 的状态标记更新为 `[A]`（archived）
- [ ] 新建的 track 与旧 track 独立，不互相影响

## Dependencies

## Out of Scope

- 删除旧 track 目录（保留作为历史记录）
- 自动迁移旧 track 的排查结果到新 track
- 修改 track 创建流程本身的逻辑
