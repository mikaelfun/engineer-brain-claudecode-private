# Specification: Issue 与 Track 状态统一 — Track metadata 作为单一事实源

**Track ID:** track-status-unify_20260321
**Type:** Refactor
**Created:** 2026-03-21
**Status:** Draft

## Summary

Issue 绑定 trackId 后，状态不再双写 issue JSON 和 track metadata，改为由 track metadata 作为单一事实源。后端 GET /api/issues 返回时自动从 track metadata 派生 effectiveStatus 和 verification 信息。写操作只更新 track metadata。

## Context

当前 issue 和 track 两个文件各自维护状态，每次状态转换都在 issues.ts 路由中 `updateIssue(id, { status: '...' })` 双写，容易不一致。例如 implement 完成时写 issue `implemented` + track `complete`，两边独立写入如果一边失败就不一致。

Track metadata 已有 `status` 字段（pending/in_progress/complete），缺少：
- `issueId` 双向关联
- `verification` 验证阶段信息（status/method/result）

Issue JSON 的 `status` 和 `verifyResult` 在有 trackId 时完全可以从 track metadata 派生。

## User Story

As a developer, I want issue status to be derived from a single source of truth (track metadata) so that state never becomes inconsistent between issue JSON and track metadata.

## Acceptance Criteria

- [ ] TrackMetadata 类型新增 `issueId?: string` 和 `verification?: TrackVerification` 字段
- [ ] `GET /api/issues` 返回的 issue，有 trackId 时其 status 和 verifyResult 从 track metadata 派生
- [ ] `POST /:id/start-implement` 只写 track metadata（status: in_progress），不写 issue status
- [ ] `POST /:id/verify` 完成后把 verifyResult 写入 track metadata.verification，不写 issue JSON
- [ ] `POST /:id/mark-done` 写 track metadata（verification: { status: 'skipped', method: 'manual' }），不写 issue status
- [ ] `POST /:id/reopen` 清 track metadata 的 verification 字段，不写 issue status
- [ ] `POST /:id/create-track` 完成后在 track metadata 写入 issueId
- [ ] 前端无感知变化 — 返回的 issue 对象结构不变（status + verifyResult 由后端派生填充）
- [ ] 无 trackId 的 issue（pending/tracking）仍用 issue JSON 自身的 status

## Dependencies

## Technical Notes

### 状态派生映射
```
track.status = "pending"/"planned"  → issue effectiveStatus = "tracked"
track.status = "in_progress"        → issue effectiveStatus = "in-progress"
track.status = "complete"           → 看 verification:
  verification 无 / pending         → "implemented"
  verification.status = "running"   → "implemented"（保持）
  verification.status = "passed"    → "done"
  verification.status = "skipped"   → "done"（manual mark done）
  verification.status = "failed"    → "implemented"
```

### TrackVerification 类型
```ts
interface TrackVerification {
  status: 'pending' | 'running' | 'passed' | 'failed' | 'skipped'
  method?: 'auto' | 'manual'
  result?: {
    unitTest: { success: boolean; output: string }
    uiTest: { success: boolean; output: string }
    verifiedAt: string
  }
}
```

### 关键实现点
- `listIssues()` 和 `getIssue()` 返回纯 JSON 数据，不做派生
- 在 issues.ts 路由的 GET 处理中，对有 trackId 的 issue 做 `enrichIssueFromTrack(issue)` 填充 effectiveStatus
- 写操作改为只调 `updateTrackMetadata()`，不再调 `updateIssue(id, { status: ... })`
- issue JSON 中 `status` 字段对有 trackId 的 issue 变成 "最后已知状态"（作为 fallback，track metadata 读取失败时使用）
