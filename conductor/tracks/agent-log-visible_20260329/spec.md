# Specification: Dashboard API 返回 agent executionSummary

**Track ID:** agent-log-visible_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft
**Issue:** ISS-136

## Summary

Dashboard API response 增加 executionSummary 字段：包含 toolCalls(success/fail)、mcpServers(实际可用列表)、agentType、turns。让测试可以断言 MCP 可用性和 tool 调用成功率，将 agent 执行从黑盒变为白盒。

## Context

当前 spawned agent 的内部执行对 API 消费者是黑盒。测试只能检查最终 HTTP response 的 body 内容，无法断言：agent 是否成功连接了 MCP server、哪些 tool 调用成功/失败、agent 执行了多少轮。这导致测试诊断困难——一个 "response 内容不完整" 的失败可能是 MCP 连接失败、tool 调用超时、或 agent turns 不够等多种原因。

## User Story

As a test developer, I want API responses to include execution summary data so that I can diagnose agent failures by checking MCP availability, tool success rates, and turn counts.

## Acceptance Criteria

- [ ] AC1: API response 包含 executionSummary 字段
- [ ] AC2: executionSummary 包含 toolCalls 数组（每个 tool call 的 success/fail）
- [ ] AC3: executionSummary 包含 mcpServers 列表（实际可用的 MCP）
- [ ] AC4: executionSummary 包含 agentType 和 turns
- [ ] AC5: 不影响现有 API 消费者（additive change）

## Dependencies

- 依赖 Claude Agent SDK 的 session result 结构（需确认可提取哪些执行数据）
- 依赖 case-session-manager.ts session 执行流程
- 依赖 Dashboard API route 定义

## Out of Scope

- 不修改 agent 执行逻辑本身
- 不添加实时 streaming execution data
- 不存储历史 execution summary（仅返回当次执行）

## Technical Notes

- 定义 `ExecutionSummary` TypeScript 接口
- 修改 `case-session-manager.ts`，在 session 执行完成后从 SDK result 提取数据
- 修改 API route handler，将 summary 附加到 response
- 接口设计：
  ```typescript
  interface ExecutionSummary {
    agentType: string;
    turns: number;
    mcpServers: string[];
    toolCalls: { name: string; success: boolean; error?: string }[];
  }
  ```
