# Specification: Fix Case Title Truncation When Title Contains Pipe Characters

**Track ID:** title-truncate-fix_20260320
**Type:** Bug
**Created:** 2026-03-20
**Status:** Draft
**Source:** ISS-021

## Summary

Case titles containing `|` characters are truncated in the Dashboard Case Detail header because the markdown table parser regex `([^|]*)` stops at the first pipe in the value.

## Problem Description

**Steps to reproduce:**
1. Open Case 2602130040001034 in Dashboard Case Detail page
2. Observe the title displayed in the header

**Expected:** Full title: "Unified <Sev B> | PRC | Azure container registry | ACR requirements so that image push or pull is restricted to the production environment"

**Actual:** Title shows only "Unified <Sev B>" — everything after the first `|` in the title value is lost.

**Root Cause:**
In `dashboard/src/services/case-reader.ts`, `extractTableValue()` uses:
```typescript
const regex = new RegExp(`\\|\\s*${key}\\s*\\|\\s*([^|]*)\\|`, 'i')
```
The capture group `([^|]*)` stops at the first `|`, but D365 case titles commonly contain pipe characters as delimiters (e.g., `Unified <Sev B> | PRC | ...`).

The `case-info.md` file stores title in a markdown table row:
```
| Title | Unified <Sev B> | PRC | Azure container registry | ... |
```
Since the value itself contains `|`, the regex captures only the first segment.

## Acceptance Criteria

- [ ] Case titles containing `|` are displayed in full on the Case Detail header
- [ ] Other fields parsed from case-info.md tables still work correctly
- [ ] The fix handles edge cases: title with multiple `|`, title as last row, title with no `|`

## Dependencies

None

## Out of Scope

- Changing how case-info.md is generated (upstream format stays the same)
- Fixing truncation in other systems outside the dashboard

## Technical Notes

**Approach options:**
1. **Change parser**: Instead of `([^|]*)`, capture everything between the key column's `|` and the row's final `|` (last pipe on the line). E.g., greedy match up to the last `|` on the line.
2. **Change data format**: Escape pipes in case-info.md when generating it. But this requires changing upstream scripts and is out of scope.

Recommended: Option 1 — fix the regex to capture from the key's trailing `|` to the line's final `|`.
