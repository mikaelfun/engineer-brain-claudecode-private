# Specification: NoteGap 新 Case 日终 Note 生成

**Track ID:** notegap-newcase-note_20260402
**Type:** Feature
**Created:** 2026-04-02
**Status:** Draft
**Issue:** ISS-208

## Summary

note-gap 对当天新建的 Case 不生成 Note，应在日终自动生成一条基础操作记录 Note 草稿。

## Context

当前 note-gap 的检测逻辑（SKILL.md Step 3-4）依赖 `lastNoteDate` 与当前时间的差值判断 gap。对于新 Case（当天创建、无任何人工 note 或只有系统自动分配 note），要么 `lastNoteDate` 不存在，要么 gap 天数为 0（不满足阈值），导致直接跳过，永远不会生成 note 草稿。

但新 Case 在当天结束时仍需要记录当天做了什么，包括：Meet IR、发了 First Quality Response、做了哪些排查操作。

## User Story

As a support engineer, I want note-gap to detect new cases with no notes and generate a day-summary note draft, so that I don't forget to log basic operations (IR, first response, troubleshooting) at end of day.

## Acceptance Criteria

- [ ] AC1: 新 Case（当天创建、notes.md 中无人工 note 或只有系统 note）触发 note-gap 时，生成一条基础 note 草稿
- [ ] AC2: 草稿内容从 case-summary.md 和 casehealth-meta.json 提取当天操作（IR 状态、是否发了邮件、排查进展）
- [ ] AC3: 草稿格式遵循现有 Step 5 用户偏好（`fangkun note` title、`YYMMDD--` 格式）
- [ ] AC4: 对已有人工 note 的 Case，行为不变（仍走原 gap 阈值逻辑）

## Dependencies

- 现有 note-gap SKILL.md（修改对象）
- case-summary.md 和 casehealth-meta.json（数据源）

## Out of Scope

- Dashboard UI 修改
- 后端 API 修改
- 批量 note-gap 检测逻辑修改（ISS-204 范围）
- D365 写入逻辑修改

## Technical Notes

修改点仅在 SKILL.md 的 Step 3 和 Step 4 之间插入新的分支判断路径：
- Step 3 解析 notes 时，如果无人工 note → 标记 `isNewCase = true`
- Step 4 判断时，`isNewCase` 直接进入生成流程（跳过 gapDays 阈值检查）
- Step 6 读取进展时，除了 case-summary.md 还额外读取 casehealth-meta.json 的 IR/email 状态
