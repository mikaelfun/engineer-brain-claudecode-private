# Specification: AI Assistant Chat Queue During Step Execution

**Track ID:** chat-queue_20260322
**Type:** Feature
**Created:** 2026-03-22
**Status:** Draft
**Source:** ISS-083

## Summary

Allow users to type and send chat messages while a step is executing in the AI Assistant. Messages are queued locally in the frontend and auto-sent sequentially after the step completes, rather than being blocked by the disabled input.

## Context

The AI Assistant uses a unified Claude SDK session for both step execution and freeform chat (both call `chatCaseSession` for resume). Currently, the frontend disables the chat input whenever `isStepActive` (step is `active` or `waiting-input`), forcing users to wait until the step finishes before typing. The backend uses a per-case operation lock and rejects concurrent requests with 409.

## User Story

As an Azure Support Engineer, I want to type follow-up messages while a step is running so that I don't lose my train of thought waiting for the step to finish.

## Acceptance Criteria

- [ ] Chat input remains enabled during step execution (not disabled when `isStepActive`)
- [ ] Messages sent during step execution are shown in chat with a "queued" visual indicator
- [ ] Queued messages are auto-sent sequentially when the step completes (session status → `paused`)
- [ ] If multiple messages are queued, they are sent one-by-one in order
- [ ] Queue is cleared and messages update to "sent" status after processing

## Dependencies

Depends on existing `CaseAIPanel.tsx`, `case-session-manager.ts`, and the per-case SSE event flow (`case-step-completed`).

## Out of Scope

- Backend queue mechanism (this is a frontend-only queue)
- Concurrent SDK session calls (still sequential — queue waits for step to finish)
- Queue persistence across page refreshes

## Technical Notes

**Approach: Frontend-only queue**
1. Remove the `disabled={isStepActive}` condition from chat input
2. Add a `queuedMessages` state array in `CaseAIPanel`
3. When user sends during an active step, push to queue instead of calling API
4. Show queued messages in the chat stream with a "⏳ Queued" badge
5. Listen for step completion (SSE `case-step-completed` or status change to `paused`)
6. On completion, drain queue by calling `POST /case/:id/chat` for each message sequentially
7. Update message status from "queued" → normal after each send
