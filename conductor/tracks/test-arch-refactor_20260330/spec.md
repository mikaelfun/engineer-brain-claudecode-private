# Specification: Test Architecture Refactor — Pipeline Model

**Track ID:** test-arch-refactor_20260330
**Type:** Refactor
**Created:** 2026-03-30
**Status:** Draft

## Summary

Refactor the test-supervisor/test-loop architecture from confusing nested loop model (Round/Runner Loop/Test-Loop/Phase) to a clear Pipeline Model (Cycle/Tick/Stage) with unified dashboard presentation. Preserve the 2-agent Observer-Subject architecture for self-healing capability while simplifying naming, state management, and dashboard monitoring.

## Context

The current test automation framework has:
- **4 overlapping concepts**: Round, Runner Loop, Test-Loop, Phase — causing confusion
- **Dashboard shows two parallel pipelines**: Runner Steps (5 dots) and Phase Pipeline (5 stages) without clear relationship
- **Monolithic state.json (200+ lines)**: mixing pipeline progress, queue management, stats, audit log, runner status, and skip registry
- **Naming is misleading**: "test-loop" doesn't loop, "runner" implies it just runs things (actually it's an observer/healer)

The architecture's **core value** — reasoning-driven self-discovery, self-healing, and self-evolution — is correct and must be preserved. The Observer (supervisor) must remain separate from the Subject (executor) for the self-healing pattern to work.

## Motivation

- Clearer mental model for developers and dashboard users
- Dashboard should showcase the framework's intelligence and self-healing capability
- State files should be small, focused, and fast to read
- Naming should communicate the architectural intent

## Technical Notes

- **2-agent model preserved**: Supervisor (observer/healer) + Stage Worker (executor)
- **Dashboard design**: Intelligence Layer (Reasoning Steps as narrative) above Stage Pipeline (progress bars)
- **Smart folding**: Normal tick = one-line summary; Self-heal = expanded reasoning detail
- **State split**: Each file has single responsibility, Dashboard reads specific files for specific UI areas
- **Backward compatibility**: state-writer.sh should support both old and new format during transition
