# Specification: State Tracking Authoritative Status

**Track ID:** state-authority_20260330
**Type:** Bug
**Created:** 2026-03-30
**Status:** Draft
**Issue:** ISS-180

## Summary

pipeline.json and supervisor.json lack authoritative `status` fields to indicate whether something is actively running. All consumers (dashboard-renderer, WebUI SSE, health-check) rely on heuristic inference, causing incorrect status display (e.g., showing RUNNING when idle).

## Context

The test supervisor framework uses split state files (pipeline.json, supervisor.json, queues.json, stats.json) managed via `state-writer.sh`. Stage-worker and supervisor-runner agents write state transitions as they execute. Dashboard-renderer.sh and WebUI SSE endpoints read these files to display status.

## Problem Description

**Root Causes:**

1. **pipeline.json has no idle concept** — `currentStage` is always one of SCAN/GENERATE/TEST/FIX/VERIFY, never "idle". It means "what stage comes next", not "what's running now".

2. **Dashboard health detection is heuristic-based** — uses `sup.active` timestamp + 30-min window and `.progress-*.json` file existence to guess if something is running. No authoritative field.

3. **Supervisor step rendering has no idle handling** — renders `reasoning` object keys as step progress, but doesn't detect "all steps done + idle" → shows `🔶Decide` when supervisor already completed + went idle.

4. **WebUI SSE uses same fuzzy inference** — backend reads the same state files with same ambiguous logic.

**Steps to Reproduce:**

1. Run `/test-supervisor run` → let cycle complete → supervisor returns to idle
2. Run `/test-supervisor db` → dashboard shows `🏃 RUNNING` + `🔶Decide`
3. Inspect JSONs: supervisor.json `status=idle, step=null`, pipeline.json `currentStage=FIX, FIX.status=pending`
4. WebUI TestLab page also shows incorrect running indicator

**Expected:** Dashboard shows `✅ IDLE`, supervisor steps all ✅ or hidden
**Actual:** Shows `🏃 RUNNING`, supervisor stuck at `🔶Decide`

## Acceptance Criteria

- [ ] pipeline.json has authoritative `pipelineStatus` field: `idle | running`
- [ ] supervisor.json has authoritative `status` field correctly written: `idle | running`
- [ ] stage-worker writes `pipelineStatus: "running"` on start, `"idle"` on completion
- [ ] supervisor-runner writes `status: "running"` on start, `"idle"` on completion
- [ ] dashboard-renderer reads authoritative fields instead of heuristic inference for health/running detection
- [ ] WebUI SSE endpoint reads authoritative fields for status display
- [ ] health-check.sh reads authoritative fields for diagnostics
- [ ] When both supervisor and stage-worker are idle, dashboard shows IDLE (not RUNNING)

## Technical Notes

- `state-writer.sh --target pipeline --merge` already supports merging arbitrary fields — no changes needed to state-writer itself
- The `pipelineStatus` field is separate from `currentStage` — stage tells "what's next", status tells "is anything executing"
- supervisor.json already has a `status` field but it's not consistently written by the runner
