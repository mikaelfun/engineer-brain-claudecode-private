# Specification: Fix Create Track Question Dialog Not Showing (Round 2)

**Track ID:** track-question-fix2_20260321
**Type:** Bug
**Created:** 2026-03-21

**Source:** ISS-029

## Summary

Create Track from WebUI fails because the interactive question dialog does not appear. The ISS-027 fix (sessionRef pattern + SSE fallback) is applied but insufficient — the bug recurs due to 3 interrelated root causes.

## Problem Description

When clicking "Create Track" on the Issues page, the Claude SDK session needs interactive answers (e.g., confirming spec). The WebUI should show a question dialog, but it often doesn't appear, causing the track creation to time out.

### Steps to Reproduce

1. Open Dashboard Issues page
2. Click "Create Track" on any issue (e.g., ISS-018)
3. Backend starts Claude query for `/conductor:new-track`
4. Claude asks AskUserQuestion (spec confirmation)
5. **Expected:** Question dialog appears in TrackProgressPanel
6. **Actual:** No dialog appears; track creation times out

### Root Cause Analysis

Three interrelated bugs cause this failure:

#### Bug 1: TrackProgressPanel mount race condition
- `TrackProgressPanel` only renders when `issue.status === 'tracking'` OR `hasTrackMessages > 0`
- Issue status update requires React Query invalidation + refetch (async, ~100-500ms)
- If `AskUserQuestion` fires before the component mounts, the SSE `issue-track-question` event is processed by `useSSE` but `TrackProgressPanel` is not yet in the DOM
- The store receives the question, but no component is subscribed to render it
- When the panel eventually mounts, the recovery query may fail (see Bug 3)

#### Bug 2: Backend pendingQuestion not stored when sessionId is missing
- `createCanUseTool()` only calls `issueTrackState.setPendingQuestion()` when `sessionId` exists (line 157-160)
- If `AskUserQuestion` fires before `session_id` is captured from the message stream, backend state has NO pendingQuestion
- Consequence 1: Recovery endpoint `/api/issues/:id/track-progress` returns no `pendingQuestion`
- Consequence 2: Answer submission `/api/issues/:id/track-answer` returns 404 "No pending question"
- The SSE fallback (ISS-027 fix) sends the question to frontend, but backend can't process the answer

#### Bug 3: Recovery logic too restrictive
- `TrackProgressPanel` recovery only restores `pendingQuestion` when `recoveredData.messages.length > 0 && messages.length === 0`
- If the panel mounts after SSE messages were already processed by the store, `messages.length > 0` so recovery is skipped (`hasRecoveredRef.current` guard)
- Or if backend returns `pendingQuestion` but no messages, recovery condition fails

## Technical Notes

### Fix Strategy

1. **Backend: Always store pendingQuestion** — Remove the `if (sessionId)` guard. Store with `sessionId: ''` when not yet available. Backfill sessionId when it becomes available.
2. **Backend: Backfill sessionId** — In `processQueryMessages`, after capturing `session_id`, check if there's a stored pendingQuestion with empty sessionId and update it.
3. **Frontend: Optimistic panel mount** — When the create-track mutation succeeds (200 response), immediately set local state to show TrackProgressPanel, don't wait for SSE/refetch.
4. **Frontend: Fix recovery logic** — Remove the `messages.length === 0` guard; always attempt recovery if `pendingQuestion` exists in response and not yet in store.
5. **Answer endpoint: Handle empty sessionId** — Use the stored pendingQuestion's sessionId (which may have been backfilled) instead of requiring it from frontend.
