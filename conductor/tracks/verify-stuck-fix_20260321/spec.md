# Specification: Fix Verify "Running Tests" Stuck After All Pass

**Track ID:** verify-stuck-fix_20260321
**Type:** Bug
**Created:** 2026-03-21
**Status:** Draft
**Source:** ISS-054

## Summary

Issue Tracker 页面中，verify 运行测试后所有测试结果都已 pass，但 UI 上 "Running Tests..." 的进行中状态没有结束，spinner 仍在转动。应该在所有测试通过后自动更新为完成状态。

## Problem Description

### Steps to Reproduce
1. 在 Issues 页面点击一个 implemented issue 的 Verify 按钮
2. 后台测试执行完毕，所有测试 pass
3. UI 上的 VerifyProgressPanel 显示了 tool-result 消息（测试输出），但 header 仍显示 "Running Tests..." + spinner

### Expected Behavior
- 当 `issue-verify-completed` SSE 事件到达时，`isActive` 变为 `false`，header 切换为 "All Tests Passed ✓" 或 "Tests Failed"

### Actual Behavior
- `isActive` 保持 `true`，header 卡在 "Running Tests..." 不变

### Root Cause Analysis

The `VerifyProgressPanel` header display depends on `isActive` from Zustand store `activeVerify[issueId]`. This flag is only set to `false` when:
1. SSE `issue-verify-completed` arrives → calls `setVerifyActive(issueId, false)`
2. SSE `issue-verify-error` arrives → calls `setVerifyActive(issueId, false)`

The recovery mechanism (`useVerifyStatus`) only runs when `messages.length === 0`, so it won't fire if progress messages were already received.

**Potential failure modes:**
- SSE reconnection loses the `issue-verify-completed` event (EventSource reconnects between the event being sent and being received)
- `useSSE` effect re-runs and creates a new EventSource, missing the completion event on the old one
- `connect()` callback dependency list changes cause reconnection during verify
- Backend sends `issue-verify-completed` but the SSE `data.data` nesting doesn't match what frontend expects (the SSE manager wraps data in an `event` envelope with `{type, data, timestamp}`, but frontend code does `const d = data.data || data`)

## Acceptance Criteria

- [ ] When verify tests all pass, "Running Tests..." transitions to "All Tests Passed ✓" within seconds
- [ ] When verify tests fail, "Running Tests..." transitions to "Tests Failed" within seconds
- [ ] If SSE completion event is missed, a polling/timeout fallback detects completion and updates the UI
- [ ] Page refresh during or after verify correctly recovers state (not stuck as "Running Tests...")

## Dependencies

## Out of Scope

- Changing the VerifyResult data structure

## Technical Notes

Fix should add a polling fallback: when `isActive` is `true` and the panel has messages, periodically poll `GET /api/issues/:id/verify-status` to check if backend has already completed. If `status !== 'active'`, sync the frontend state accordingly.
