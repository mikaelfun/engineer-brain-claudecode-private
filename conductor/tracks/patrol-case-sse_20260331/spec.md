# Specification: Patrol Case SSE Shows Full Casework Process

**Track ID:** patrol-case-sse_20260331
**Type:** Bug
**Created:** 2026-03-31
**Status:** Draft

## Summary

Patrol case monitor SSE only shows patrol-level step names with empty content. Each case should display the full casework execution process (thinking, tool calls, step progress) identical to single-case processing.

## Problem Description

When clicking "Start Patrol" in WebUI, each case's monitor SSE displays only patrol-level information:
- Messages show `step: 'patrol'` but content is empty
- No thinking text, tool call details, or step progress visible
- Agent Monitor shows the case session but messages are empty/minimal

**Expected behavior:** Each case during patrol should display the same detailed SSE messages as when processing a single case via "Full Process" ŌĆö thinking content, tool names with summaries, step-grouped display.

**Root Cause:**

The patrol loop in `_runPatrol()` (case-session-manager.ts:1214-1237) uses:
```typescript
const eventType = getSSEEventType(message)    // ŌåÆ 'case-session-thinking'
const formatted = formatMessageForSSE(message) // ŌåÆ { content: '' } ŌåÉ EMPTY!
```

`formatMessageForSSE` calls `extractContent(message)` which reads `message.content`, but SDK messages have content nested at `message.message.content` (array of content blocks). So extraction returns empty string.

Meanwhile, single-case processing uses `broadcastSDKMessages()` (case-routes.ts:138-250) which correctly:
1. Deep-parses `message.message.content` array
2. Extracts `tool_use`, `text`, `thinking` blocks individually
3. Uses `summarizeToolInput()` for readable tool descriptions
4. Broadcasts as `case-step-progress` events (not `case-session-thinking`)
5. Persists with correct frontend types

**Two issues:**
1. **Content extraction**: `extractContent` doesn't handle nested SDK structure ŌåÆ empty content
2. **Event type**: Patrol uses `case-session-thinking`/`case-session-tool-call` while single-case uses `case-step-progress` with `kind` field ŌåÆ frontend step-grouped display doesn't work for patrol

## Acceptance Criteria

- [ ] During patrol, each case's SSE messages contain actual thinking text, tool names, and tool input summaries (not empty strings)
- [ ] Patrol case messages use the same SSE event type (`case-step-progress`) as single-case processing for frontend consistency
- [ ] Agent Monitor CaseSessionDetail shows the same level of detail for patrol cases as for single-case "Full Process"
- [ ] Patrol step name is preserved (e.g., `step: 'patrol'` or `step: 'casework'`) so messages can be distinguished from user-initiated processing

## Technical Notes

The fix should refactor `_runPatrol`'s message loop to use `broadcastSDKMessages()` (or a shared version of it) instead of the manual `getSSEEventType` + `formatMessageForSSE` approach. This ensures patrol cases get the same deep-parsing, summarization, and event typing as single-case processing.

`broadcastSDKMessages` is currently a local function in `case-routes.ts` ŌĆö it should be extracted to a shared module (e.g., `utils/sdk-message-broadcaster.ts`) so both case-routes and patrol can use it.
