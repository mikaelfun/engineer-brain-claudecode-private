# Specification: Supervisor gate=error 推理驱动恢复

**Track ID:** gate-recovery_20260330
**Type:** Feature
**Created:** 2026-03-30
**Status:** Draft

## Summary

当前 test-supervisor-runner Step 1 对 `gate=error` 一刀切退出，不尝试任何恢复。改为推理驱动恢复：读取 gateReason → 推理错误性质/可用手段/安全性 → 匹配 `recovery-tools.yaml` 注册的恢复工具 → 尝试恢复 → 重新 pre-flight → 仍失败才退出。

## Context

test-supervisor 是测试框架的推理引擎，负责监督 test-loop 执行。Step 1 Pre-flight 检查通过 `pre-flight.sh` 获取 gate 状态，当 `gate=error` 时（如后端未运行导致 health-check 失败），直接跳到 Step 5 输出错误摘要，整轮浪费。

已有先例：`gate=stuck` 时 supervisor 会尝试清理 stale progress 文件再重试。但 `gate=error` 没有类似的恢复逻辑。

同时，后端 restart API (`POST /api/restart/backend`) 已经完整实现（ISS-089, ISS-100），但没有被集成到测试框架的故障恢复流程中。

## User Story

As a test-supervisor runner, I want to 推理判断 gate=error 是否可恢复并自动尝试恢复 so that 环境级故障（如后端未运行）不会浪费整轮 supervisor 执行，且新的恢复能力只需注册工具而非改代码。

## Acceptance Criteria

- [ ] AC1: `gate=error` 时 supervisor 不再一刀切退出，而是进入恢复推理流程
- [ ] AC2: 新增 `tests/recovery-tools.yaml` 注册可用恢复工具（含 name、description、check、action、safety、waitSeconds）
- [ ] AC3: Supervisor 推理三个问题：错误性质（环境/数据/框架）、匹配哪个恢复工具、恢复是否安全
- [ ] AC4: 恢复成功后重新运行 `pre-flight.sh`，通过则正常继续 test-loop
- [ ] AC5: 恢复失败或判断不可恢复 → 仍跳到 Step 5 退出（保留原有兜底）
- [ ] AC6: 恢复动作记录到 learnings（`learnings-writer.sh`）

## Dependencies

- `tests/executors/pre-flight.sh` — gate/gateReason 输出
- `.claude/agents/test-supervisor-runner.md` — Step 1 gate 处理逻辑
- `tests/executors/learnings-writer.sh` — 记录恢复动作
- Dashboard restart API (`POST /api/restart/backend`) — 首个注册的恢复工具

## Out of Scope

1. 不改 `pre-flight.sh` 的 gate 输出逻辑（它仍然输出 `gate=error`，恢复推理在 supervisor 层做）
2. 不改 test-loop 的任何文件（环境恢复是 supervisor 的职责）
3. 不实现具体恢复工具的代码（restart API 已存在，只需注册到 yaml）
4. 不做自动重试上限管理（首版一次恢复尝试，失败即退出）

## Technical Notes

- `recovery-tools.yaml` 为声明式工具注册表，每个工具包含：`name`、`description`（场景描述，供 LLM 推理匹配）、`check`（诊断命令）、`action`（恢复命令）、`safety`（safe/review/dangerous）、`waitSeconds`
- Supervisor 推理匹配用 gateReason 文本 + 工具 description 做语义匹配，不做关键词 if-else
- 首个注册工具：backend-restart（后端没运行时 API 不可用，需 fallback 到进程级启动）
- Immutable Core 保护不受影响（恢复动作不会修改 4 个受保护文件）
