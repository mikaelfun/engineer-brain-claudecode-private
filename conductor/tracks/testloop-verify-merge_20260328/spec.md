## Summary

Test-Loop 和 Conductor:Verify 在测试执行引擎上高度重叠（E2E/API/UI runner），合并为"一个测试引擎 + 一个验证协调者"的架构，消除重复维护。

## Context

EngineerBrain 有两套测试执行系统：
1. **Test-Loop** — 连续循环测试框架（SCAN→GENERATE→TEST→FIX→VERIFY），有成熟的 executor 脚本、assertion 框架、状态机、自愈能力
2. **Conductor:Verify** — Track 实现完成后的一次性验证，有自己的 E2E/Interaction/Visual/API 测试执行逻辑

两者在 E2E、API、UI 测试执行上高度重叠，但各有独特价值。

## Motivation

- 维护两套测试执行逻辑浪费精力且易不一致
- Conductor:Verify 的 spec-driven 测试生成和 unit test 能力是 test-loop 缺失的
- Test-Loop 的状态机、自愈、回归追踪是 conductor:verify 没有的
- 合并后：一个引擎跑所有测试，一个协调者管理 track 生命周期

## Technical Notes

### 并发安全设计
- state.json 保持单写者（仅 test-loop 写）
- conductor:verify 通过 directives.json 提交补测请求（type: "add_tests"）
- 检测 test-loop 运行状态：检查 `.progress-*.json` 存在或 `state.json.currentTest` 非空
- conductor:verify 独立跑 executor 时使用特殊 round 标记（如 `verify-{trackId}`），不干扰 state.json

### Test-Loop 新增能力
- unit-runner.sh：封装 `npm test`，输出标准 `{round}-{testId}.json`
- spec-scanner：SCAN 阶段读取 `conductor/tracks/*/spec.md`，提取验收标准，与 `tests/manifest.json` 比对

### Conductor:Verify 瘦身后流程
1. `npm test` → 自己跑 unit test
2. 查询 `tests/results/` → 检查相关测试是否已 PASS
3. 有未覆盖验收标准 → 写 `directives.json` 或独立跑 executor
4. 全部通过 → 写 `metadata.json` verification 字段
5. 有失败 → 报告未通过项
