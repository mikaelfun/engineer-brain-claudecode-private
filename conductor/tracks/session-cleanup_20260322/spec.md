# Specification: SDK Session 进程清理

**Track ID:** session-cleanup_20260322
**Type:** Bug
**Created:** 2026-03-22
**Status:** Draft

## Summary

Claude Code SDK session 的 endSession()、step cancel、patrol cancel、backend restart 均只更新元数据状态，未调用 SDK 的 abortController.abort() 或 query.close() 终止底层进程，导致 MCP server 子进程树泄漏为僵尸进程。

## Context

Dashboard 后端通过 `@anthropic-ai/claude-agent-sdk` 的 `query()` 函数创建 Claude Code SDK session，每个 session 会 spawn 一个 CLI 子进程，该子进程再启动多个 MCP server（cmd.exe/bash.exe + node.exe）。当 session 在应用层"结束"时，仅更新 JSON 状态文件，底层进程树无人终止。

## Problem Description

### 泄漏路径

1. **endSession()** — 仅设 `status=completed`，不 abort 底层 query
2. **step cancel** — 仅设 cancellation flag + break 消息循环，无 AbortController
3. **patrol cancel** — 仅阻止下一批，不 abort 当前批次 session
4. **restartBackend()** — 直接 `process.exit(0)`，无 shutdown hook
5. **index.ts** — 无 SIGINT/SIGTERM handler

### 观察到的症状

- ~109 个孤儿 cmd/bash 进程不属于任何活跃 Claude session
- 每个进程下挂一个 node.exe（MCP server），大部分 <1MB 但持续占用句柄
- 随使用时间累积，进程数不断增长

### SDK 已有的清理机制

- `Options.abortController?: AbortController` — "When aborted, the query will stop and clean up resources"
- `Query.close(): void` — "forcefully ends the query, cleaning up all resources including pending requests, MCP transports, and the CLI subprocess"

## Acceptance Criteria

- [ ] 每个 query() 调用传入 AbortController，存储在活跃操作注册表中
- [ ] endSession() 调用 abortController.abort() 终止底层进程
- [ ] step cancel 通过 abort 实际终止 SDK query
- [ ] patrol cancel abort 当前批次所有 session
- [ ] index.ts 添加 SIGINT/SIGTERM handler，shutdown 时 abort 所有活跃 session
- [ ] restartBackend() 先 abort 所有活跃 session 再退出
- [ ] 单元测试覆盖 abort 注册/清理逻辑

## Dependencies

- `@anthropic-ai/claude-agent-sdk`（已安装，`Options.abortController` 已有）

## Out of Scope

- SDK 内部进程管理改进
- 已泄漏进程的自动清理（手动杀即可）
- V2 session API 迁移
