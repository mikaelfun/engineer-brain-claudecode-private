# Specification: SCAN 阶段回收 regression/retry/unverified 旧问题

**Track ID:** scan-recycle_20260329
**Type:** Feature
**Created:** 2026-03-29
**Status:** Draft
**Issue:** ISS-132

## Summary

当前 SCAN 阶段只扫描新的未测试功能（gap），不回收已发现但未解决的旧问题（regression/retry-needed/fixed-unverified）。导致 test-loop 在达到 95% 覆盖率后所有 queue 为空、循环空转。需要在 SCAN 阶段开头插入回收检查，将旧问题重新加入 testQueue。

## Context

Test-loop 经过 20 轮运行，发现 26 个问题，其中 11 个 verified、10 个 regression、2 个 retry-needed、3 个 fixed-unverified。SCAN 只检查"有没有新的未测试功能"，当没有新 gap 时 → testQueue 为空 → round++ → 再 SCAN → 仍无 gap → 空转。而 regression/retry/unverified 的问题堆在各自 queue 里永远不会被重新处理。

## User Story

As a test-loop operator, I want SCAN to automatically recycle unresolved issues from previous rounds so that the test loop continues making progress even when no new features are discovered.

## Acceptance Criteria

- [ ] AC1: SCAN 阶段开头检查 regressionQueue，将其中所有项移入 testQueue 重跑
- [ ] AC2: SCAN 阶段开头检查 fixQueue 中 retryCount < 3 的项，回收到 testQueue 重试
- [ ] AC3: SCAN 阶段开头检查 verifyQueue，将未验证的项移入 testQueue 补验证
- [ ] AC4: 回收后清空原 queue 中已移动的项（避免重复处理）
- [ ] AC5: 回收操作在 SCAN 的其他扫描步骤之前执行（Step 0 位置）
- [ ] AC6: state.json phaseHistory 记录回收数量和来源

## Dependencies

- 依赖现有 state.json 结构（regressionQueue、fixQueue、verifyQueue 字段）
- 依赖 state-writer.sh 原子写入

## Out of Scope

- 不修改 FIX/VERIFY 阶段的逻辑
- 不修改 fixQueue 中 retryCount >= 3 的项（已标记为 "需人工"）

## Technical Notes

- 修改位置：`tests/executors/` 或 SKILL.md 的 SCAN 阶段逻辑
- 回收逻辑应在 SCAN 的 Step 0（加载状态）之后、Step 1（扫描端点）之前执行
- 需要通过 state-writer.sh 原子更新 state.json
