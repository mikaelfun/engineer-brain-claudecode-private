# Specification: Implemented Issue Mark Done Button

**Track ID:** mark-done-btn_20260321
**Type:** Feature
**Created:** 2026-03-21
**Status:** Draft
**Source:** ISS-038

## Summary

Add a new `implemented` status to the issue lifecycle and a "Mark Done" button on the Issues page. When an implementation completes, the issue transitions to `implemented` (instead of auto `done`). The user manually clicks "Mark Done" after verifying the result, which sets the issue to `done` and synchronizes the conductor track metadata status to `completed`.

## Context

The EngineerBrain Dashboard Issue Tracker currently auto-transitions issues to `done` when `/conductor:implement` completes successfully. This skips human validation. Users need a manual verification step to confirm the implementation meets expectations before closing the issue.

## User Story

As a support engineer, I want to manually mark an implemented issue as done after verifying the result, so that I have a human-validated confirmation step before closing an issue.

## Acceptance Criteria

- [ ] New `implemented` status added to `IssueStatus` type
- [ ] Implementation completion sets issue status to `implemented` (not `done`)
- [ ] Issues page shows "Mark Done" button for `implemented` status issues
- [ ] Clicking "Mark Done" updates issue JSON status to `done`
- [ ] Clicking "Mark Done" also updates conductor track `metadata.json` status to `completed`
- [ ] `implemented` issues appear in a distinct visual group on the Issues page
- [ ] Status badge for `implemented` has appropriate styling (e.g., blue/teal)

## Out of Scope

- Batch mark-done (marking multiple issues done at once)
- Notifications on mark-done
- Undo mark-done (reopen already exists as a separate action)
- Changing the verify endpoint behavior

## Technical Notes

- The `IssueStatus` type in `dashboard/src/types/index.ts` needs to add `'implemented'`
- The `start-implement` route's success handler should set `implemented` instead of `done`
- A new `POST /api/issues/:id/mark-done` endpoint follows the existing pattern (like `reopen`)
- The mark-done endpoint must write both: issue JSON status + conductor track `metadata.json`
- The conductor-reader service needs a new `updateTrackMetadata()` function
- Frontend status grouping logic needs to handle the new `implemented` status (group with `tracked` or create a new group)
