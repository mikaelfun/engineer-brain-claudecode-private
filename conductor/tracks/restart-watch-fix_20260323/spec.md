# Specification: Fix Restart Service --watch Mode Process Kill & Spawn Mismatch

**Track ID:** restart-watch-fix_20260323
**Type:** Bug
**Created:** 2026-03-23
**Status:** Draft
**Source:** ISS-100

## Summary

restart-service.ts 的 spawn 命令与 package.json dev 脚本不一致：后端用 `npx tsx src/index.ts`（无 --watch）而 dev 脚本用 `node --import tsx/esm --watch src/index.ts`（有 --watch）。且杀进程时只杀端口占用的子进程，无法杀掉 --watch 模式的 watcher 父进程，导致端口冲突（EADDRINUSE）。

## Problem Description

### Root Cause

`node --watch` 模式的进程树结构：
```
watcher 父进程 (PID A) ← 不监听端口，netstat 找不到
  └── 实际执行子进程 (PID B) ← 监听 3010 端口
```

restart-service 的 `killSavedAndPort()` 逻辑：
1. 杀 `lastBackendPid`（仅在 restart service 自己 spawn 的进程有效，对 `npm run dev` 启动的无效，因为 PID 未记录）
2. 杀端口占用的 PID → 杀的是子进程 B
3. watcher 父进程 A 存活 → 检测到子进程退出或文件变化 → 重新 spawn 新子进程 → 占端口

### Symptoms

1. 点击 BE 重启按钮 → EADDRINUSE（watcher 重新 spawn 了子进程）
2. 重启后 spawn `npx tsx src/index.ts` 无 `--watch`，丢失热重载
3. 前端类似：`npx vite --port 5173` vs `cd web && npm run dev`（cwd 不一致）

### Steps to Reproduce

1. 用 `npm run dev`（或 `npm run dev:server`）启动后端（--watch 模式）
2. 在 Dashboard 点击 BE 重启按钮
3. 观察 EADDRINUSE 错误

## Technical Notes

- Windows 环境，进程杀掉用 `taskkill /F /T /PID`
- `/T` 已有但只在知道父 PID 时有效
- 对于 `npm run dev` 启动的进程，`lastBackendPid` 为 null（不是 restart service spawn 的），需要通过端口向上追溯进程树
- 方案：找到端口占用的子进程 PID → `wmic process get ParentProcessId where ProcessId=X` → 杀父进程树
