# Specification: Fix Issues Page & Agent Monitor State Sync

**Track ID:** issue-session-sync_20260322
**Type:** Bug
**Created:** 2026-03-22
**Status:** Draft

## Summary

Issues page doesn't show TrackProgressPanel for issues that have active backend track-creation sessions when the frontend missed SSE events (page not loaded during session start, or page refresh).

## Context

EngineerBrain Dashboard has two pages that display track-creation session state:
- **Agent Monitor** — reads from `/sessions/all` API which aggregates backend in-memory `issueTrackState`
- **Issues page** — relies on local Zustand `issueTrackStore` populated by SSE events, plus `issue.status` from JSON files

When track creation starts, the backend sets `issueTrackState` and emits SSE events. The Issues page uses SSE to populate local state. But if the page wasn't loaded when creation started (or was refreshed), SSE events are missed, and the `showProgress` gate blocks the TrackProgressPanel from mounting.

## Problem Description

**Steps to reproduce:**
1. Navigate to Issues page, note ISS-045 is `pending`
2. Via another mechanism (CLI or API), start track creation for ISS-045
3. Switch to Agent Monitor — shows ISS-045 session as "Awaiting input"
4. Switch back to Issues page — still shows `pending`, no TrackProgressPanel visible

**Expected:** Issues page should detect the active backend session and show TrackProgressPanel with recovery data.

**Actual:** Issues page shows `pending` with no progress panel. User cannot interact with the pending question.

**Root Cause:**
```typescript
// Issues.tsx IssueRow component — line 688
const showProgress = isTracking || hasTrackMessages || isOptimisticTracking
```
- `isTracking` = `issue.status === 'tracking'` → false (issue.json still says `pending`)
- `hasTrackMessages` = local store has messages → false (SSE missed)
- `isOptimisticTracking` = local flag → false (button wasn't clicked on this page load)

Result: `showProgress = false` → `TrackProgressPanel` never mounts → its internal recovery (`useTrackProgress`) never fires → circular dead state.

## Out of Scope

## Technical Notes

**Approach: Use unified sessions data to detect active track-creation sessions**

The most efficient fix: on Issues page load, fetch `/sessions/all?type=track-creation&status=active` once, then for each issue that has an active session, trigger the existing `useTrackProgress` recovery to hydrate the store.

This avoids N+1 API calls (one per issue) and reuses the same API that Agent Monitor already calls.
