# Specification: CaseAIPanel Timeline + Filter Tab Redesign

**Track ID:** ai-panel-timeline_20260322
**Type:** Refactor
**Created:** 2026-03-22
**Status:** Draft

## Summary

Replace CaseAIPanel's per-executionId tab isolation with a single session timeline using `groupByStep` + step filter tabs, aligning with the pattern already implemented in AgentMonitor's `CaseSessionDetail`.

## Context

The Dashboard's AI Assistant panel (CaseAIPanel) currently creates a separate tab for each step execution (keyed by `executionId`). However, all step executions within a case reuse the same backend Claude session. This creates a semantic mismatch: multiple tabs suggest independent sessions, but chat input from any tab routes to the same session.

AgentMonitor's `CaseSessionDetail` already implements the correct pattern — using `SessionMessageList` with `groupByStep=true` to render a unified timeline with collapsible step sections. CaseAIPanel should adopt this same approach, with the addition of filter tabs for quick step navigation.

## Motivation

1. **Tab semantics are misleading** — Multiple tabs imply independent sessions, but all share one backend Claude session. Chat routing is ambiguous.
2. **Dual-store complexity** — `caseAssistantStore` (executionId-driven) + `caseSessionStore` (sessionId-driven) coexist with adapter functions and fallback logic.
3. **Inconsistency with AgentMonitor** — AgentMonitor already uses `groupByStep` correctly; CaseAIPanel lags behind.

## Acceptance Criteria

- [ ] CaseAIPanel full mode message area uses `SessionMessageList` with `groupByStep` timeline display
- [ ] Tab bar changes from per-executionId isolation to step filter view: `[All]` default + per-step filters (e.g. `[data-refresh ✅] [troubleshoot 🔄]`)
- [ ] Clicking a filter tab shows only that step's messages; `[All]` shows the complete timeline
- [ ] Chat input always routes to the real backend session regardless of selected filter tab
- [ ] `StepQuestionForm` extracted as shared component, usable in both CaseAIPanel and AgentMonitor
- [ ] `caseAssistantStore` executionId message bucketing removed; messages flow through `caseSessionStore` per-session storage
- [ ] Compact mode remains functional: shows current active step + latest message preview
- [ ] Existing features preserved: action buttons, email type menu, message queue, step cancel, step progress recovery

## Dependencies

- `SessionMessageList` with `groupByStep` support (already implemented in `dashboard/web/src/components/session/SessionMessageList.tsx`)
- `caseSessionStore` per-session message storage (already implemented)
- SSE event infrastructure in `useSSE.ts` (will be modified)

## Out of Scope

- Changing the backend session model (per-case session reuse stays as-is)
- Modifying AgentMonitor's display for non-case session types (implement/verify/track-creation stay flat)
- Store unification for non-case session types (issueTrackStore stays separate)
- Adding new step types or changing step execution flow

## Technical Notes

- Phase 1 focuses on UI display layer (swap rendering, add filter tabs) — minimal store changes
- Phase 2 migrates data flow from `caseAssistantStore` to `caseSessionStore` — SSE handler + recovery changes
- `StepQuestionForm` extraction is independent and can be done in either phase
- The `groupByStep` in `SessionMessageList` already handles the case where messages have no `step` field (falls back to flat display), so non-step messages (chat, system) integrate naturally
