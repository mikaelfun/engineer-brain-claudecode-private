# Specification: test-loop 创建的 Issue 自动生成 Conductor Track

**Track ID:** auto-track-issues_20260330
**Type:** Enhancement
**Created:** 2026-03-30
**Status:** Draft

## Summary

当 test-loop 通过 `issue-creator.sh` 创建 Issue 后，自动生成 conductor track（含 spec.md），将 issue 直接从 `pending` 推进到 `tracked`，让下轮 SCAN 自动检测 gap 并生成测试。

## Context

当前 test-loop 有两条路径创建 Issue：
1. **FIX 阶段**：测试失败被分类为 product bug → `issue-creator.sh` 创建 Issue
2. **Supervisor 升级**：fixQueue 中反复修不好的测试 → `issue-creator.sh` 创建 Issue

创建后 Issue 状态为 `pending`，无人跟进。因为：
- TestLab 不展示这些 Issue（ISS-175 另行处理）
- 用户事后已失去修复上下文
- 手动 `/conductor:new-track` 需要用户知道 Issue 存在

而 `issue-creator.sh` 执行时已经有丰富的上下文信息：title、description、priority、fixRef（指向详细的根因分析和修复记录），完全可以自动生成 spec.md。

## Acceptance Criteria

- [ ] AC1: `issue-creator.sh` 创建 Issue 后自动生成 conductor track 目录（含 spec.md、metadata.json）
- [ ] AC2: spec.md 的 Acceptance Criteria 从 fix analysis 文件（fixRef 参数指向的文件）自动提取
- [ ] AC3: Issue JSON 自动回写 `trackId` 和 `status: "tracked"`
- [ ] AC4: `conductor/tracks.md` 自动追加新 track 行
- [ ] AC5: 如果 fixRef 为空或文件不存在，仍可创建 track（spec.md 使用 title+description 作为 AC）
- [ ] AC6: 去重逻辑不受影响 — `ISSUE_EXISTS` 时不创建新 track
- [ ] AC7: issue-scanner.sh 能在下轮 SCAN 中检测到新创建的 tracked issue 并提取 gap

## Dependencies

- `tests/executors/issue-creator.sh` — 主要修改文件
- `tests/executors/common.sh` — 可能需要新增工具函数
- `conductor/tracks.md` — 追加行
- `tests/executors/issue-scanner.sh` — 不需修改，仅验证兼容性

## Out of Scope

- TestLab UI 展示（由 ISS-175 处理）
- plan.md 自动生成（track 的目的是让 issue-scanner 能提取 AC，不需要完整 plan）
- 自动执行 `/conductor:implement`（仅到 tracked 状态，实现仍由人决定）

## Technical Notes

- Track ID 格式：`testloop-{issueId}_{YYYYMMDD}`（如 `testloop-ISS-176_20260330`）
- spec.md 的 AC 提取策略：
  1. 如果 fixRef 存在 → 读取 fix analysis 文件，提取「应该正确工作的行为」作为 AC
  2. 如果 fixRef 不存在 → 用 title + description 构造单条 AC
- 生成的 track 只包含 spec.md + metadata.json（最小可用），不生成 plan.md 和 index.md
- 所有文件操作使用 Node.js（与现有 issue-creator.sh 风格一致，避免 shell 转义问题）
