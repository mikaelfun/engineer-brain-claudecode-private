# Specification: Teams Watch 心跳机制

**Track ID:** watch-heartbeat_20260422
**Type:** Bug
**Created:** 2026-04-22
**Status:** Draft
**Issue:** ISS-237

## Summary

Teams Watch 在 WebUI 上 turn on 后，前端仅根据 PID 存活判断状态为 running/stopped，但 daemon 可能 PID 存活却不再有效轮询（poll 脚本失败、agency proxy 挂了等），导致用户看到 "running" 但实际没有新消息检测。

## Context

Dashboard Automations 页面的 Teams Watch tab 允许用户配置对 Teams 聊天的自动监听。后端 `teams-watch-reader.ts` 通过 `listWatches()` 读取 state 文件并做 PID 存活检查。前端 `useTeamsWatches()` 每 15 秒轮询一次后端 API。

**现有数据**：后端已返回 `lastPollAt`（最后一次成功 poll 的时间）、`interval`（轮询间隔秒数）、`consecutiveErrors`（连续错误次数），前端已接收但未用于 staleness 判断。

## Problem Description

1. PID 存活 ≠ 有效轮询。bash 进程可能挂在 sleep、poll 脚本内部报错、agency proxy 不可达等
2. 前端只显示 `running` / `stopped` 二元状态，无法区分 "正在轮询" 和 "PID 活着但长时间未更新 state"
3. `consecutiveErrors > 0` 时前端无任何告警
4. 用户无法感知 watch 的健康状态

## Acceptance Criteria

- [ ] AC1: 前端根据 `lastPollAt` + `interval` 计算 staleness：如果 `now - lastPollAt > 3 × interval`，标记为 stale
- [ ] AC2: Watch 卡片状态显示三态：🟢 running / 🟡 stale / 🔴 stopped，stale 态附带 "最后轮询 X 分钟前" 提示
- [ ] AC3: `consecutiveErrors > 0` 时在 Watch 卡片上显示错误计数徽章
- [ ] AC4: Watch 详情面板显示 lastPollAt 的相对时间（如 "2 分钟前"）和 pollCount
- [ ] AC5: Stale 状态的 Watch 卡片提供 "Restart" 快捷按钮（stop + start）

## Dependencies

- 后端 `listWatches()` 已返回所有必要字段（lastPollAt, interval, consecutiveErrors, pollCount）
- 前端 `useTeamsWatches()` 已有 15s 轮询

## Out of Scope

- 后端自动重启 stale watch（可作为 follow-up）
- SSE 推送 watch 状态变更（当前 polling 模式够用）
- Token Daemon 心跳机制（ISS-229 独立 track）

## Technical Notes

- Staleness 计算纯前端逻辑，无需后端改动
- 使用 `Date.now() - new Date(lastPollAt).getTime()` 对比 `3 * interval * 1000`
- 三倍 interval 作为阈值是因为：1× 是正常间隔，2× 允许网络延迟和 sleep 误差，3× 确认异常
