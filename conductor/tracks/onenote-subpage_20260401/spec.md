# Specification: onenote-export Sub-Page Recursive Export

**Track ID:** onenote-subpage_20260401
**Type:** Bug
**Created:** 2026-04-01
**Status:** Draft
**Issue:** ISS-194

## Summary

Fix `onenote-export.ps1` so that when a OneNote link targeting a specific page is passed, all sub-pages underneath it are also exported recursively.

## Problem Description

When `/onenote-export` receives a OneNote page link (e.g. `onenote:...#PageName&section-id=...`), it extracts the page name from the URL fragment and creates a strict regex filter `^PageName$`. This filter is applied to all pages in the section, which eliminates sub-pages because they have different names.

In OneNote's COM API data model, sub-pages are sibling `<one:Page>` elements that appear sequentially after their parent page, distinguished by having a higher `pageLevel` attribute value (1 = top-level, 2 = sub-page, 3 = sub-sub-page). The current filter at line ~612 discards all pages except the exact name match.

### Steps to Reproduce

1. Have a OneNote page "ADFS" with sub-pages "Management UI" and "RPT" underneath
2. Run `/onenote-export` with the OneNote link pointing to "ADFS"
3. Only "ADFS" is exported; "Management UI" and "RPT" are not

### Expected Behavior

All three pages exported: "ADFS" (level 1), "Management UI" (level 2), "RPT" (level 2)

### Root Cause

`onenote-export.ps1` line ~612:
```powershell
$allPages = @($allPages | Where-Object { $_.Name -match $PageName })
```

This filters by name only. Sub-pages have different names and higher `pageLevel` values. They are silently dropped. The existing sub-page directory nesting logic (lines ~655-679) works correctly but never fires because sub-pages are already eliminated by the filter.

## Acceptance Criteria

- [ ] When a link targets a page with sub-pages, all sub-pages are included in the export
- [ ] Sub-page nesting (level 2 under level 1, level 3 under level 2) is preserved in directory structure
- [ ] When a link targets a page with NO sub-pages, behavior is unchanged (only that page exported)
- [ ] When no link is provided (full notebook/section export), behavior is unchanged

## Dependencies

- `skills/onenote-export/onenote-export.ps1` — the PowerShell script with the bug
- OneNote COM API (`$OneNote.GetHierarchy`) — provides the page hierarchy XML

## Out of Scope

- Changing how full notebook/section exports work (already correct)
- Changing the link parsing logic (already correct)
- Adding new export formats or destinations

## Technical Notes

- Fix location: after the `$PageName` regex filter, scan forward in `$allPagesUnfiltered` from each matched page's position, including consecutive pages with `pageLevel > target.Level`, stopping at `pageLevel <= target.Level`
- The ancestor-lookup code at lines ~656-679 already does a similar backward scan — the fix is a symmetric forward scan at filter time
- Must operate on `$allPagesUnfiltered` which preserves original order
