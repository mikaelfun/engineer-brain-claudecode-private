# Specification: CaseDetail AI Panel — More Single-Step Operations

**Track ID:** ai-panel-ops_20260320
**Type:** Feature
**Created:** 2026-03-20
**Status:** Draft
**Source:** ISS-023

## Summary

Expose all 8 existing backend step routes as individual action buttons in CaseAIPanel, organized by category. Currently only 3 buttons exist (Full Process, Troubleshoot, Draft Email); the backend already supports 8 step routes that are fully functional but hidden from the UI.

## Context

The backend step routes (`POST /api/case/:id/step/:step`) already support: data-refresh, compliance-check, status-judge, teams-search, troubleshoot, draft-email, inspection, generate-kb. The CaseAIPanel currently only exposes 3 of these. Users need granular control to trigger individual operations without running the full casework pipeline.

## User Story

As an Azure Support Engineer, I want to trigger individual case operations (like refresh data, check compliance, search Teams) from the case detail page so that I can perform targeted actions without waiting for the full pipeline to complete.

## Acceptance Criteria

- [ ] All 8 existing step routes are accessible as individual buttons in CaseAIPanel
- [ ] Buttons are visually grouped by category: Data (refresh, teams-search), Analysis (compliance, status-judge, troubleshoot), Output (draft-email, inspection, generate-kb)
- [ ] Full Process button remains prominently placed as the primary action
- [ ] Each button shows its description on hover (tooltip)
- [ ] Existing operation lock mechanism works correctly (only one action at a time)
- [ ] Real-time SSE message display works for all step types (already supported)
- [ ] Draft Email retains its split-button with email type dropdown
- [ ] UI remains clean and not overwhelming despite more buttons

## Dependencies

- Existing backend step routes in `dashboard/src/routes/steps.ts` (already implemented)
- Existing SSE infrastructure for real-time updates (already implemented)
- Existing operation lock mechanism (already implemented)

## Out of Scope

- Creating new backend routes for sub-operations (refresh-emails-only, refresh-notes-only, etc.)
- Modifying backend logic — this is a pure frontend change
- Adding D365 write operations as buttons (those stay behind Todo confirmation)
