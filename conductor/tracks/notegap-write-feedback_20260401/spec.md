# Specification: NoteGap 写入 D365 后无成功反馈且内容丢失

**Track ID:** notegap-write-feedback_20260401
**Type:** Bug
**Created:** 2026-04-01
**Status:** Draft
**Issue:** ISS-203

## Summary

NoteGap 功能在点击"写入 D365"后存在两个 bug：(1) 写入完成后卡片立即消失，用户无法确认成功/失败；(2) 最终写入 D365 的 note 内容比 draft 丢失大量信息。

## Problem Description

### Bug #1: 写入完成后无状态反馈

**现象:** 点击"写入 D365"后，NoteGapCard 在 ~200ms 内消失，用户不知道是否成功。

**根因分析:**
1. `useSubmitNote` 的 `onSuccess` 立即调用 `qc.invalidateQueries({ queryKey: ['note-gap', caseId] })`
2. TanStack Query 自动 refetch GET `/case/{id}/note-gap`
3. 后端已删除 `note-draft.md`，返回 `{ hasGap: false, draft: null }`
4. `NoteGapCard` L36: `if (!data?.hasGap || !data.draft) return null` → 卡片 unmount
5. L123-126 的成功消息随卡片 unmount 一起消失

**影响文件:**
- `dashboard/web/src/components/NoteGapCard.tsx` (L36, L123-126)
- `dashboard/web/src/api/hooks.ts` (L1047-1055, `useSubmitNote`)

### Bug #2: 写入内容比 draft 丢失信息

**现象:** 最终写入 D365 的 note 内容不完整，比编辑器中的 draft 少了很多内容。

**根因分析:**
1. `note-gap-routes.ts` L106 将 body 嵌入命令行字符串:
   ```javascript
   const addCmd = `pwsh "add-note.ps1" -Body '${escapedBody}'`
   ```
2. 仅转义了 `'` → `''`，但多行内容、`$`、`` ` ``、`"`、`\n` 等字符在 PowerShell 单引号参数传递中可能被截断或解释
3. 命令行长度限制（Windows 约 8191 字符）可能截断长 body
4. `add-note.ps1` L70 中 `$Body -replace '\\n', "\`n"` 可能误处理已有的 `\n` 文本

**影响文件:**
- `dashboard/src/routes/note-gap-routes.ts` (L103-107)
- `skills/d365-case-ops/scripts/add-note.ps1` (L69-71)

## Acceptance Criteria

- [ ] 写入成功后显示明确的成功状态，持续至少 3 秒可见
- [ ] 写入失败后显示错误信息，不自动消失
- [ ] 多行内容、特殊字符（`$`、`` ` ``、`'`、`"`、中文）的 note 能完整写入 D365
- [ ] 长内容（>2000 字符）的 note 能完整写入不截断

## Dependencies

- D365 OData API (annotations endpoint)
- PowerShell `add-note.ps1` 脚本
- TanStack Query 缓存机制

## Out of Scope

- NoteGap 检测逻辑的修改
- NoteGap draft 生成逻辑的修改

## Technical Notes

**Bug #1 修复方向:** 提交成功后不立即 invalidate query，改为先展示成功状态（手动更新 cache 或延迟 invalidate），3 秒后再让卡片淡出。

**Bug #2 修复方向:** 不通过命令行参数传递 body，改为写临时文件让 PowerShell 脚本从文件读取（避免 shell 转义和长度限制问题）。或者直接在 Node.js 中调用 D365 OData API 而不经过 PowerShell。
