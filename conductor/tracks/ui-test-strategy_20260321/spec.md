# Specification: UI Test Strategy — Issue-Driven Functional Verification

**Track ID:** ui-test-strategy_20260321
**Type:** Feature
**Created:** 2026-03-21
**Status:** Draft
**Source:** ISS-056

## Summary

Enhance `conductor/workflow.md` Step 2 (UI Tests) to generate issue-driven functional verification
instead of generic screenshot-only tests. The test strategy should be derived from the Issue's
problem description, not just the spec's acceptance criteria.

## Context

The current Verify flow's UI Test (Step 2) reads spec.md acceptance criteria and generates
Playwright tests. In practice, these tests are overly conservative — mostly navigation +
screenshots + element visibility checks. They don't exercise the actual user interaction flow
that the Issue describes.

Example: ISS-050 (Cancel Track Creation Button) should test:
1. Navigate to Issues page
2. Create a new Issue via API
3. Click "New Track" button → verify progress panel appears
4. Click "Cancel" button → verify Issue status reverts to pending, panel disappears

Instead, the generated test just checks "Cancel button is visible on the page" — which proves
nothing about actual functionality.

## User Story

As a developer, I want the Verify UI tests to simulate the actual user workflow described in
the Issue so that I can be confident the fix/feature actually works end-to-end.

## Acceptance Criteria

- [ ] workflow.md Step 2 includes a new "Test Plan Generation" sub-step that reads the Issue description to design interaction-based tests
- [ ] A test classification system distinguishes between interaction-heavy Issues (buttons, dialogs, state transitions) and visual-only Issues (layout, styling)
- [ ] Interaction Issues get multi-step Playwright tests with clicks, waits, and state assertions
- [ ] Visual Issues get screenshot-based tests (current behavior preserved)
- [ ] Safety red lines are explicitly referenced in test plan generation — forbidden buttons/endpoints must never be tested via click
- [ ] Test plan is shown to user before execution for confirmation

## Dependencies

- Safety red lines in CLAUDE.md (must be respected)
- Existing Playwright patterns in workflow.md

## Out of Scope

- Changing the unit test (Step 1) strategy
- Creating a persistent Playwright test suite (tests are generated per-verify run)
- Modifying the verify route implementation in issues.ts

## Technical Notes

The change is purely to `conductor/workflow.md` — adding more detailed instructions for
how the Verify agent should plan UI tests. No code changes needed. The key insight is:
- Current: spec.md acceptance criteria → generic assertions
- New: Issue description + spec.md acceptance criteria → scenario-based interaction tests

The test plan should classify each acceptance criterion into one of:
1. **Interaction test** — requires clicks, form inputs, waits, state verification
2. **Visual test** — requires screenshot comparison
3. **API test** — requires curl/fetch verification
4. **Skip** — backend-only, not testable via UI
