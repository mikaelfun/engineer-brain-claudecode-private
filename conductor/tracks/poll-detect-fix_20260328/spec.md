# Specification: Fix poll_for_completion 完成检测逻辑

**Track ID:** poll-detect-fix_20260328
**Type:** Bug
**Created:** 2026-03-28
**Status:** Draft

## Summary

`poll_for_completion()` 通过 `GET /api/case/{id}/operation` 检测 casework 完成状态，但 API 响应格式与解析逻辑不一致，导致 casework 实际完成后 poll 永远无法检测到，所有 E2E 测试超时失败。

## Context

EngineerBrain 的自动化测试框架中，E2E 测试通过 `poll_for_completion()` 等待异步 casework 操作完成。该函数定义在 `tests/executors/common.sh`，是所有 workflow-e2e 类测试的核心基础设施。当前 Round 0 中，至少 5 个 E2E 测试因此 bug 超时失败（fast-path, state-routing, incremental, error-recovery, full-scenario）。

## Problem Description

**Steps to Reproduce:**
1. 运行任意 E2E 测试（如 `bash tests/executors/e2e-runner.sh fast-path 0`）
2. casework 在 ~30s 内完成（`timing.json` 已生成，`infer_casework_phase()` 返回 `timing`）
3. `poll_for_completion()` 持续轮询 `GET /api/case/{id}/operation`
4. API 响应格式与 poll 解析逻辑不匹配
5. poll 一直判定操作仍在运行，直到 timeout（300-1200s）

**Expected:** casework 完成后，poll 在下一次轮询时检测到完成，返回 0
**Actual:** casework 30s 完成，poll 持续轮询直到 timeout，从未检测到完成

**API Response Format (discovered):**
- Active: `{ caseNumber, operation: { operationType, startedAt } }` — no `status` field
- Complete: `{ caseNumber, operation: null }`

**Poll Logic Mismatch:** checks `op.status === 'completed'|'done'|'failed'` but API never returns `status` field.

## Acceptance Criteria

- [ ] `poll_for_completion()` 能正确检测 API 返回的操作完成状态
- [ ] 增加文件系统级 fallback 检测（timing.json 存在即视为完成）
- [ ] 所有依赖 poll 的 E2E 测试不再因完成检测失败而超时
- [ ] 现有 `infer_casework_phase()` 逻辑不受影响

## Technical Notes

Root cause: `poll_for_completion()` 第 381-391 行的 node 解析逻辑 `d.operation || d` 在 `operation: null` 时回退到整个 response 对象 `{ caseNumber, operation: null }`，该对象没有 `status`/`active` 字段，最终落入 `else` 分支输出 `'false'`（理论上应该 work）。需要进一步诊断是 lock 未释放（API 始终返回非 null operation）还是 poll 解析的 edge case。修复方向：(1) 对齐解析逻辑；(2) 增加 timing.json fallback。
