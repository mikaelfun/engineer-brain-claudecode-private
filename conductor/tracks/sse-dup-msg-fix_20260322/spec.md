# Specification: Fix SSE Duplicate Messages in AI Assistant Panel

**Track ID:** sse-dup-msg-fix_20260322
**Type:** Bug
**Created:** 2026-03-22
**Status:** Draft
**Source:** ISS-091

## Summary

AI Assistant Panel on Case Detail page displays SSE message streams twice, while Agent Monitor shows the correct single stream. Root cause is a combination of backend emitting duplicate terminal events and frontend allowing multiple panel instances to independently hydrate the same shared store.

## Context

The Dashboard uses a single global SSE endpoint (`/api/events`) to stream all case/session events. Frontend subscribes once via `useSSE()` in App, then populates a shared Zustand store (`caseSessionStore`). CaseAIPanel reads from case-level aggregate (`messages[caseNumber]`), while Agent Monitor reads from per-session bucket (`sessionMessages[case:sessionId]`). This architectural difference, combined with backend dual-event emission and multi-instance recovery hydration, causes the AI panel to display duplicated content.

## Problem Description

### Symptoms
- SSE messages appear twice in CaseAIPanel (AI Assistant tab on Case Detail page)
- Agent Monitor shows correct single stream for the same session
- Observed during step execution (exact trigger uncertain)

### Root Causes Identified

1. **Backend dual terminal events**: Step routes in `steps.ts` emit both `case-step-completed` AND raw `case-session-completed` for the same step completion. Frontend `useSSE` handles both, appending two completion messages.

2. **Multiple CaseAIPanel instances**: CaseDetail page can mount both full and compact CaseAIPanel for the same case simultaneously. Each instance has independent `hasRecoveredRef` / `hasStepRecoveredRef`, so both can hydrate persisted messages into the same shared store — doubling recovered messages.

3. **No store-level de-duplication**: `caseSessionStore.addMessage()` appends blindly without checking for duplicate content/timestamp/id.

4. **Step cancel double events**: Cancel broadcasts both `case-step-failed` and `case-session-completed`.

### Why Agent Monitor is unaffected
Agent Monitor prefers per-session scoped messages (`sessionMessages[case:sessionId]`), which is narrower and less susceptible to aggregate contamination. CaseAIPanel uses case-level aggregate (`messages[caseNumber]`), which accumulates all events including duplicates.

## Acceptance Criteria

- [ ] Step completion no longer produces two terminal messages in the AI panel
- [ ] Only one CaseAIPanel instance performs recovery hydration per case
- [ ] Store-level de-duplication prevents identical messages from appearing twice
- [ ] Step cancel produces a single terminal message (not both failed + completed)
- [ ] Agent Monitor continues to work correctly (no regression)

## Dependencies

- Existing SSE infrastructure (`useSSE`, `caseSessionStore`, `sseManager`)
- CaseAIPanel recovery logic
- Backend step routes

## Out of Scope

- Changing the single global SSE channel architecture
- Refactoring Agent Monitor's message display
- Adding per-message unique IDs from backend (nice to have but not required for this fix)
