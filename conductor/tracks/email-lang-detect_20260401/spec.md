# Specification: Email Drafter Auto Language Detection

**Track ID:** email-lang-detect_20260401
**Type:** Feature
**Created:** 2026-04-01
**Status:** Draft

## Summary

Email drafter should automatically detect the customer's language from email history and draft in the same language, instead of defaulting to English.

## Context

Currently `email-drafter.md` has `language` param defaulting to `en`. The `draft-email/SKILL.md` supports `en`/`zh` as optional suffix. But when called without explicit language (most common case — from casework routing or WebUI), it always drafts in English, even for Chinese-speaking customers.

## User Story

As an Azure Support Engineer, I want the email drafter to automatically match the customer's language so that I don't have to manually specify `zh` for Chinese customers or get English drafts sent to Chinese-speaking customers.

## Acceptance Criteria

- [ ] When no language is explicitly specified, email-drafter reads `{caseDir}/emails.md` and detects the dominant language of customer emails
- [ ] Detected language is logged in `email-drafter.log` (e.g., `STEP 0 OK | lang=zh (auto-detected from emails)`)
- [ ] Correct humanizer is selected based on detected language (humanizer for en, humanizer-zh for zh)
- [ ] Explicit language parameter still overrides auto-detection

## Dependencies

- Existing `emails.md` file in case directory (populated by data-refresh)
- Existing humanizer / humanizer-zh skills

## Out of Scope

- Supporting languages beyond en/zh (can extend later)
- Changing WebUI draft-email step to show detected language

## Technical Notes

- Detection logic lives in `email-drafter.md` Step 1 (read context) — add a sub-step to scan `emails.md` for CJK characters
- Simple heuristic: if customer emails contain >30% CJK characters → `zh`, else `en`
- The `draft-email/SKILL.md` passes `language` to the agent; if not provided, agent auto-detects
