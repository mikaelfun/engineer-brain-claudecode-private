# Product-Learn Auto-Enrich v2 — Brainstorming 结论

## 日期: 2026-04-04

## 核心架构变化
- v1: 4 phase 循环 × 12 产品 × maxCycles → 散装 JSONL
- **v2: 3 阶段 Pipeline (EXTRACT → SYNTHESIZE → REFRESH) × 穷尽扫描 → JSONL 原始层 + Markdown 综合指南层**

## 关键决策

### 1. 目标定位
- **主动扫描为主**，troubleshooter/case-review 按需写入为辅

### 2. 完整性要求
- **完整扫描所有页面**，不限 maxCycles
- 用 `scanned-sources.json` 记录已扫描页面路径/URL，防重复+保证覆盖
- 穷尽后标记 `exhausted`

### 3. 输出格式：双层
- **JSONL 原始层**: 每条标 sourceRef + quality + date（审计追踪）
- **Markdown 综合指南层**: 按主题聚类合并，内联来源标注（可点击链接）
  - OneNote: 相对路径
  - ADO Wiki: 完整 URL
  - MS Learn: 完整 URL
  - Case: case 编号
- Troubleshooter 只读指南，不读 JSONL

### 4. 质量控制
- EXTRACT: **宽松写入**，有 symptom 就记录
- SYNTHESIZE: **严格过滤**，去掉不完整/过时/无关的条目
- 指南生成摘要需记录：丢弃的条目路径+原因，保留的条目路径+原因

### 5. 增量更新
- **钩子模式**: onenote-export sync 团队 notebook 后自动触发
- 只重扫变更的页面 → 更新 JSONL → 重生成受影响的指南
- 个人笔记不作为 auto-enrich 来源

### 6. Troubleshooter 消费方式
- 新增 Step 1.5: 查 `guides/_index.md` → 匹配指南 → 获得预构建排查路径
- 未命中 → **优先搜索流程**（onenote → ado wiki → mslearn → kusto queries）→ 汇总后才决定排查方法

### 7. 产品优先级
- intune → vm → aks → monitor → entra-id → networking → disk → acr → arm → avd → purview → eop

### 8. 产品范围
- **不局限于现有 12 个目录**，按 Support Topic List 的 VM+SCIM POD 服务列表动态生成
- POD 配置写入 `config.json`

### 9. Case 复盘
- 保持独立入口 `/product-learn case-review`，现阶段不纳入 auto-enrich pipeline
- 后续可通过 owa-email-search 周期性拉取邮件，按增量判断

### 10. Content Idea KB（v2.1 待探索）
- ADO Content Idea 包含大量内部 KB，质量参差不齐
- 需探索：能否通过 ADO MCP 穷举 KB work items → 获取 URL → 读取内容
- 每个 KB 产品不确定，需做筛查
- 标记为 v2.1 独立数据源，不阻塞 v2 主体
