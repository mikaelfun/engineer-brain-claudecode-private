
## Teams Queue 架构重构 (2026-04-12 深夜)

### 问题诊断
- 旧架构 Queue 处理 3 case 耗时 23 min → 90% 时间在 LLM 思考/JSON 构建（非 MCP 搜索）
- Queue 不等 casework 写 request.json 就退出
- casework QUEUE_MODE 不等 Teams 数据就跑 status-judge
- casework-timing.sh 删除 timing markers 导致 patrol-monitor 失效

### 重构内容
1. **Queue MCP-Only**：Queue 只做 MCP 搜索 + dump `_mcp-raw.json`，不做 write-teams/relevance/digest
2. **Queue Haiku**：model 从 Sonnet 换 Haiku（零 LLM 推理，只需执行工具调用）
3. **Casework spawn teams-search agent**：QUEUE_MODE 下写 request.json + spawn teams-search(QUEUE_MODE)，subagent 等 `_mcp-raw.json` → 用 `build-input-from-raw.py` 转格式 → write-teams.ps1 → relevance+digest
4. **`build-input-from-raw.py` 新脚本**：纯 Python 转换 MCP 原始格式 → write-teams.ps1 格式
5. **Queue 无 idle timeout**：只听 `.teams-queue-stop` 退出，patrol 控制生命周期
6. **Patrol 等 Queue 就绪**：启动 casework 前先等 `.teams-queue-active` 出现
7. **Patrol 排空检查**：发 stop 前轮询 request.json 归零
8. **Timing markers 生命周期修复**：casework 开始时 `rm .t_*`（而非结束时），完成时写 `.t_done`
9. **dump + next-case 合并**：Queue 一个 Bash 完成当前 case dump + 下一个 case 预检

### 预期效果
- Queue 3 cases: 23min → ~90s（Haiku + MCP-only + 流水线）
- Casework teams 等待: 180s timeout → subagent 冷启动与 queue 搜索重叠，无额外等待
- patrol-monitor: 能持续看到 timing markers 直到下次 casework

### 待验证
