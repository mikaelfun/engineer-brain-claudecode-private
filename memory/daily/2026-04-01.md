# 2026-04-01 Daily Memory

## 主题：Mail MCP vs Outlook COM vs OWA 邮件获取对比调研

### 核心结论
1. **COM 方案放弃** — DASL Restrict 在无 Instant Search 时静默失败；手动遍历 24k 项耗时 63s；MCP 覆盖更全（含 sent+received）
2. **MCP (Graph API) 可用但有限流** — 3 并发触发限流，单 Case 最长 49 分钟；批量 30 Case 用了 7 波 worker 共 ~60 分钟
3. **OWA (Playwright) 最优** — 无限流，~45s/case 稳定，10/10 测试通过，纯文本 innerText 不需要 HTML 清洗

### MCP 批量测试
- 创建了 `email-batch-worker` agent（mcpServers: mail, model: opus）
- 任务池模式（JSONL + worker 并行拉取）
- 30/30 完成，0 失败，29 个 md 文件 156KB
- **发现**: maxTurns=50 导致 worker 退出遗留 stuck running → 已去掉 maxTurns
- **发现**: worker 花大量 turns 处理 JSON 转义 → 创建了 mcp-response-handler.ps1

### OWA Playwright 探索
- persistent profile + SSO（`$TEMP/playwright-owa-profile`）
- session 隔离：`-s=owa` vs D365 default session
- 搜索：`[role=search] [role=combobox]` → type → Enter
- 展开：click "Expand conversation" 按钮
- body 懒加载：`scrollIntoView + click` 每个 `[role=listitem]`
- 提取：`[aria-label="Message body"]` 的 `innerText`
- 过滤：跳过 "发件人:" / "From:" 开头的嵌套引用

### OWA 待解决 P0
**多 Conversation 合并**：一个 Case 的邮件可能分散在多个 conversation 中（subject 不同）。搜索返回多个 option，需要逐个 click + 提取 + 存入数组。当前脚本只取第一个 option。
