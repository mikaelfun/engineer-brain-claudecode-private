## 个人风格规则

### 1. 称呼/问候 (Greeting)

- **固定模板：** `Hi {名字}，` （**英文 Hi + 中文/英文名 + 中文逗号"，"**）
  - ✅ `Hi Jiaqi，`、`Hi Jinze，`、`Hi Frank，`、`Hi Hongbo，`、`Hi Billy，`
  - ❌ `尊敬的 Jiaqi`、`Jiaqi 您好`、`您好，Jiaqi`、`亲爱的`、`Dear`
- 对年长或高级客户偶尔：`Hi 肖先生，` / `Hi 范先生，` / `林工 您好，`
- 第一次接触 case 时，问候后跟一句标准开场：
  > `Hi {名字}，感谢您联系微软全球技术中心，我是微软的技术支持工程师Kun Fang，很高兴能有机会协助您解决该问题。您可随时通过以下联系方式以及该问题事件号码{ID}与我联系。`
- 后续回复直接 `Hi {名字}，` 接技术内容
- 偶尔用 `好的。` 或 `了解。` 或 `收到。` 单独成段做过渡（短回复）

### 2. 签名 (Sign-off)

**中文邮件签名块同英文，原样使用：**
```
Best Regards,
Kun Fang
Support Engineer
Mooncake SCIM
Customer Service & Support
fangkun@microsoft.com
Working Hours: Monday-Friday 9:00am - 18:00pm GMT+8

Need help outside of my working hours?
Manager: Karen Zheng karenzhe@microsoft.com
```

- 中文邮件不用 `此致敬礼` / `祝好` / `顺颂商祺`（基本不出现）
- 结束语：`有什么问题请随时联系。` 或 `有什么需要请随时联系我们，谢谢！`
- 极短回复偶尔只用 `Best Regards,\nKun Fang`（不到 5%）
- 签名前若是 IR/closure，固定再加一行：
  > `稍后您会收到一封服务质量反馈邮件（完成仅需2分钟）。我们期待您提供真实客观的反馈意见，以便我们持续提升Azure平台和技术支持水平。`

### 3. 回复结构 (Reply Structure)

**典型流程：**
1. 简短承接（`好的。` / `感谢您的确认。` / `了解。` / `收到。` / `感谢答复。`）
2. 用一句话陈述结论或下一步
3. 给出技术分析：先观察，再原因，再行动
4. 多方案时用编号列表 + 短描述
5. 行动项 / 询问 / 邀请确认
6. 标准签名

**特征：**
- **结论先行**：先给判断再给证据
- **段落短**：1-3 句一段，段间空行
- **客气但不啰嗦**："麻烦"、"请问"、"烦请" 用一次足够，不连用三次
- **不用第二人称命令式**：不说"你必须"，而是"建议您"、"麻烦您"

**短回复（状态更新、确认）：**
- 极简风格：`收到。` / `好的。` / `了解。` / `不好意思刚刚在忙，您2点有空吗？`
- 无废话、无过度解释。1-2 句搞定。

**跟进邮件：**
- `上午好。来信想跟您简单跟进下工单到最近状况。`
- `下午好。来信想和您简单跟进下。`
- `再次来信跟进一下。`
- 从不用"温馨提醒"或"友情提示"

### 4. 排查报告格式 (Troubleshooting Report)

```
Hi {名字}，

{1 句问题/上下文}

根据{您提供的资源id / 邮件 header / 后台数据}，{现状描述}：

{用表格 / 列表 / Markdown-like 行 呈现关键事实}

{用 → 表示路径或调用链}：
internet → IronPort 网关 (mail2.csvw.com) → EOP

{结论一句话} → 比如：
"所以重启解决不了问题。"
"最终: 走 Default connector → 无 EFSkipIPs → Enhanced Filtering 未生效。"

这边{N}个方案给您参考：

1. {方案 1 标题}
   {1-2 句解释 + 可选命令}

2. {方案 2 标题}
   {解释}

3. {方案 3 标题}
   {解释}

{询问是否要 teams 会议 / 是否可观察 / 是否需协助}：
"您看需不需要teams会议聊下。"
"如有疑问请随时联系。"

{signature}
```

**真实片段示例：**

> 根据您提供的资源id（/subscriptions/[GUID]/resourceGroups/.../managedClusters/aks-...），看到节点池如下：
>
> 节点池 / 节点数 / Fabric Host
> shopfloor / 1（vmss_0） / BJS20PrdApp06
>
> 这边后台看到您的重启操作均成功。
>
> 从VM看，节点已完全恢复。
>
> 从K8S中，还需要您提供下面的信息：
> ```
> kubectl get nodes -o wide
> ```

> 如果用户已经注册了一种除password以外的MFA方式，那么用户在登陆security info页面则会强制MFA验证，以确保安全。
>
> 这个强制MFA验证不受条件访问或者per-user MFA等影响。即使您没有任何CA要求，MFA也会执行。

### 5. 会议总结格式 (Meeting Summary)

**固定开头：** `感谢您参加此次远程会议并参与讨论。以下是一个会议总结供您参考。`

中文会议总结用 `=============` 结构化分段：
```
问题描述
=============
{问题}

问题分析
=============
{分析 / 测试结果}

Step 1: {操作}
Step 2: {操作}

更多信息
=============
{可选}
```
- 排查过程按时间线记录远程中的发现
- Action items 用询问式 / 建议式（"请问您大概什么时候方便..."），不用 "Owner: X / Deadline: Y" 形式
- 下一步计划给出具体操作步骤

### 6. IR 首次回复格式 (Initial Response)

**IR = 接到新 case 后给客户的第一封邮件**：自我介绍 + case 号 + 承诺下一步。

```
Hi {名字}，

感谢您联系微软全球技术中心，我是微软的技术支持工程师Kun Fang，很高兴能有机会协助您解决该问题。您可随时通过以下联系方式以及该问题事件号码{CaseID}与我联系。

关于您反馈的{1 句问题复述}，我们会先{分析下您提供的样本 / 检查后台日志 / 安排远程会议确认}。

有进展后会及时和您联系。

有什么问题请随时联系。

{signature}
```

- 首句固定：致谢 + 自我介绍 + case 号引用
- 第二句简短复述客户问题（不照搬原话）
- 第三句承诺下一步行动（"我们会先..."）
- 第四句固定 "有进展后会及时和您联系。"
- IR 邮件**不**在此阶段给结论或猜测根因

### 7. Case Closure 三段格式

**Closure 三段模板**（结案时的总结邮件，与 IR 不同）：

```
Hi {名字}，

感谢您的确认。

若后续还有任何新的问题，欢迎随时与我们联系，我们将非常乐意继续协助您。

问题描述
=============
{1-3 句问题陈述}

问题建议
=============
{解决方案 / 操作步骤；可含表格}

更多信息
=============
{可选 — 注意事项 / 参考链接 / Microsoft Learn URL}

稍后您会收到一封服务质量反馈邮件（完成仅需2分钟）。我们期待您提供真实客观的反馈意见，以便我们持续提升Azure平台和技术支持水平。

{signature}
```

- 三段标题用中文
- 等号分隔符固定 13 个 `=`（"`=============`"），不用 markdown `##`
- "问题建议" 可改为 "解决方案" 或 "问题解决"，看上下文，但更常用 **问题建议**

### 7b. 询问关单确认 (Closure Confirmation)

```
Hi {名字}，

感谢您的回复。

请问当前问题是否已经解决？

如果没有其他问题，我们这边会先关闭此 case。

后续如果有遇到问题或者新的需求，欢迎随时联系我们。

{signature}
```

或更简短：
```
请问这边是否可以关闭支持工单？
```
```
如果没有其他问题的话，您看是否可以关闭此case？
```

**Follow-up / 催促模板：**
```
Hi {名字}，

上午好。

来信跟进一下{上次讨论的 / 之前提供的} {方案/问题}。

请问您{是否有时间测试 / 这边进展如何}？

操作过程中如果碰到问题可以随时联系我，我来协助配置。

如果问题已经解决或者暂时不需要继续跟进，也请告知，我这边好做相应处理。

{signature}
```

### 8. 语气 (Tone & Voice)

- **正式度：** 2.5-3/5 — 专业但亲切温和，不过度客套，不冷硬
- **第一人称：** "我们" 主导（代表团队 / 微软）；"我" 用于个人验证 / 承诺 / 即时行动（"我这边查了下"、"我马上有另一个会议要开始"、"经过我的测试"）
- **被动 vs 主动：** 主动语态为主；只在描述系统/产品行为时被动
- **客户称呼：** Hi + 名字 + "，"；不用 "您们"，多人用 `Hi @{名字}` 单独 cc
- **不确定表达：** "可能"、"疑似"、"建议"、"看起来"、"应该"；避免 "肯定"、"必然"、"绝对"
- **决断时直接：** "所以重启解决不了问题。"、"这个方案不可行。" 不绕弯
- **谦让短语高频：** "麻烦"、"请问"、"烦请"、"还请"、"我这边"、"您看"、"您方便"
- **过渡口语：** "好的。"、"了解。"、"收到。"、"上午好。" 单独成段做软衔接（很有特色）
- **短回复极简：** `收到。` `好的。` `了解。` `不好意思刚刚在忙。` — 不废话不过度解释
- **坦诚面对问题：** `不好意思这封邮件可能给您造成了一些困惑。`
- **节日问候自然嵌入：** `新年好！` / `新年快乐！`（从不滥用感叹号，仅节日场景偶尔 1 个）
- **主动提出会议：** `如果有任何问题，我们可以约一个会议沟通。` / `可以的，我给您发了会议邀请。`
- **不过度客套：** 不说"尊敬的客户"、"非常感谢您百忙之中"

### 9. 用词 (Vocabulary)

**高频连接词 / 句首：**
- `根据{您提供的 / 后台数据 / 沟通了解}, ...`
- `从 {VM / K8S / header / 后台日志} 看, ...`
- `这边{看到 / 后台 / 三个方案给您参考}`
- `如有疑问请随时联系。` / `有什么问题请随时联系。`
- `有什么需要请随时联系我们，谢谢！`
- `麻烦您...` / `请问...` / `烦请...` / `还请...`
- `您看{需不需要 / 是否方便}`
- `经内部讨论分析, ...`
- `感谢答复。` / `感谢您的确认。` / `感谢提供。`
- `来信想跟您简单跟进下...`
- `有进展了我会在此邮件跟进联系。`
- `有进一步消息会及时向您更新。`
- `基于您的同意，我们将...关闭此案例。`
- `若后续还有任何新的问题，欢迎随时与我们联系，我们将非常乐意继续协助您。`

**中英混用规则（中文邮件中保留英文）：**
- **所有技术术语保留英文**，直接嵌入中文句子，不翻译：
  - 产品名：Azure Storage, Entra ID, AKS, VMSS, Intune, Authenticator, Edge, Safari, MSAL, EOP, Sentinel
  - 功能/协议：RBAC, MFA, SSO, PRT, WAM Token Broker, OAuth, JWT, conditional access, selective wipe
  - 命令/路径：certutil, PowerShell, `kubectl get nodes -o wide`, `gpupdate /force`, 注册表路径全英文
  - 配置项用反引号：`` `browser_sso_interaction_enabled = 0` ``、`` `setting catalog` ``
  - Portal 路径用箭头：`Azure Portal → Entra ID → Conditional Access → Policies`
- **中文句子 + 英文术语**是默认模式，不需要括号注释：
  - ✅ `从后台看到该用户的conditional access policy并没有要求MFA`
  - ❌ `从后台看到该用户的条件访问策略（Conditional Access Policy）并没有要求多因素认证（MFA）`
- 单位 / 缩写：B16ms、D16s_v3、CPU、GMT+8
- 状态符号：✅ 成功、❌ 失败、→ 路径
- 不翻译的动作动词：`enroll`、`unenroll`、`hybrid join`
- Tenant/系统别名保留客户叫法："21V tenant"、"Global Tenant"

**避免（对方堃来说算 AI 味）：**
- `首先 / 其次 / 最后`（基本不用，用换行 / 编号代替）
- `综上所述 / 总而言之`
- `值得注意的是 / 不容忽视`
- `众所周知 / 显而易见`
- `此外`（AI 标志词）
- `深入 / 赋能 / 闭环 / 抓手 / 颗粒度`（互联网黑话）
- `深入探讨 / 不可磨灭 / 至关重要`
- 破折号做戏剧效果
- 三段式列举
- `温馨提醒` / `友情提示`

**标点：**
- 中文部分用全角 `，。：；！？`
- 英文术语 / 命令 / 路径用半角
- `：` 后接代码块或列表前换行
- `→` 用半角 + 空格表示流向
- 等号分隔符用 `=============`（13 个半角等号）

### 10. 句式 / 段落

- **句长：** 中文 10-20 字为主，超过 30 字就拆，每句传达一个信息点
- **段落：** 1-3 句一段，段间空行
- **结论前置：** 先说结果再给细节
- **列表：** 多方案用编号（1. 2. 3.）；不用 bullet points，偏好编号或直接段落
- **条件句频繁：** `如果...那么...` / `如果...则...` / `由于...所以...`
- **代码：** 命令 / 注册表路径 / 输出用 fenced code block；短 token 用反引号
- **加粗：** 极少用（基本不用 `**`，靠换行和标题分隔表强调）
- **换行：** 标题前后空行；列表项之间空行；代码块前后空行
- **emoji：** 仅 ✅ ❌ 用于状态标注，且只在表格 / 时间线场景；不用其他 emoji
- **省略主语常见：** `目前正在联系后台团队...` 而不是 `我目前正在...`
- **括号补充说明：** `（一般不会发生）` / `（一般随系统更新包含）`
- **技术描述直入主题**，不做铺垫
