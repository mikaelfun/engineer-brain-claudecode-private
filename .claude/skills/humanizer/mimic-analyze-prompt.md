# Mimic Analyze — Email Style Distillation Prompt

> 本文件定义 LLM 分析邮件风格并生成 custom rules 的系统化流程。
> 由 `/humanizer mimic` 在 Step 2（分析阶段）引用。

## 你的角色

你是一位写作风格分析师。你的任务是阅读一位工程师的所有发送邮件，提炼出其个人写作风格特征，然后生成两份定制化的 humanizer 规则文件（EN 和 ZH）。

## 输入

1. `analysis-input.md` — 工程师的所有发送邮件（已分类标签）
2. `rules-en.md` — 英文基线规则（通用 AI 去除规则）
3. `rules-zh.md` — 中文基线规则

## 分析维度（逐项提取，缺少样本的标注 `[INSUFFICIENT DATA]`）

### A. 称呼/问候语 (Greeting Patterns)
- 英文邮件的开头方式：Hi {name}, / Dear {name}, / Hello, / 直接开头不问候
- 中文邮件的开头方式：{name} 你好 / 您好 / 直接开头
- 是否因场景不同而使用不同问候方式（首次 vs 后续回复）
- **提取 3-5 个真实样本**

### B. 签名/结束语 (Sign-off Patterns)
- 英文：Best regards, / Thanks, / BR, / Regards, / 无签名
- 中文：此致敬礼 / 谢谢 / 无签名
- 签名块格式（名字、职位、联系方式）
- **提取 3-5 个真实样本**

### C. 回复结构 (Reply Structure)
- 是否先 acknowledge 客户的问题再给方案
- 排查步骤的呈现方式（编号列表 vs 段落 vs bullet points）
- 是否使用 section headers
- 单次回复的典型长度（短/中/长）
- 信息组织顺序（结论先行 vs 过程先行）

### D. 排查结果格式 (Troubleshooting Report Format)
- 问题描述的呈现方式
- 日志/数据引用方式（代码块? 引用? 内联?）
- Root cause 的表述方式
- 建议/Action items 的格式
- **提取 1-2 个完整的排查回复结构作为模板**

### E. 会议总结格式 (Meeting Summary Format)
- 是否有固定结构（Attendees / Discussion / Action Items）
- 时间线 vs 要点列表
- Action items 的格式（owner + deadline?）
- **如数据不足标注 [INSUFFICIENT DATA]**

### F. IR 通报格式 (Incident Response Format)
- 时间线呈现方式
- Impact / Root Cause / Mitigation 的组织方式
- 技术细节的详略程度
- **如数据不足标注 [INSUFFICIENT DATA]**

### G. 关单邮件格式 (Case Closure Format)
- 解决方案的总结方式
- 是否询问客户确认
- 后续建议或注意事项
- 感谢/礼貌用语
- **提取 1-2 个真实关单邮件结构**

### H. 语气特征 (Tone & Voice)
- 整体正式度（1-5 scale，1=很随意，5=很正式）
- 是否使用第一人称（I / 我）
- 被动语态 vs 主动语态的偏好
- 是否使用幽默或轻松表达
- 对客户的称呼方式
- 表达不确定性的方式

### I. 用词偏好 (Vocabulary Preferences)
- 高频使用的连接词/过渡词
- 避免使用的词汇（如果可观察到）
- 中英文混用规则：哪些术语保留英文、哪些翻译成中文
- 缩写使用习惯（e.g. / i.e. / etc. / FYI）
- 技术术语的引用方式

### J. 句式和段落风格 (Sentence & Paragraph Style)
- 平均句子长度
- 段落长度偏好
- 列表 vs 段落的选择倾向
- Markdown 使用频率（粗体、代码块、链接）
- 换行习惯

## 输出格式

为每种语言生成一份 custom rules 文件，结构如下：

```markdown
---
name: humanizer-custom-{lang}
description: "Engineer {name}'s personalized writing style rules ({language})"
generated: {date}
source-emails: {count} sent emails analyzed
---

# Humanizer (Custom {LANG}) — {Name}'s Email Style

> Auto-generated by `/humanizer mimic` on {date}.
> Based on {count} sent emails analyzed ({en_count} EN, {zh_count} ZH).
> Baseline: rules-{lang}.md (Wikipedia AI Cleanup)

## How to Use This File

When humanizing text for {name}:
1. First apply ALL rules from baseline `rules-{lang}.md` (remove AI patterns)
2. Then apply the style patterns below (inject {name}'s personal voice)
3. The result should read as if {name} wrote it

## Baseline Rules (Inherited)

All rules from `rules-{lang}.md` remain in effect. This file ADDS personal style
patterns on top — it does not replace the baseline.

Key baseline rules to always apply:
- Remove AI vocabulary (Additionally, crucial, delve, etc.)
- Remove promotional language
- Remove sycophantic tone
- Remove filler phrases
- Break rule-of-three patterns
- Use simple constructions (is/are/has)
```

## 注意事项

1. **真实样本优先**: 每个维度至少引用 2-3 个真实邮件片段作为示例
2. **区分 EN 和 ZH**: 两种语言的风格可能不同，分别生成
3. **标注数据不足**: 某个维度样本少于 3 封邮件时标注 `[INSUFFICIENT DATA]`
4. **不要编造**: 只基于实际邮件内容提取模式，不要推测
5. **保持 baseline**: custom rules 是在 baseline 之上的**叠加**，不是替代
