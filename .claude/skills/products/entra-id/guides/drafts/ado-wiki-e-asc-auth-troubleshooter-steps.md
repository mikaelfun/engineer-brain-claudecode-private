# TSG: General ASC Auth Troubleshooter Steps

General guidance on how to use Azure Support Center (ASC)'s Auth Troubleshooter to debug a customer's Entra sign-in failure.

## Prerequisites

To see "Troubleshoot this sign-in" link in ASC for Auth Troubleshooter, ensure you are using your TSE/MID account and have joined the required Access Package described in [Onboarding (TSE)](https://dev.azure.com/IdentityCommunities/Community%20-%20Identity%20Diagnostics%20and%20Tooling/_wiki/wikis/Community---Identity-Diagnostics-and-Tooling.wiki/1168/Onboarding-(TSE)).

> **Note**: ASC Auth Troubleshooter is not available in ASC US Gov environment. For US Gov customers, add scenarios/ICM/cases to the [CSS AAD Log Access Tracking Spreadsheet](https://aka.ms/cssaadlogaccess).

## Step 1: Request or obtain sign-in failure details

To run the ASC authentication troubleshooter, you need at minimum a **Timestamp in UTC** and a **Correlation ID** or **Request ID** generated by login.microsoftonline.com during the sign-in failure.

### Option 1: Use ASC -> Tenant Explorer -> Sign In Logs

1. Determine the principal and timestamp of the sign-in from customer
2. Visit ASC -> Tenant Explorer -> Sign In Logs
3. Filter by user and timeframe
4. Locate the failed sign-in and choose "Troubleshoot Signin" option

### Option 2: Request Troubleshooting Details Output

1. Ask the customer to reproduce the sign-in failure
2. On the `login.microsoftonline.com` page, click the three dots `***` in the lower right corner to open the **Troubleshooting details** blade
3. Click "Copy info to clipboard" and send the details

### Option 3: Request HAR or Fiddler trace

1. Request the customer reproduce the sign-in failure while capturing a [browser trace](https://learn.microsoft.com/azure/azure-portal/capture-browser-trace) or a [FiddlerCap](https://www.telerik.com/fiddler/fiddlercap) (with HTTPS decryption enabled)
2. Open the trace in Fiddler on your workstation
   - To open HAR in Fiddler: Import Sessions -> HTTP Archive -> choose .HAR file
3. Locate frames to login.microsoftonline.com containing **client-request-id** or **x-ms-request-id**
   - Use Edit -> Find Sessions to search for these IDs
   - Check Headers tab in both Request (top) and Response (bottom)
4. Also locate the exact **Timestamp** via Response -> Headers -> Date field

**Key mapping:**
- `client-request-id` = ASC's **Correlation ID**
- `x-ms-request-id` = ASC's **Request ID**

## Step 2: Analyze Sign-in With ASC's Authentication Troubleshooter

1. Browse to Azure Support Center -> Open your case SR #
2. Browse to Tenant Explorer -> Sign-Ins blade
3. Choose the Diagnostics tab
4. Open the link for "Go to the new sign-in troubleshooting experience for this tenant"
5. Choose "Search by Correlation ID or Request ID"
6. Enter the value from Step 1 and press Next
   - `client-request-id` = Correlation ID
   - `x-ms-request-id` = Request ID
7. Review the Summary and Expert View to understand the cause of the sign-in failure
8. Use the **Copy Link** button to share results with TA or Escalation colleague
