# SBSL guide for unknown problem investigation

If needed, seek help from a Subject Matter Expert (SME) or use the slow logo virtual team (slowlogonVT) and slow boot virtual team (slowbootVT) for unknown problems. If possible, root cause the problem yourself using data analysis and troubleshooting techniques learned from training and research. 

Take the SBSL (Slow Boot Slow Logon) Training and learn how to analyze data for new issues.  

Course Link: [http://aka.ms/SlowBootSlowLogon](http://aka.ms/SlowBootSlowLogon)
  
![SBSL Main](/.attachments/SBSL/SBSL_Main.png)

Search Streams for content: [DbgLearn](https://microsoft.sharepoint.com/teams/DbgLearn) or [CloudAcademy](https://cloudacademy.com)

## 'Kill the snake' by creating content and sharing with others 

If you found and root caused a new problem, it is good practice to create an internal article documenting the issue using the SBSL prefix. Share your findings and article with the SBSL CORE team by sending it to [sbslcore@microsoft.com](mailto:sbslcore@microsoft.com).

This practice avoids other people working on the same issue and reduces the time to resolve cases. 
- See KB [4056426](https://internal.evergreen.microsoft.com/topic/4056426): ADDS: Tools: Kill The Snake. Driving a culture of killing repeat problems in your POD and replicating it to all others.  

## Content good practice

When creating internal content, it is good practice to use the following as a template for understanding what type of information should be included:
- KB [3147689](https://internal.evergreen.microsoft.com/topic/3147689): Bemis: Article template for Commercial Support Bemis articles 
- KB [4018870](https://internal.evergreen.microsoft.com/topic/4018870): Content: How to write an internal KB article for Commercial Windows 

See also other SBSL resources for further guidance.
