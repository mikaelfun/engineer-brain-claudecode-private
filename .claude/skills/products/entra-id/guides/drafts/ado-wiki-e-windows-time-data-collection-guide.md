This data collection guide is for common Windows Time problems or failures. For specific issues, you may need to collect extra data or logging, such as a network trace or Windows Time debug log.

[[_TOC_]]

### Summary
This guide provides a comprehensive plan for collecting data to troubleshoot Windows Time synchronization issues. It includes steps for both client machines and domain controllers, as well as advanced data collection methods.

### Data collection plan

- Get a detailed description of the problem. Use scoping questions to gather all possible information about the problem, the environment, and the deployment.
- Enable the Windows Time debug log on the machine with the problem and the Network Time Protocol (NTP) server (the domain controller to which the client is targeting).
- Get simultaneous network traces from both the NTP client and server.
- Enable audit to track the process or account used when the time changed (useful in uncontrolled time jump scenarios).
- Gather a Windows Time Service Service Delivery Platform (SDP) if possible.
- Ensure the traces are taken simultaneously while reproducing the issue.

:memo: **Note:** Collecting data from both a working and non-working system can be helpful if you are having difficulty finding the issue with just the non-working data set.

### Client machine data collection

Follow these steps from a computer experiencing Windows Time synchronization issues.

#### Action plan TSS PowerShell

**Preparation:**  
1. Download the tools from the following URL: [TSS](http://aka.ms/gettss).
2. Create a directory on the machine where the tracing will run, for example, **_c:\ms_**, and extract the files to it.

**Execution:**  
1. Open an elevated PowerShell and navigate to the **_c:\ms_** directory. Start the TSS capture by executing:
   ```powershell
   .\TSS.ps1 -ADS_W32Time -Netsh -PSR -AcceptEula
   ```
2. Reproduce the issue you are trying to capture. Once completed, proceed to step 3. **Note: Please keep note of the timestamp of the error in the following format HH:MM:SS.**
3. In the same PowerShell window and path (**_c:\ms_**), stop the TSS when the following message appears on the PowerShell screen.
4. Zip the folder created in **_C:\MS_DATA_** and upload it to the workspace.

**Workspace:**

:warning: This action plan is customer ready.

:memo: **Notes:**

- **w32tm /query /source**
  - **"Domain Controller Fully Qualified Domain Name (FQDN)"**: For domain-joined Windows clients and member servers that use the domain hierarchy to determine their time source (NT5DS), the source should be one of the domain controllers.
  - **"A hostname or IP of an external NTP server"**: This is the expected output when querying the source on the Primary Domain Controller Emulator (PDCe).
  - **"VM IC Time Synchronization Provider"**: In this case, the computer is using Hyper-V virtualization services as its time source. It is recommended not to use virtualization services as a time source for a domain-joined computer (Hyper-V or VMware); use the domain hierarchy method (NT5DS) for Windows clients and servers and a reliable external NTP source for the root PDC.
  - **"Local CMOS Clock"**: If this is a physical computer, it means it is using the hardware clock as its time source. If this is a VMware guest, it means that the virtual machine is synchronizing to the VMware host.
  - **"Running System Clock"**: The computer is not using any external source but depends on the time tick generated by the System Idle Process running on the computer.

### Domain controllers data collection

To troubleshoot domain controllers' time sync issues, run this script from an elevated PowerShell window to determine the time status, NTP type, and time difference of all domain controllers in the forest.

```powershell
# This script queries and presents the time replication configuration and status on all DCs in the forest 
$VerbosePreference = "Continue" 
$Ntp = Read-Host "Please enter the name or IP of the NTP server `nThis information will be used to measure the time difference from the DCs to the NTP server" 
$domains = [System.DirectoryServices.ActiveDirectory.Forest]::GetCurrentForest().Domains 
$dcs = @() 
$PDC = "" 
foreach ($domain in $domains) 
{ 
    $domain.DomainControllers | % {$dcs += $_.name; if ($_.Roles -contains "PdcRole") {$PDC = $_.Name}} 
}
function New-DC() 
{ 
  param ($Name, $TimeSource, $TimeConfig, $OffsetFromNTP) 
  $DC = new-object PSObject 
  $DC | Add-Member -type NoteProperty -Name Name -Value "" 
  $DC | Add-Member -type NoteProperty -Name TimeSource -Value "" 
  $DC | Add-Member -type NoteProperty -Name TimeConfig -Value "" 
  $DC | Add-Member -type NoteProperty -Name AnnounceFlags -Value "" 
  $DC | Add-Member -type NoteProperty -Name OffsetFromNTP -Value "" 
  $DC | Add-Member -type NoteProperty -Name IsRootPdce -Value "" 
  return $DC 
} 
$dcArr = @() 
foreach ($dc in $dcs) 
{ 
    Write-Verbose "Connecting to $dc" 
    $dc1 = New-DC 
    $dc1.Name = $dc 
    if ($PDC.ToLower() -eq $dc.ToLower()) 
    { 
        $dc1.IsRootPdce = "Yes" 
    } 
    else 
    { 
        $dc1.IsRootPdce = "No" 
    } 
    if ((Test-Connection $dc -Quiet) -eq $false) 
    { 
        Write-Verbose "Server $dc is unreachable" 
        continue 
    } 
    $session = New-PSSession -ComputerName $dc1.Name 
    Write-Verbose "Collecting info from $dc" 
    $config = Invoke-Command -Session $session -scriptblock {w32tm /query /configuration} 
    foreach ($c in $config) 
    { 
        if($c -match "Type:") 
        { 
            $type = $c 
        } 
        if($c -match "NtpServer:") 
        { 
            $NtpServer = $c 
        } 
        if($c -match "AnnounceFlags:") 
        { 
            $AnnounceFlags = $c 
        } 
    } 
    $dc1.TimeSource = Invoke-Command -Session $session -scriptblock {w32tm /query /source} 
    $dc1.TimeConfig = $type + "`t" + $NtpServer 
    $dc1.AnnounceFlags = $AnnounceFlags 
    # The following command is saved as a variable so we can pass the $ntp variable to the remote session 
    $cmdStr = '$offset = (w32tm /stripchart /computer:' + "$ntp" + ' /samples:1 /dataonly); $offset[3]' 
    $command = [scriptblock]::Create($cmdStr) 
    $dc1.OffsetFromNTP = Invoke-Command -Session $session -scriptblock $command 
    Remove-PSSession -Session $session 
    $DCArr += $dc1 
    $dc1 = $null 
    $config = $null 
    $NtpServer = $null 
    $type = $null 
    $AnnounceFlags = $null 
} 
$DCArr | Out-GridView
```

### Data collection - Advanced description

This section covers the additional data to collect when troubleshooting scenarios where the Windows Time Service failed to synchronize. Below is a list of additional pieces of data and their main uses:

| **Data** | **Uses** |
|--|--|
| **Network Traces** | It is important to enable simultaneous network traces on both the Windows NTP client and NTP source while forcing w32time sync, that is, **W32tm/resync/rediscover**. |
| **Windows Time Service Debug Log** | Enable this logging in cases where the Windows Time Service fails to sync with its source. |

### Network traces

To start the capture, execute the following command from an elevated Command Prompt:

```cmd
netsh trace start capture=yes tracefile=c:\%computername%_netmon.etl persistent=yes scenario=NetConnection
```

To stop the capture, execute the following command:

```cmd
netsh trace stop
```

:warning: **Note:**
- Enable the trace on the source NTP server. For **NTD5DS** clients, the source server is the domain controller.
- Run **_W32tm /resync /rediscover_** and stop the traces.
- To stop the capture, run: **netsh trace stop**

### Windows Time Service debug log

To start the logging, stop the service, configure the debug log, and start the service:

```cmd
net stop w32time
```

|  |  |
|--|--|
| **For Windows 2000 to 2012R2/8.1** | w32tm /debug /enable /file:%SystemRoot%\temp\W32Time.log /size:10485760 /entries:0-300 |
| **For Windows 2016/10 and later** | w32tm /debug /enable /file:%SystemRoot%\temp\W32Time.log /size:10485760 /entries:0-1003 |

```cmd
net start w32time
```

Wait 30 seconds for the service initialization and resync the time by running **_W32tm /resync /rediscover_**.

An alternative is to run the _stripchart_ command to see if there is a time difference between the NTP source server and local time by running:

```cmd
w32tm /stripchart /computer:myntp.consoto.com
```

Since **Windows Server 2019**, there are also two specific **event logs** to collect: **_Microsoft-Windows-Time-Service/Operational_** and **_Microsoft-Windows-Time-Service-PTP-Provider/PTP-Operational_**.

### Security audit for time jump scenario

Security event 4616 is always logged regardless of the audit "**Security State Change**" sub-category setting, so there is no need to enable any audit setting.

This is described at [4616(S) The system time was changed. - Windows 10 | Microsoft Learn](https://learn.microsoft.com/en-us/previous-versions/windows/it-pro/windows-10/security/threat-protection/auditing/event-4616).

![Workflow_Windows_Time_Computer_policy](/.attachments/WindowsTime/Workflow_Windows_Time_Data_Collection.gif)

Export both **Application**, **Security**, and **System** logs for the last 7 days when the issue occurs again (in .evtx format).

Check if the Windows Time service (in Service Control Manager in the System event log) was started or not.

Check Kernel-General entries indicating the time has changed (Event ID 1 in the System event log).

Since Windows 2019 and Windows 10 RS3, there are new event logs at **Microsoft-Windows-Time-Service/Operational** containing Time Service data. Check [Windows Time for Traceability](https://learn.microsoft.com/en-us/windows-server/networking/windows-time-service/windows-time-for-traceability?tabs=266).
