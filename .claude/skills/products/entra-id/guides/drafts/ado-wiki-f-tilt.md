## Purpose

To reduce overall time spent investigating telemetry issues that impact customers, you can work with customers to run this tool [Telemetry Investigation Logging Tool (TILT).](https://www.microsoft.com/download/details.aspx?id=106302)
The tool will collect data needed to diagnose issues with the M365/office applications.

##Intructions using the Telemetry Investigation Logging Tool.

- Download the TelemetryInvestigationLoggingTool.exe using the download button above.
- Once downloaded copy the executable into a folder where you would like to run it from and where the output logs of the tool will be generated.
- To run the tool, open a command prompt and navigate to the folder where you placed the executable.
- Run the executable at the command prompt:
  - TelemetryInvestigationLoggingTool.exe [-h | -help] [-path <output_path>]
- This will run the tool and generate the output logs.


## Additional Information
If the tool can diagnose an issue, then the tool will list potential mitigation steps that you may choose to follow to help enable proper collection of information needed to power reporting experiences. Please review these potential mitigations and take action to correct them as appropriate for your environment.\
The tool will produce a structured set of folders containing the output of the tool run, in addition, a ZIP file with that content is also created to simplify packaging of the data if you choose to provide it to Microsoft support.

-   The output can be found in a "TILT Output" folder, located in the same directory as the executable or in a directory specified in the command line when the tool was run. You will also find a "TILT Output.zip" file in this same folder which is the packaged output.
-   The "TILT Output" folder contains a series of subfolders that hold the collected information. We strongly recommend reviewing the contents of these folders before providing the ZIP file or these folders to Microsoft to ensure you are comfortable with the information being provided.
    -   TILT Generated Metadata - Contains 2 files generated by the tool
        -   TILT_Mitigation_Steps.txt - contains a record of any proposed mitigations that the tool provided during its run. This file is a great resource for following up on mitigations so that you can review them at your leisure.
        -   TILT_Tool_Logs.txt -- This file contains the information that was gathered by the tool and provides an opportunity for you to review the information before providing it to support.
    -   Rules Files
        -   These are files that are included with the Office applications that include details on the type of telemetry that is enabled on the client. Collecting them allows us to determine if they are out of date.
    -   Collected Office Diagnostic Logs
        -   This collects any diagnostic logs that have been generated on the client. The generation of these logs is governed by your privacy controls so it is possible that some of the subfolders under this path will be empty.

## Verifications to Run:

The following list contains known verifications that should check and record the outcome of. Where possible customer deployable mitigations are noted.

**1. DisableTelemetry regkey is deployed**

- Check the following registry locations record a Boolean for each, True if the regkey is found, False if it is not present. We should also write which path the key was found in as that will indicate which mitigation mechanism is most needed (OCPS update, GPO update, other).

  - HKEY_CURRENT_USER\Software\Policies\Microsoft\Cloud\Microsoft\Office\Common\ClientTelemetry

  - HKEY_CURRENT_USER\Software\Policies\Microsoft\Office\Common\ClientTelemetry

  - HKEY_CURRENT_USER\Software\Microsoft\Office\Common\ClientTelemetry\DisableTelemetry

  - HKEY_LOCAL_MACHINE\Software\Policies\Microsoft\Cloud\Microsoft\Office\Common\ClientTelemetry\DisableTelemetry

  - HKEY_LOCAL_MACHINE\Software\Policies\Microsoft\Office\Common\ClientTelemetry\DisableTelemetry

  - HKEY_LOCAL_MACHINE\Software\Microsoft\Office\Common\ClientTelemetry\DisableTelemetry

**2. User is logged into Office**

- User should be logged into Office app in order for data to flow into telemetry.

**3. User privacy controls are appropriate )**

- Verify the contents of Group Policy and Local Device registry hives that store the privacy control values.
  - [Overview of privacy controls for Microsoft 365 Apps for enterprise](https://learn.microsoft.com/microsoft-365-apps/privacy/overview-privacy-controls)
  - [Use policy settings to manage privacy controls for Microsoft 365 Apps for enterprise](https://learn.microsoft.com/microsoft-365-apps/privacy/manage-privacy-controls)

**4. Office Telemetry Endpoint are accessible**

- Customer should attempt to ping each of the following endpoints and record the HTTP status code or ping result

  - nexusrules.officeapps.live.com

  - nexus.officeapp.live.com

  - kusto.aria.microsoft.com

  - events.data.microsoft.com

  - otelrules.svc.static.microsoft

  - ecs.office.com

  for example you could loop through all of the hostname and test connectivity to port 443 (HTTPS)

  ```PowerShell
  # Define the hostnames and ports
  $hostnames = @(
      "nexusrules.officeapps.live.com",
      "nexus.officeapp.live.com",
      "kusto.aria.microsoft.com",
      "events.data.microsoft.com",
      "otelrules.svc.static.microsoft",
      "ecs.office.com"
  )
  $ports = @(443)

  # Initialize an array to store the results
  $results = @()

  # Loop through each hostname and port, and run Test-NetConnection with detailed information level
  foreach ($hostname in $hostnames) {
      foreach ($port in $ports) {
          try {
              $result = Test-NetConnection -ComputerName $hostname -Port $port -InformationLevel Detailed
              $status = @{
                  Hostname = $hostname
                  Port = $port
                  PingSucceeded = $result.PingSucceeded
                  TcpTestSucceeded = $result.TcpTestSucceeded
                  DetailedOutput = $result
              }
              $results += $status
          } catch {
              $status = @{
                  Hostname = $hostname
                  Port = $port
                  PingSucceeded = $false
                  TcpTestSucceeded = $false
                  DetailedOutput = "Failed"
              }
              $results += $status
          }
      }
  }

  # Output the summary of results
  Write-Output "Summary of Results"
  Write-Output "------------------"
  foreach ($result in $results) {
      Write-Output "Hostname: $($result.Hostname)"
      Write-Output "Port: $($result.Port)"
      Write-Output "Ping Status: $($result.PingSucceeded)"
      Write-Output "TCP Connection Status: $($result.TcpTestSucceeded)"
      Write-Output ""
  }
  ```

**5. User network is not metered**

- Check the network type of metering settings and record the metering setting

**6. Rules download folder is accessible**

- Verify that the rules folders mentioned below have Read and Write permissions and write a Boolean for each where True means accessible and False means inaccessible. There should be a value for each of Read and Write for each folder.

  - %localappdata%\Microsoft\Office\16.0\

  - %localappdata%\Microsoft\Office\OTele\

**7. The application version of Office being run**

- Record the full build number of that application

**8. User?s country setting associated with their account or tenant**

- Record the country identifier associated with their account

**9. Is the user running in a restricted environment**

- Indicate if the user is running in a restricted environment like GCC, DoD, Sovereign, etc.). Logging the specific TelemetryRegion would be helpful if that is not EUPI data (or if it is EUPI lets make sure we get the right privacy review)

## Additional Logs to Gather:

In addition to generating a response file with the results of the checks listed previously, also package the following data into a zip or similar file for inclusion in the package sent to engineering via ICM for analysis. Each item in the list below should be placed in a separate folder within the upload package to ensure easy consumption.

- The contents of Rules folders, including all Rule.xml files (\*.rules.xml)

- All diagnostic logs in the %temp%\Diagnostics folder. There should be one folder for each application, all should be collected (EUPI, but should only contain data consistent with the users Privacy controls)
