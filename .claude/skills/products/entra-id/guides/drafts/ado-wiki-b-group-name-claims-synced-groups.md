# Feature Overview

**Group Name claims for on-premises groups** allows applications that have been deployed on premises on AD FS, and consume the names of the user's groups for in-app authorization purposes, to be moved from AD FS to Azure Active Directory.

Entra ID has been able to emit group claims containing the Entra ID object ID of each group that a user is a member of, but this isn't useful for existing apps that have been deployed on Entra ID and need to be moved. With this addition customers can configure the application registration in Entra ID to emit the name of the group instead of the object ID.

Supported name format options are **SAMAccountName, NetbiosDomain/SAMAccountName, DNSDomain/SAMAccountName, OnPremisesSecurityIdentifier,** in addition to **group OID**.

Configuration is via the Application Manifest, or the Enterprise Applications SSO experience.

## Common Scenarios

Moving SAML or WS-Fed (esp SharePoint On-Prem) apps from ADFS RP to AAD RP

  - User and group management remains mastered on prem
  - On premises, many apps consume user group info in the form of group names source from attributes on the on-prem AD groups
  - AAD, up until now only provided group information in the form of Group Object IDs, so groups can’t be moved.
  - This feature adds the capability to emit group or role claims in the following forms for groups synced from on-premises

Adds group claim formats:

  - SAMAccountName
  - DNSdomain\\SAMAaccountName
  - NetBiosDomain\\SAMAccountName
  - OnPremGroupSID

Group membership evaluation is transitive (nested groups are supported)

## Architecture Flow walk-throughs

  - SAAS App A consumes group claims in group name format. (generated by tokenGroups() fn)
  - When SAAS App A is migrated to AAD, when the same user access it, the token should be configured to contain the same groups in the same format as the original so no changes to the app or migration/re-mapping of app data required

# How to configure & manage

-	**Configure group claims for a gallery or non-gallery application via single sign-on (SSO):**

1. 	Open Enterprise Applications, select your application in the list, click on Single Sign On configuration, and then click on User Attributes & Claims.

2.	Select Add a group claim.
 

3.	Use the options to select which groups should be included in the token.
 

4.	To emit groups by using Active Directory attributes synced from Active Directory, select the required format from the Source attribute drop-down list.  

-	**Configure group claims for App registration:** 

1. Open App Registration, select the application in the list, select Token configuration, and then Add Groups claim.

2.	Use the options to select which groups should be included in the token
 
3.	Specify which token-type optional claim should be applied in the optional claims section

    •	idToken for the OIDC ID token

    •	Access Token for the OAuth/OIDC access token

    •	Saml2Token for SAML tokens  

4.	To emit groups by using Active Directory attributes synced from Active Directory select the required format from the Source attribute drop-down list. 

 
-   **Configure group claims in the optional claims section of the application manifest:**
1. 	In the portal, select Azure Active Directory > Application Registrations > Select Application > Manifest.
2.	In the portal, select Azure Active Directory > Application Registrations > Select Application > Manifest.

      Valid values are:
      - All
      - SecurityGroup
      - DirectoryRole
      - ApplicationGroup
      - None
 
For example: "groupMembershipClaims": "SecurityGroup"
 
By default, group **ObjectID** attributes will be emitted in the group claim value. To modify the claim value to contain on-premises group attributes, or to change the claim type to a role, use the optionalClaims configuration described in the next step.

3.	Set optional claims for group name configuration.

If you want the groups in the token to contain the on-premises Active Directory group attributes, specify which token-type optional claim should be applied in the optionalClaims section. You can list multiple token types:

- idToken for the OIDC ID token
- accessToken for the OAuth/OIDC access token
- Saml2Token for SAML tokens

For each relevant token type, modify the group claim to use the optional Claims section in the manifest. The optional Claims schema is as follows:

![GroupsClaims.png](/.attachments/GroupsClaims-06458db7-3333-41f4-8ed6-ad97925835a1.png)
 

In additionalProperties, only one of "sam_account_name", "dns_domain_and_sam_account_name", or "netbios_domain_and_sam_account_name" is required. If more than one is present, the first is used and any others are ignored.

Some applications require group information about the user in the role claim. To change the claim type to from a group claim to a role claim, add "emit_as_roles" to additional properties. The group values will be emitted in the role claim.

To emit group display name for cloud-only groups, you can add "cloud_displayname" to additional properties. This option will work only when “groupMembershipClaims” is set to ApplicationGroup

# TSG Scenarios

## Likely issues

  - For users with large numbers of group memberships, the same limits apply as for group claims in the existing format.   (We’ll be addressing this with the ability to limit groups to those assigned to the application)
  - [Token bloat](https://blogs.technet.microsoft.com/tspring/2013/07/25/token-bloat-apparently-not-caused-by-spicy-food/) with large numbers of groups can cause errors due to token size in some circumstances
  - Doesn’t yet work with Cloud only groups (and therefore cloud only users), since the same name attribites are not supported and managed directly by AAD. Mis-understanding this is likely to be a source of support cases.  We’ll be addressing the gap later this year.
  - Group names not synced.  Need to use up-to-date AD Connect version > 1.2.70;
  - It’s hard to tell if group attributes are actually synced as they are not exposed in the group UI or in graph;
  - if “Emit groups as role claims” are selected on “Groups optional claims configuration”, group claim will be changed to the role claim, and the group values will be emitted in the role claim;

  - If you are using Implicit flow, Group claims have a five-group limit, if the user is in more than five groups, the tokens requested via the implicit flow will have a _"hasgroups":true_; 

   - If “Security Groups” option is selected, make sure the number of groups where user is a member does not exceed the limit that Entra ID will add to a token. The number of groups emitted in a token is limited to 150 for SAML assertions and 200 for JWT.

  - Can’t emit groups and roles mixed together as role claims;

  - Security:
      - Name clashes are possible.  Within AD domains group names are unique but may be may not be when multiple AD domains are synced to the same AAD tenant.   Qualifying group name with the domain name helps but doesn’t guarante uniqueness.  Recommend transitioning to AAD Object ID or onPremGroupSID
      - Some customers may want to use the group display-name.   We believe this is insecure enough to not support it on AAD.
      - PG would like to know if cases arise on either name clashes or group display-name occur please – contact [paulgarn@microsoft.com](mailto:paulbgarn@microsoft.com)
