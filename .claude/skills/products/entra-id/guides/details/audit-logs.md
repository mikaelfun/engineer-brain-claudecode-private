### Audit records show app@sharepoint as the user performing file activities (search queries, file access, list operations) instead of a real user account
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: The application has SharePoint App-Only access, performing organization-wide actions on behalf of a user/admin/service. Common triggers: M365 Groups creation, Teams creation, compliance features (retention policies, eDiscovery holds, auto-labeling)

**Solution**: This is expected behavior. Check ApplicationDisplayName and EventData fields in the audit record to identify the triggering scenario. Multiple rapid audit records from app@sharepoint within seconds indicate automated application-triggered actions

### Customer sees "Set Company Information" audit log entries performed by a Global Admin account without their knowledge.
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: When a Rave engineer runs a troubleshooting flow, a backend process impersonates a random Global Admin to obtain tenant info. The audit trail is incorrectly labeled as "Set Company Information" instead of "Set Application Data".

**Solution**: Verify by running Kusto query on GlobalIfxUlsEvents checking for "SetApplicationData" with matching correlationId and tenant. Look for CN=boxuserimp.m365admin.office.com. If confirmed, provide statement: "Benign internal operation when domain health is checked by support agents." If not accepted, open ICM to Office365 Admin Portal > Support Central.

### "Update company" audit log shows company display name changed but no actor ("initiated by" is empty).
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: Known issue. "Update company" event has no actor but has modified properties. "Set Company Information" event has actor but no modified properties. Feature request 1200929.

**Solution**: Check if both "Update company" and "Set Company Information" entries exist at similar timestamps - actor from Set Company Information made the change (via Office portal). If no Set Company Info entry, check Jarvis query (requires SAW/AME). Also check Commerce via Jarvis.

### Unfamiliar audit log activities appear such as Features_GetFeaturesAsync, GroupsODataV4_Get, Group_GetDynamicGroupProperties in Entra ID
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: These activities are logged by the Self-Service Group Management (SSGM) service. They are triggered by internal service calls when users access MyApps, MyGroups portal, or Azure portal group pages.

**Solution**: These are normal internal service activities and can be safely ignored. Reference: https://learn.microsoft.com/en-us/entra/identity/monitoring-health/reference-audit-activities#self-service-group-management

## Phase 2: Gbl

### Group-based licensing (GBL) fails with 'Conflicting service plans' error when assigning E5 license to group while users already have E3 license (di...
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: Service plans Exchange Online Plan 2, SharePoint Online Plan 2, Skype for Business Online Plan 2 cannot coexist from different SKUs (E3 and E5 both include them). Azure AD will not auto-deconflict.

**Solution**: 1) Disable conflicting service plans on E3 direct assignment before applying E5 via group. 2) Or modify group E5 license assignment to disable overlapping plans. 3) Or remove E3 license if E5 is superset. Use ASC Tenant Explorer > Audit Logs with filter Activity='Set group license', Result='Failure' to diagnose. Check violation details with MSODS Kusto query filtering tagId=='9ln1' and message has 'violation'.

## Phase 3: Tap
> 2 related entries

### User is prompted for password instead of Temporary Access Pass (TAP) passcode after admin configured TAP
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: TAP policy not enabled for user, TAP not yet active (up to 10 min activation delay), TAP expired or already used (one-time), or policy changed from multiuse to one-time after TAP was issued

**Solution**: Verify: 1) TAP policy is enabled and user is assigned, 2) TAP has not expired, 3) For one-time TAP check audit logs for AccessPassUsage=3 (Used), 4) If policy changed, delete old TAP and issue new one

### User is prompted for password instead of TAP passcode after entering UPN
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: TAP policy not enabled for user, TAP not yet activated (up to 10 min delay), TAP expired, or TAP one-time already used

**Solution**: 1) Verify TAP policy is Enabled and user is assigned. 2) Wait up to 10 min after creation. 3) Check TAP expiry. 4) Check Audit log for AccessPassUsage=3 (Used). 5) If valid TAP exists but blocked, delete and recreate (Issue 7).

## Phase 5: Pim
> 1 related entries

### PIM eligible assignment extend request fails with RoleAssignmentRequestPolicyValidationFailed in audit logs. No detailed error message shown in Azu...
**Score**: 🟢 9.0 | **Source**: OneNote

**Root Cause**: The extend request duration exceeds the PIM policy MaximumGrantPeriod (default 365 days). The ExpirationRule policy validation fails silently with only RoleAssignmentRequestPolicyValidationFailed shown.

**Solution**: 1) Check Jarvis logs with timestamp + correlation ID to see detailed failure reason. 2) Look for 'The duration in the request is greater than maximum allowed duration' in Jarvis logs. 3) Adjust the request StopDateTime to be within MaximumGrantPeriod, or increase MaximumGrantPeriod in PIM role settings. Ref: https://learn.microsoft.com/en-us/rest/api/authorization/includes/privileged-role-common-errors

## Phase 6: Proxycalc
> 1 related entries

### IT admin sees mass 'Update User' events in Azure AD audit logs with no Actor information, causing confusion about what triggered the bulk user changes
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: Backend ProxyCalc (Proxy Address sanitization) task triggered by VerifiedDomainChange event. Adding/removing a verified domain triggers this async task to ensure all UPNs and ProxyAddresses are consistent; audit logs do not yet display Actor info correlating the user updates to the triggering VerifiedDomainChange event.

**Solution**: Check audit logs for an 'Add verified domain' or 'Remove verified domain' event 5-10 minutes before the mass 'Update User' events — this is the trigger. ProxyCalc-driven updates are by-design behavior ensuring UPN/ProxyAddress consistency across the tenant. No action needed unless actual UPN/proxy address changes are unintended.

## Phase 8: Reporting Api
> 1 related entries

### Microsoft Graph Reporting API returns 429 'Too Many Requests' or 'UnknownError' with message 'This request is throttled' when querying auditLogs/si...
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: Reporting API throttling: 5 req/10s per app per tenant, 122/10s across all tenants. Complex filters and large result sets increase throttling risk.

**Solution**: 1. Retry with Retry-After header. 2. Reduce $top (start 900, decrease by 100). 3. Narrow $filter time range. 4. Simplify $filter. 5. Run during off-peak hours. 6. Combine solutions.

## Phase 10: Authentication
> 1 related entries

### Individual user account disabled and cannot authenticate. Audit log shows: "Core Directory, User Management, Disable account, Success, <user name>,...
**Score**: 🔵 5.5 | **Source**: ADO Wiki

**Root Cause**: User is subject to government trade sanctions. The PTSS (trade screening) application automatically disabled the user account by setting accountEnabled=false, per Microsoft compliance obligations with governmental sanctions decisions.

**Solution**: Verify via audit log (look for PTSS as disabling app) or CMAT portal (cmat.azure.com -> Email Address tab -> check Trade Screening Status = Blocked). Get TA confirmation first. If blocked user != requester: send CELA message about government restriction. If blocked user = requester: send CELA message and close immediately. Do NOT re-enable account. Escalation: tradehlp@microsoft.com with Tenant Name, Tenant ID, user ID, audit log entry.

### Cannot identify who enabled or disabled report privacy settings (concealed user/group/site names) in Microsoft 365 Admin Center, no native audit lo...
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: Microsoft 365 Admin Center does not provide native audit logs for report settings changes

**Solution**: Use Microsoft Purview Audit solution: 1) Ensure audit log search is enabled (on by default for enterprise orgs). 2) Login to Microsoft Purview (purview.microsoft.com) with admin permissions. 3) Search for operation UpdatedUsageReportsSettings (friendly name: Updated usage report privacy settings). ## Phase 12: Isexchangecloudmanaged
> 1 related entries

### Set IsExchangeCloudManaged to True and edited Exchange attribute in EXO but change did not take effect
**Score**: 🟢 8.5 | **Source**: ADO Wiki

**Root Cause**: SOA transfer for Exchange attributes to Exchange Online takes up to 24 hours to complete. Individual attribute updates after SOA transfer can also take up to 24 hours.

**Solution**: Wait 24 hours. Then check Azure Audit logs for the change (Actor: Microsoft Substrate Service). If change not in logs after 24 hours, escalate to EXOManageability team. Changes flow: EXO -> Entra AD -> forward sync back to EXO.
