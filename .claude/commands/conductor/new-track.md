# New Track

Create a new track (feature, bug fix, chore, or refactor) with a detailed specification and phased implementation plan.

## Pre-flight Checks

1. Verify Conductor is initialized:
   - Check `conductor/product.md` exists
   - Check `conductor/tech-stack.md` exists
   - Check `conductor/workflow.md` exists
   - If missing: Display error and suggest running `/conductor:setup` first

2. Load context files:
   - Read `conductor/product.md` for product context
   - Read `conductor/tech-stack.md` for technical context
   - Read `conductor/workflow.md` for TDD/commit preferences

## Track Classification

Determine track type based on description or ask user:

```
What type of track is this?

1. Feature - New functionality
2. Bug - Fix for existing issue
3. Chore - Maintenance, dependencies, config
4. Refactor - Code improvement without behavior change
```

## Interactive Specification Gathering

**CRITICAL RULES:**

- Ask ONE question per turn
- Wait for user response before proceeding
- Tailor questions based on track type
- Maximum 6 questions total

## Track ID Generation

Generate track ID in format: `{shortname}_{YYYYMMDD}`

- Extract shortname from feature/bug summary (2-3 words, lowercase, hyphenated)
- Use current date
- Example: `user-auth_20250115`, `nav-bug_20250115`

Validate uniqueness:

- Check `conductor/tracks.md` for existing IDs
- If collision, append counter: `user-auth_20250115_2`

### Phase Guidelines

- Group related tasks into logical phases
- Each phase should be independently verifiable
- Include verification task after each phase
- TDD tracks: Include test writing tasks before implementation tasks
- Typical structure:
  1. **Setup/Foundation** - Initial scaffolding, interfaces
  2. **Core Implementation** - Main functionality
  3. **Integration** - Connect with existing system
  4. **Polish** - Error handling, edge cases, docs

### Verification Plan Guidelines

When generating the `## Verification Plan` section in plan.md, follow these rules:

**核心原则：只要不是 D365 写入和执行操作，都可以自动化验收。** 不用担心花费时间和本地数据操作——数据都是可恢复和重构的。优先选择 E2E 类型，只有涉及 D365 写入/执行的才用 Skip。

1. **Read the acceptance criteria** from the spec — each criterion becomes a row in the Verification Plan table
2. **Classify each criterion** using the rules from `conductor/workflow.md` Step 2a:
   - **E2E** — Workflow outputs, file generation, script behavior, data pipeline, performance — backup → run → verify → restore
   - **Interaction** — User actions: clicking buttons, filling forms, opening/closing dialogs, state transitions
   - **Visual** — Appearance: layout, styling, theme, responsive design, icon display
   - **API** — Backend behavior: new endpoint, changed response shape (when no workflow execution needed)
   - **Skip** — **仅限** D365 write operations (add-note, record-labor, SAP update) or dangerous execute actions
3. **E2E 优先原则**：对于 refactor/feature tracks 涉及工作流、脚本、文件生成的，**默认用 E2E** 而非 Skip 或 API
4. **E2E test steps 必须包含**：数据备份 → 场景构造 → 执行实际脚本/工作流 → 验证文件输出 + API 响应 + UI 渲染 → 数据恢复
5. **Write high-level test steps** — describe WHAT to test, not the exact Playwright/bash code (the verify agent will generate code later)
6. **Reference the safety red lines** from CLAUDE.md — never plan tests that click Full Process, Troubleshoot, Draft Email, or Execute buttons
7. **For Interaction tests**, prefer API-seeded data patterns (create test data via API, then interact via UI)

**Example output (E2E-heavy refactor track):**

```markdown
## Verification Plan

| # | Acceptance Criterion | Test Type | Test Steps |
|---|---------------------|-----------|------------|
| 1 | generate-todo.sh 规则矩阵正确 | E2E | Backup todo/ → 构造 6 种 meta.json 场景 → 运行 generate-todo.sh → 验证 stdout 格式 + 文件内容 → restore |
| 2 | case-summary NO_CHANGE 跳过 | E2E | 确保 case-summary.md 存在 → 记录 mtime → 模拟 NO_CHANGE 快速路径 → 验证 mtime 未变 + todo 仍生成 |
| 3 | Dashboard Summary tab 渲染 | E2E | Backup → 删除 case-summary.md → API 验证 legacy=true fallback → 创建新 summary → API 验证 legacy=false → Playwright 截图验证 → restore |
| 4 | casework 快速路径无 LLM | Skip | 需运行完整 casework（D365 交互），跳过 |

**Test Type Legend:**
- **E2E** — Backup data → run actual workflow/script → verify file outputs + API + UI → restore
- **Interaction** — Playwright clicks, form fills, state assertions
- **Visual** — Navigate + screenshot + visual inspection
- **API** — curl/fetch endpoint + assert response
- **Skip** — D365 write/execute operations only (must justify)
```

## Track Creation

After plan approval:

1. Create directory structure:

   ```
   conductor/tracks/{trackId}/
   ├── spec.md
   ├── plan.md
   ├── metadata.json
   └── index.md
   ```

2. Create `metadata.json`:

   ```json
   {
     "id": "{trackId}",
     "title": "{Track Title}",
     "type": "feature|bug|chore|refactor",
     "status": "pending",
     "created": "ISO_TIMESTAMP",
     "updated": "ISO_TIMESTAMP",
     "phases": {
       "total": N,
       "completed": 0
     },
     "tasks": {
       "total": M,
       "completed": 0
     }
   }
   ```

3. Create `index.md`:

   ```markdown
   # Track: {Track Title}

   **ID:** {trackId}
   **Status:** Pending

   ## Documents

   - [Specification](./spec.md)
   - [Implementation Plan](./plan.md)

   ## Progress

   - Phases: 0/{N} complete
   - Tasks: 0/{M} complete

   ## Quick Links

   - [Back to Tracks](../../tracks.md)
   - [Product Context](../../product.md)
   ```

4. Register in `conductor/tracks.md`:
   - Add row to tracks table
   - Format: `| [ ] | {trackId} | {title} | {created} | {created} |`

5. Update `conductor/index.md`:
   - Add track to "Active Tracks" section

## Error Handling

- If directory creation fails: Halt and report, do not register in tracks.md
- If any file write fails: Clean up partial track, report error
- If tracks.md update fails: Warn user to manually register track
