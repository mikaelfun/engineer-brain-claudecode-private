Patrol Page UI 重设计方案

基于真实 state.json 数据结构，解决"每个阶段看不到做了什么"的问题

核心问题 → 设计目标

❌ 现在
  • assess 只显示一个点，不知道在做什么
  • summarize 只显示状态，无结果内容
  • 全局 pipeline 和 case pipeline 信息割裂
  • 阶段间没有时间感
✅ 目标
  • 每个步骤都有明确的子状态和结果
  • assess 显示判定结果（needs-action/no-change）
  • act 显示执行了哪些 action 及结果
  • summarize 显示生成的 todo 条数

Summary

两个方案的共同改进

  • assess 结果可见 — 显示 needs-action / no-change / pending-customer
  • act 子动作链 — troubleshooter → email-drafter → challenger 每个独立显示状态
  • data-refresh 并行子任务 — d365/teams/onenote/icm/attachments 各自显示
  • skipped 步骤明确标注 — 灰色 "skip" 标记，不是空白
  • 每步独立计时 — 不再只有总时间
