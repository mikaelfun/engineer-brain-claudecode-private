Case Row 重新设计 — 3 种方案

核心挑战：Act 阶段包含 不确定数量的 agent（assess → troubleshoot → reassess → challenger → email-drafter），
需要清晰展示动态流程，同时保持视觉整洁。以下每种方案都展示同一个复杂 case 的中间状态。

A Pipeline Header + Action Chain ★ Recommended

顶部水平 4 步 pipeline（Refresh → Assess → Act → Summarize）保持整体进度一目了然。
下方展开 Act 的动作链——水平 flow 排列，每个 agent 是一个小节点卡片，自动换行。兼顾整体进度感和细节可见性。

✓ Pros

  • 整体进度 (4 步) 和详细动作链同时可见
  • 动作链水平排列，自动换行，适应任意长度
  • 视觉层级清晰：pipeline → action chain → details
  • 与当前 CaseworkPipeline 组件风格一致

✗ Cons

  • 动作多时换行可能占较多垂直空间
  • 节点间的连线在换行处断开
B Vertical Timeline

整个 case 流程用一条垂直时间线展示，每个步骤（包括 Act 内的每个 agent）是时间线上的一个节点。
类似 GitHub PR 的 timeline 或 CI/CD pipeline 的纵向展示。自然支持任意长度的动作链。

✓ Pros

  • 所有步骤一条线，不区分"主步骤"和"子步骤"，简单直观
  • 天然支持任意长度，不用考虑换行
  • 每个步骤有足够空间放详细信息
  • 与左侧 patrol pipeline sidebar 的 vertical 风格一致

✗ Cons

  • 失去 4 步 pipeline 的整体进度概览
  • 复杂 case（6+ 步骤）时占较多垂直空间
  • 所有步骤同级，缺少"阶段归属感"
C Swimlane Columns

4 个主阶段变成 4 列泳道，每列内部垂直排列该阶段的具体步骤。
Act 列内展示所有 agent 的迷你时间线。类似看板 (Kanban) 的视觉效果。

✓ Pros

  • 4 个阶段并排可见，整体进度一目了然
  • Act 列可以容纳任意多个 agent
  • Kanban 风格，信息密度高

✗ Cons

  • 列宽分配不均——Act 需要最多空间
  • 简单 case (assess → no-change) 时列内容空洞
  • 水平空间有限时会挤压
  • 与左侧 sidebar 的垂直 pipeline 风格冲突
