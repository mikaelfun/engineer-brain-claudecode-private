# 邮件撰写指南

从 mailplanner agent 提取的领域知识。caseworker 写邮件前读取本文件。

## 参考样本

`playbooks/email-samples/` 目录包含真实邮件样本（.txt + .msg）：
- `initial response.txt` — 首次回复
- `initial response 21v convert.txt` — 21v 转换 Case 首次回复
- `answer customer query.txt` — 回答客户问题
- `follow up.txt` — 跟进
- `closure confirm.txt` — 关单确认（closure-confirm）
- `case closure.txt` — 关单

写邮件草稿前**必须先读取对应类型的样本文件**，严格匹配其格式结构（分隔符、段落顺序、标题名称）。
特别是 closure 类型，**必须**使用 `=============` 分隔的三段式结构，不可用自由格式替代。

## 邮件类型（7 种）

### 1. initial-response（普通 IR）
- **用途**：首次回复客户，建立联系
- **结构**：
  1. 自我介绍（姓名 + 角色）
  2. 问题理解复述（用自己的话总结客户描述）
  3. 初步方向（已知的排查思路或需要的信息）
  4. 下一步说明
- **长度**：中等，不要太短显得敷衍，不要太长

### 2. 21v-convert-ir（21v 转换专用 IR）
- **用途**：从 21v 团队转过来的 Case
- **结构**：
  1. 自我介绍
  2. 说明已接手，提及 21v 工单号和原工程师
  3. 问题理解
  4. 初步方向
- **注意**：使用 21v 转换专用模板，要让客户知道这是同一个问题的延续

### 3. request-info（追要信息/日志）
- **用途**：排查需要客户提供额外信息
- **结构**：
  1. 简短进展说明
  2. 明确列出需要的信息/日志（编号列表）
  3. 说明每项为什么需要
  4. 如何获取（给具体命令或步骤）
- **注意**：一次说清楚，避免来回追问

### 4. result-confirm（排查结论确认）
- **用途**：找到原因/解决方案后确认
- **结构**：
  1. 问题原因总结
  2. 解决方案/建议
  3. 验证步骤（客户可以自行验证）
  4. 请客户确认是否解决
- **注意**：结论要明确，不要模糊

### 5. follow-up（跟进提醒）
- **用途**：客户长时间未回复，或 pending-pg 时需向客户更新进展
- **结构**：轻量 3-5 句
  1. 礼貌提醒上次沟通内容
  2. 询问是否需要进一步帮助（pending-customer 场景），或告知当前状态（pending-pg 场景：如"我们已将此问题升级至产品团队，目前仍在调查中，有进展会第一时间通知您"）
  3. pending-customer 时：提醒不回复可能关单；pending-pg 时：不提关单
- **注意**：不要回顾技术细节，保持简短

### 6. closure-confirm（关单确认）
- **用途**：客户暗示问题已解决（如感谢回复），但未明确说可以关单时发送
- **结构**：轻量 3-5 句
  1. 感谢客户的回复
  2. 引用之前给出的具体方案，确认是否生效
  3. 提议关单，但留余地让客户反馈还有问题
- **与 closure 的区别**：closure-confirm 是**询问**能否关单，closure 是**执行**关单
- **注意**：
  - 不执行任何关单操作（不改 D365 Status）
  - 必须引用具体方案名称，让客户能快速对应上下文
  - 不要过度总结技术细节，客户刚回复过，上下文还在

### 7. closure（关单总结）
- **用途**：问题解决，准备关单
- **必须严格参考** `case closure.txt` 样本的格式
- **结构**（固定三段式 + `=============` 分隔符）：
  1. 开头感谢（1-2 句，如"感谢您的确认。若后续还有任何新的问题，欢迎随时与我们联系"）
  2. **问题描述** `=============` — 用 2-3 句概括客户的原始问题和现象
  3. **问题建议/Resolution** `=============` — 说明根因 + 解决方案/操作步骤，中等详细度
  4. **更多信息/Additional Information** `=============`（可选）— 补充说明、注意事项、参考文档链接
  5. Survey 引导句（"稍后您会收到一封服务质量反馈邮件..."）
- **格式规则**：
  - 三个段落标题后紧跟 `=============` 分隔线，这是固定格式
  - 中文 case 用"问题描述 / 问题建议 / 更多信息"，英文 case 用"Issue Description / Resolution / Additional Information"
  - 解决方案部分要给具体操作步骤（编号列表），不要只写抽象描述
  - 如有 case 范围外的 open items，在"更多信息"段落简要提及并建议后续路径
- **注意**：不要用自由格式（如 Issue/Resolution 加粗标题），必须用 `=============` 分隔符

## 风格约束

- **Subject**：不自编，用原邮件线程的 Subject
- **署名**：完整多行格式，含 Customer Service & Support + Manager 信息
- **语言**：中文为主 + 英文技术术语（如 AKS, ACR, NSG 等保持英文）
- **称呼**：根据客户邮件里的习惯来（X 工、X 先生/女士、Dear X）
- **语气**：专业但不生硬，有工程师的直接感

## 邮件审核流程

caseworker 写好草稿后：
1. 输出到 `${CASES_ROOT}/active/{case-id}/drafts/`
2. 展示给用户审核
3. **不自动发送**，用户自己决定发不发
