🧮 Action 综合评分算法

每个 Action Item 都有一个 0-100 的 urgency score，由以下因子加权计算。

Urgency Score Formula
score = w1 × prod_impact // 生产环境紧急度 (0-30) + w2 × customer_sentiment // 客户情绪感知 (0-20) + w3 × severity // Case 等级 A=25 B=15 C=5 + w4 × contact_gap // 上次联系间隔 (0-15) + w5 × teams_pressure // Teams 催促信号 (0-10) + sla_bonus // SLA < 4h → +20, < 8h → +10
FactorRange数据来源示例
🏭 prod_impact0-30 Case title + SAP 关键词匹配 宕机=30, 性能慢=20, 配置问题=10, 咨询=5
😤 customer_sentiment0-20 最近邮件/note 情绪分析 催促/不满=20, 焦急=15, 中性=5, 满意=0
🔴 severity5-25 Case severity 字段 Sev A=25, Sev B=15, Sev C=5
📭 contact_gap0-15 meta.daysSinceLastContact >5d=15, 3-5d=10, 1-3d=5, <1d=0
💬 teams_pressure0-10 Teams swarming 频道搜索 有催促消息=10, 有讨论=5, 无=0
⏱ sla_bonus0-20 SLA 剩余时间计算 <4h=+20, <8h=+10, 其他=0
🏗️ 数据架构

Dashboard v2 需要的数据源和后端扩展。

现有可用
Pipeline + Metrics
来自 /api/cases：meta.actualStatus (分列)、severity、daysSinceLastContact、irSla、ccAccount (RDSE)、compliance (Entitlement)。无需后端改动。
现有可用
Todo 数据
来自 /api/todos：patrol 生成的 todo + 手动 todo。合并到 Action List，标记来源。
需要新增
Urgency Score 计算
后端新增 scoring service：读取 case meta + 邮件情绪 + Teams 搜索结果 → 输出 score + factors。可在 patrol 或 data-refresh 时计算并缓存到 meta.json。
需要新增
Team Intel Feed
定期（或按需）从 Teams swarming 频道 + WorkIQ 搜集 Known Issue / LSI / 权责变更。存入 dataRoot/intel/ 目录，前端消费。后续 troubleshooter 可直接引用。
