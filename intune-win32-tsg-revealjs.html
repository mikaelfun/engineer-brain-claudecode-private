📐 Architecture Overview

Win32 App 部署通过 Intune Management Extension (IME / SideCar) 执行

Intune Portal
Admin uploads .intunewin
→
Intune Service
Policy & Content
→
IME / SideCar
Agent on Device
IME (IntuneManagementExtension.exe) 是运行在 Windows 客户端上的 Agent 服务， 独立于 MDM OMA-DM 通道，负责 Win32 App、PowerShell Script、Proactive Remediation 的执行。
IME 全称 Intune Management Extension，也叫 SideCar Agent。它是一个 Windows 服务， 独立于 MDM enrollment 通道（OMA-DM）运行。有自己的轮询机制和日志系统。 关键进程：IntuneManagementExtension.exe，服务名 IntuneManagementExtension。

📐 两条并行架构

OMA-DM 通道

Store App / LOB App
通过 MDM Agent (OMA-DM)
→ EnterpriseModernAppManagement CSP
  • MSI / MSIX / Store App
  • MDM Check-in 触发
  • 仅支持简单安装

IME / SideCar 通道 ⭐

Win32 App / PS Script / Remediation
独立轮询 SideCar Gateway
→ StatelessSideCarGatewayService
  • Win32 (.intunewin)
  • PowerShell Scripts
  • Proactive Remediation
  • 独立 Check-in ~58 min
这是理解排查路径的第一步：确认 app 类型走的是哪条通道。 Win32 App 走的是 IME 通道，和 LOB/Store app 的 OMA-DM 通道完全独立。 如果客户 report "app 没有推下来"，首先确认 app 类型 → 决定看哪个日志。 IME 通道的 Gateway 叫 StatelessSideCarGatewayService，客户端通过 HTTPS 轮询获取 policy。

🔄 Deployment Pipeline

Win32 App 部署的 5 个阶段（每个阶段都可能失败）

1️⃣ Metadata Processing
IME 从 SideCar Gateway 下载 policy metadata，解析 app 配置（detection rules, requirements, dependencies）
2️⃣ Detection & Applicability
检查 requirement rule（OS version, disk space）和 detection rule（是否已安装）
3️⃣ Download & Decrypt
从 Azure CDN (*.manage.microsoft.com) 下载 .bin 加密文件 → 解密 → 解压为 .intunewin
4️⃣ Installation
在 SYSTEM 或 User context 执行 install command（如 msiexec, setup.exe）
5️⃣ Post-Install Verification
再次运行 detection rule，确认 app 已成功安装 → 上报 status 到 Intune
部署 pipeline 共5个阶段。IME log 里每个阶段都有对应的日志关键词： Stage 1: "Get policies", "Deserializing" Stage 2: "DetectionManager", "ApplicabilityManager" Stage 3: "Downloading", "ContentManager" Stage 4: "Installing", "InstallerManager" Stage 5: "PostInstall detection" 排查时首先定位失败发生在哪个阶段。

⚙️ IME Prerequisites

支持的 Join Type

  • Supported Azure AD Joined Type 4
  • Supported Hybrid AAD Joined Type 9
  • Supported Co-managed Type 10
  • NOT Supported AAD Registered (BYOD) Type 1
⚠️ Type 1 (Workplace Join) 设备上 IME 不会安装，Win32 App 完全不工作。 这是最常见的误配问题之一。

关键限制

  • 📦 上传大小限制：
    • 免费 subscription: 8 GB
    • 付费 subscription: 30 GB
  • 🔄 Check-in 间隔：~58 分钟
  • 🖥️ 仅支持 Windows 10 1607+
  • 📡 需要访问 CDN endpoint: *.manage.microsoft.com

验证 Join Type: dsregcmd /status → AzureAdJoined / DomainJoined / WorkplaceJoined

Join Type 是排查第一关。如果客户是 BYOD (AAD Registered / Type 1)，IME 根本不会安装。 验证方法：dsregcmd /status 查看 AzureAdJoined: YES + DomainJoined: YES = Hybrid。 另外 Check-in 间隔约 58 分钟，客户说"deploy 了但设备上看不到"时，先等一个小时。 手动触发 sync: Company Portal → Settings → Sync, 或 Settings → Accounts → Access work → Sync。

⬇️ Top Download Failures

5.1 — CDN Unreachable

症状：IME Log 显示 download timeout 或 HTTP 403/404
CDN Domain：swda01-mscdn.manage.microsoft.com (Global)
China 21V：cnpasu01.manage.microsoftonline.cn
  • Proxy/Firewall 阻止 *.manage.microsoft.com
  • SSL Inspection 替换了证书 → 导致下载校验失败
  • China Telecom CDN 延迟 → 大文件下载 timeout
    • Verizon CDN 节点在中国覆盖差
// IME Log 示例 - Download failure
[Win32App] Downloading app content, contentUrl =
  https://swda01-mscdn.manage.microsoft.com/...
[Win32App] Download failed with error 0x80072EE7
  (ERROR_INTERNET_NAME_NOT_RESOLVED)

⬇️ Top Download Failures

5.2 — BITS Service Failure

IME 使用 BITS (Background Intelligent Transfer Service) 下载内容

  • BITS 服务未运行 → net start BITS
  • BITS 队列堆积 → bitsadmin /list /allusers
  • Proxy 配置冲突 → BITS 使用 WinHTTP proxy，不是 WinINET 
    netsh winhttp show proxy
netsh winhttp import proxy source=ie
  • Delivery Optimization (DO) 失败 → fallback 到 BITS
    • DO 日志: Get-DeliveryOptimizationLog

⬇️ Top Download Failures

5.3 — EnforcementRetryCount Exhausted

⚠️ 经典问题：app 反复下载失败后，IME 停止重试（retry count = 0）
即使修复了网络，app 也不会再尝试下载。

解决方案：重置 Registry 中的 retry count

## 查看当前 retry count
$path = "HKLM:\SOFTWARE\Microsoft\IntuneManagementExtension\Win32App"
Get-ChildItem $path -Recurse |
  Get-ItemProperty |
  Select-Object PSPath, EnforcementRetryCount

## 重置 retry count (需要 admin)
Set-ItemProperty -Path "$path\{UserGUID}\{AppGUID}" `
  -Name "EnforcementRetryCount" -Value 3

重置后重启 IntuneManagementExtension 服务触发新一轮尝试

EnforcementRetryCount 是一个很常见的 gotcha。客户修复了网络但 app 仍然不下载， 因为 retry count 已经耗尽。这个值在注册表里，手动重置为 3 即可。 注意每个 app 每个用户有独立的 retry count，路径是 HKLM\SOFTWARE\Microsoft\IntuneManagementExtension\Win32App\{UserSID}\{AppID}

🔍 Detection & Installation Errors

Detection Rule Types

  • Type 0 — Registry Key/Value
  • Type 1 — MSI Product Code
  • Type 2 — File/Folder Path
  • Type 3 — PowerShell Script
// IME Log - Detection Flow
[Win32App] Detection rules:
  DetectionType = 0 (Registry)
  KeyPath = HKLM\SOFTWARE\MyApp
  ValueName = DisplayVersion
  Operator = GreaterThanOrEqual
  DetectionValue = 2.0.0

常见安装错误

0x80070002
Install command 中的文件名与 .intunewin 内的实际文件名不匹配（大小写/路径）
-2016345060
安装执行完成但 detection rule 未匹配 → app 被判定为"未安装"
32-bit vs 64-bit 陷阱
PowerShell detection script 跑在 32-bit context → 注册表路径重定向到 Wow6432Node
Detection rule 类型决定了 IME 如何判断 app 是否已安装。 最常见的坑： 1. 0x80070002: Install command 写的是 "setup.exe" 但 intunewin 里实际叫 "Setup.exe" 或在子目录里。 2. -2016345060: 安装成功（exit code 0）但 detection 没通过。通常是 detection rule 配置错误。 3. 32-bit: 如果 "Run in 32-bit" 被勾选，PowerShell 跑在 SysWOW64 下， HKLM:\SOFTWARE 会被重定向到 HKLM:\SOFTWARE\Wow6432Node。

📋 Error Code Reference

Error Code Description 排查建议
-2016345060 App not detected after install 检查 detection rule 配置，验证 app 是否真的安装到了预期路径
-2016345059 App still detected after uninstall Uninstall command 未完全清理，或 detection rule 残留匹配
-2016330908 App install has failed 检查 install command exit code，查看 installer 自身日志
-2016344211 User cancelled installation 确认是否配置为 User context 且弹出了 UI 提示
-2147009281 No valid license / sideloading policy AADJ 设备默认允许 sideloading；Hybrid 可能需要 GP
0x80070002 File not found (command mismatch) 校验 install command 文件名和 .intunewin 中的文件路径
0x87D1041C Application not detected (generic) 与 -2016345060 类似，通常是 detection logic 问题
0x800705B4 Timeout (ESP scenario) 大 app 在 ESP 阶段下载超时，考虑 ESP timeout 策略调整

这张表是最常用的 error code 速查表。排查时先从 Intune portal 的 Device Install Status 拿到 error code， 然后对照这张表缩小范围。特别注意 -2016345060 是最高频的错误——安装"成功"但检测"失败"。

📁 Log & Registry Locations

日志路径

## IME 主日志（最重要）
%ProgramData%\Microsoft\IntuneManagementExtension\Logs\IntuneManagementExtension.log

## IME Agent 日志
%ProgramData%\Microsoft\IntuneManagementExtension\Logs\AgentExecutor.log

## Delivery Optimization 日志
Get-DeliveryOptimizationLog | Set-Content DOLog.txt

## MDM Diagnostic (一键收集)
mdmdiagnosticstool.exe -area DeviceEnrollment;DeviceProvisioning;Autopilot -cab c:\temp\mdmdiag.cab

注册表路径

## Win32 App 状态
HKLM\SOFTWARE\Microsoft\IntuneManagementExtension
  \Win32App\{UserSID}\{AppID}

## IME Policy (上次 check-in)
HKLM\SOFTWARE\Microsoft\IntuneManagementExtension
  \SideCarPolicies\Scripts\Reports

## GRS (Graph Result Service) 缓存
HKLM\SOFTWARE\Microsoft\IntuneManagementExtension
  \Win32App\{UserSID}\GRS

缓存路径

## 下载内容缓存（解密后的 .intunewin）
C:\Program Files (x86)\Microsoft Intune
  Management Extension\Content\Incoming\{GUID}
排查核心日志是 IntuneManagementExtension.log，位于 ProgramData 下。 这个日志非常大，建议用 CMTrace 或 VS Code 打开，搜索 app ID 或 app name 缩小范围。 GRS 注册表记录了最后上报给 Intune 的检测结果，有时客户端实际已安装但 GRS 缓存了旧状态。 删除 GRS 下的对应 key + 重启 IME 服务可以强制重新检测。

🔎 Key Kusto Queries

Service-side 排查 — 适用于 Global (Iceberg) & 21V (Jarvis)

// Query 1: App Install Attempts (by device)
SideCarAppEnforcement
| where DeviceId == "{DeviceId}"
    and ApplicationId == "{AppId}"
    and Timestamp > ago(7d)
| project Timestamp, EnforcementAction,
    EnforcementResult, ErrorCode,
    ContentDownloadStatus, InstallResult
| order by Timestamp desc
// Query 2: SideCar Check-in Events
SideCarEvent
| where DeviceId == "{DeviceId}"
    and Timestamp > ago(3d)
    and EventId in (5767, 5766)
    // 5767 = check-in success, 5766 = check-in failure
| project Timestamp, EventId, EventMessage
// Query 3: Content Download Status
SideCarContentDelivery
| where DeviceId == "{DeviceId}"
    and Timestamp > ago(7d)
    and ApplicationId == "{AppId}"
| project Timestamp, DownloadStatus,
    ContentSize, DownloadDuration,
    CDNEndpoint, ErrorCode
Kusto 查询是 service-side 排查的关键手段。 Query 1 看 enforcement（安装尝试）的整体状态。 Query 2 看设备是否成功 check-in 到 SideCar Gateway — 如果 5766 频繁出现说明设备和服务端通信有问题。 Query 3 看内容下载细节，包括用了哪个 CDN endpoint、下载耗时等。 21V 环境可能需要用 Jarvis 替代 Kusto。

🇨🇳 21V (Mooncake) 特别注意

CDN & 网络

  • CDN Endpoint: cnpasu01.manage.microsoftonline.cn
  • China Telecom + Verizon CDN
    • 下载大文件极慢（>1GB 可能 timeout）
  • ESP Timeout 0x800705b4
    • 大 App 在 Autopilot ESP 阶段超时
    • 建议：调整 ESP 超时策略或拆分 app

部署 WeChat 示例

## Step 1: 打包
IntuneWinAppUtil.exe -c "C:\Source\WeChat" `
  -s "WeChatSetup.exe" -o "C:\Output"

## Step 2: Install command
WeChatSetup.exe /s
## Uninstall command
"C:\Program Files\Tencent\WeChat\Uninstall.exe" /s

## Step 3: Detection Rule
## File: C:\Program Files\Tencent\WeChat\WeChat.exe
💡 Tips: WeChat 安装路径因版本而异， 建议用 Registry detection（HKLM\SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\Uninstall） 替代 File Path detection。
21V (Mooncake) 环境有几个特殊点需要注意： 1. CDN 域名不同：cnpasu01.manage.microsoftonline.cn 2. Verizon CDN 在中国大陆覆盖不好，电信用户下载特别慢 3. 大 App（如 Office）在 ESP (Enrollment Status Page) 阶段很容易超时 4. WeChat 是 21V 客户最常部署的 Win32 App 之一，注意版本差异导致的 detection 问题 5. 21V 的 Kusto 集群和 Global 不同，需要用 Jarvis 或 21V 专用集群查询

📌 Key Takeaways

1 确认架构通道 — Win32 走 IME/SideCar，不是 OMA-DM。先检查 Join Type (dsregcmd) 和 IME 服务状态。
2 定位 Pipeline 阶段 — 从 IME Log 定位失败在 Metadata / Detection / Download / Install / Post-verify 哪个阶段。
3 结合 Service-side 数据 — 用 Kusto 查 SideCar 事件、Enforcement 状态和 CDN 下载详情，客户端 + 服务端对照分析。
